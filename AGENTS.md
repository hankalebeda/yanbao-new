# 本项目的 AI 角色与用法

本项目通过**多 AI 角色**推进 A 股个股研报平台的设计与实现。

## 角色一览

| 角色 | 何时用 | Cursor 规则 |
|------|--------|-------------|
| **数据工程师** | 采集、标准化、血缘、双源行情、report_data_usage | 打开 `app/**/collect*.py` 等采集相关文件时自动应用 |
| **研报生成工程师** | LLM 调用、推理链、高级区、四维度绩效 | 打开 `app/**/llm*.py`、`app/**/report*.py` 等时应用 |
| **前端与体验** | 站点、列表、详情、看板、高级区展示 | 打开模板/静态资源或 `routes_business.py` 时应用 |
| **测试与质量** | 验收门禁、准确率口径、契约校验、DoD | 打开 `tests/**` 或 01/07 文档时应用 |

## 在对话中指定角色（不依赖规则）

直接说即可，例如：

- 「请以**数据工程师**身份：根据 07 契约设计热搜采集的请求/响应字段。」
- 「请以**研报生成工程师**身份：检查当前 prompt 是否覆盖 evidence_items → analysis_steps 的中文展示。」
- 「请以**测试与质量**身份：对照 01 验收计划，列出本迭代必须通过的检查项。」

## 角色接力

1. 数据工程师 → 输出契约与 report_data_usage 示例  
2. 研报生成工程师 → 研报 JSON 与高级区字段  
3. 前端与体验 → 页面与交互  
4. 测试与质量 → 验收用例与门禁  

在新对话中粘贴上一角色的结论并说明「请以 XX 角色继续」即可接力。

## 共同约束（所有角色）

- **文档管理**：需保留的文档必须全部有编号；临时草稿放 `_archive/`；**禁止将临时文件保存到桌面**。
- 文档 SSOT：全部位于 `docs/core/`。需求 → `01_需求基线.md`；架构 → `02_系统架构.md`；设计 → `03_详细设计.md`；数据 → `04_数据治理与血缘.md`；API/契约 → `05_API与数据契约.md`。
- **AI 接入策略**：本部署**不需** AI 官方 API Key，已有可用接口（ai-api 四个 Web API + Codex/Gemini CLI + 本地 Ollama）；详见 `docs/core/08_AI接入策略.md`。
- 真实性红线：禁止伪造数据；证据三要素；不可用时显式报错或降级。
- 商业底线：真实胜率≥55%、盈亏比≥1.5、年化Alpha≥10%；方向命中率仅作监控；统计带样本数、覆盖率。
- 高级区：中文展示「所用数据」与「生成全过程」；缺失/降级显式说明。

更多角色（预测与回灌、运维与可观测、产品与范围）及多 Agent 编排见 `.cursor/rules/` 下各角色规则文件。
