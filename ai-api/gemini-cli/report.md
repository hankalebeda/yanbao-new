# Gemini CLI 测试分析报告

## 1. 支持使用的模型列表
根据对 Gemini CLI 工具内核逻辑和默认配置项的实际挂载及代码追踪，当前命令环境支持调用以下最新的大语言模型：

* **优先体验版模型 (Preview)**
  * `gemini-3.1-pro` 
  * `gemini-3-flash`
  *(注意：部分高阶和未彻底GA的Tool Calling特性在3.1架构上会获得更加智能的表现)*
* **稳定基础模型 (Default)**
  * `gemini-2.5-pro` (CLI默认的主力调用模型)
  * `gemini-2.5-flash`
  * `gemini-2.5-flash-lite`

**指定模型方式**：可以通过命令行携带 `--model` （或 `-m`）参数手动指定，如不指定则走 Auto 模式由系统依据复杂性分配 `2.5-pro` 或 `2.5-flash`。

## 2. 联网搜索与并发执行测试结果
**核心结论：Gemini 底层 API 及联网检索天然支持并发。但由于主机端 CLI 会维持本地会话文件态，需将异构任务分发至“不同文件夹”执行来规避锁冲突。**

* **单实例联网搜索能力**：基于 `-p "查询今天东京的天气"` 和 `-p "查询今天伦敦的天气"` 的单例调用测试执行非常完美。CLI 成功感知意图触发了本地底层的 Grounding（Web Search）工具，精确返回了当天的真实气象状况（如东京多云61华氏度、伦敦多云11摄氏度）。
* **多实例并发测试 (同一目录下)**：若直接在同一个目录（如系统的 `研报` 根目录）瞬间启调多个并行的 `gemini -y -p ...` 任务。此时底层因会尝试同时拉取并重写该目录维度的 `.gemini` 本地缓存配置文件或 SQLite 会话元数据，导致触发“竞态条件冲突”从而致使一个或多个历程直接崩溃退出 (`Exit code: 1`)。
* **多实例并发测试 (隔离目录下)**：进一步创建多个平级子文件夹（举例：`test_concurrency_1`, `test_concurrency_2`），并在子文件夹中分别独立拉起 `gemini`。此时所有的 Google 联网查询均**完全并发执行并成功响应**，互相之间不再干扰。

## 3. Gemini CLI 使用方式指引
Gemini 是高度结合代理（Agent）与终端宿主机能力的 CLI，它不仅仅是一个纯对话入口，同时具备自动运行终端、读写文件的能力。

**核心基础语法**：
```bash
gemini [options] [command]
```

**命令行自动化常用核心参数**：
* `-p, --prompt <字符串>`：开启无头非交互模式。传入提示词后，AI 只产出一次完整答案便自动结束进程，并返回给 Shell（极度适合脚手架和自动化工作流）。
* `-y, --yolo`：全自动“同意”模式！因大模型有能力执行危险系统命令或写入文件，默认均会停下弹窗提示用户授权。携带此参数则会**静默信任并自动同意所有的操作行为**。强烈推荐配合脚本调度时开启（如：`gemini -y -p "分析当前代码"`）。
* `-m, --model <模型名称>`：限定本次流处理使用的具体模型，如 `-m gemini-3.1-pro`。
* `-i, --prompt-interactive`：混合模式。初始先执行抛入的 Prompt，后续完成后继续停留并挂起在对话框与用户保留交互状态。
* `-o, --output-format <text|json|stream-json>`：严格的格式化输出模式，如果你希望直接将结果导向 Python 解析器或数据库存储，可以用 json 格式截获原始报文避免多余终端字符干扰。
* `-r, --resume <index>`：恢复上一次终端会话的内存及语境（如：`--resume latest` 为恢复上一次的状态）。
* `-s, --sandbox`：沙盒安全执行模式。需系统级预装 Docker/Podman 此功能才能正常剥离机器指令的攻击面。

**实战自动化脚本场景示范**：
假如你在编写某支股票趋势盘点的研报，要并发拉起搜网查询，可以这样做：
```bash
# 务必分发落脚路径
mkdir worker_msft
cd worker_msft
# 挂载 YOLO 放手权限执行、明确模型规格、自动查询获取反馈
gemini -y -m gemini-3.1-pro -p "联网检索微软近日AI战略新闻并浓缩为200字" > msft_news.txt
```
