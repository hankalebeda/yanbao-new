# 03_凭据与安全

## 凭据优先级

`scripts/github_guardian.py` 的 GitHub 凭据读取顺序如下：

1. `GITHUB_TOKEN`
2. `GH_TOKEN`
3. `github/token.txt`

推荐始终优先使用环境变量。

## `github/token.txt` 的定位

- 它只是本机 fallback，不是推荐主路径。
- 该文件必须保持在 `.gitignore` 覆盖范围内。
- 不要在日志、文档、截图、命令回显中暴露 token。

## `gh` CLI 与 raw HTTP 的关系

- `github_guardian.py` 自己的 push / open PR 流程可以仅依赖 token + GitHub HTTP API。
- `gh-fix-ci`、`gh-address-comments`、`babysit-pr` 这类 skill 仍要求 `gh` 与 `gh auth status` 可用。
- 因此“没有 `gh`”不等于 guardian 无法运行，但会让相关 GitHub skill 处于 degraded/unavailable 状态。

## 最低安全规则

1. 不把 PAT 写入可提交文档。
2. 不把 PAT 写入 `github/automation/**` 运行产物。
3. 若怀疑 token 已暴露，先在 GitHub 侧轮换，再更新本机环境变量或本机 fallback 文件。
4. 调试时优先打印 token 来源类型，不打印 token 内容。

## 维护建议

- 计划任务优先注入环境变量，不依赖仓库内文件。
- 若必须使用 `github/token.txt`，确保它只存在于本机、且权限最小化。
- 新增与 GitHub 交互的脚本时，沿用相同优先级，不再引入新的明文凭据文件。
