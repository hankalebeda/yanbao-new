# 01_自动化总览

## 边界

本专题描述的是仓库治理自动化，不是业务 FR。本目录回答的是“如何巡检仓库、如何维持台账、如何在 GitHub 侧形成可追溯产物”，不是“业务接口怎么设计”。

## 链路总览

| 组件 | 角色 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| `continuous_repo_audit.py` | 连续巡检 | 当前仓库代码、pytest/治理快照 | `github/automation/continuous_audit/` |
| `live_fix_loop.py` | 台账状态机 | 问题轮次指标、预检结果 | `github/automation/live_fix_loop/issue_register.md`、`review_log.md` |
| `codex_prompt6_hourly.py` | Prompt 6 多中继修复执行器 | Prompt 6、providers、运行时预检 | `github/automation/live_fix_loop/automation_runs/` |
| `github_guardian.py` | 高层编排器 | worktree、模式、providers、push/open_pr 参数 | `github/automation/runs/`、`github/automation/_local/` |
| `issue_mining_22_codex.py` | 分析-only 挖掘器 | `docs/core/22_全量功能进度总表_v7_精审.md` | `runtime/issue_mining_22/` |

## 现役与兼容

| 状态 | 入口 | 说明 |
| --- | --- | --- |
| 现役 | `scripts/continuous_repo_audit.py` | 连续巡检主入口 |
| 现役 | `scripts/live_fix_loop.py` | live fix loop 台账入口 |
| 现役 | `scripts/codex_prompt6_hourly.py` | 当前有效的 Prompt 6 自动修复入口 |
| 现役 | `scripts/github_guardian.py` | GitHub 侧高层编排入口 |
| 现役 | `scripts/issue_mining_22_codex.py` | 分析-only 问题挖掘入口 |
| 兼容 | `scripts/prompt6_hourly_codex.py` | 旧手动 Prompt 6 入口，仍被部分 helper 复用 |
| 兼容 | `scripts/run_prompt6_hourly.ps1` | 旧手动 PowerShell 入口 |

## 推荐使用顺序

1. 只做巡检时，运行 `continuous_repo_audit.py`。
2. 需要真实修复闭环时，先用 `live_fix_loop.py init` 建台账，再由 `codex_prompt6_hourly.py` 续跑。
3. 需要 worktree、push、open PR、本地镜像产物时，使用 `github_guardian.py`。
4. 只做问题挖掘、不做修复时，使用 `issue_mining_22_codex.py`。

## 与 SSOT 的关系

- GitHub 自动化可以引用 `docs/core/22_*` 作为排查地图。
- GitHub 自动化不得把 `docs/core/22_*`、`github/automation/**`、`github/docs/**` 当作业务规格来源。
- 一旦发现需求、接口、字段、验收标准缺口，仍必须回写 `docs/core/01-05`。
