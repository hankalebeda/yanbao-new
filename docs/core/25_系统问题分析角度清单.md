## 三步使用法（替代以前的"最简使用方式"）

1. **定位**：按问题外观选区间，用快速索引找候选角度（一般 2–3 个角度组合即可）。
2. **取证**：每个角度产出 ✅/⚠️/🔴/— 判定，必须配 API 证据 + UI 证据（doc25 角度 41）。
3. **回落 SSOT**：按表中"主要落点"列回到 01/02/03/04/05 找规格依据，做出"修代码 / 改文档 / 人工核验 / 冻结隔离 / 外部接入"五选一决策。

## 使用边界

- 本文只提供“看问题的角度”和“分诊顺序”，不定义需求、字段、接口或验收标准。
- 用这些角度发现问题后，仍需回到 SSOT、代码、测试、运行证据做最终确认。
- 若某问题已在总表中标记为“已收口/已过时”，先确认它是否仍真实存活，再决定是否继续追。
- 同一个问题，通常至少要同时套 `事实/契约`、`运行态`、`门禁/治理` 三组角度，避免只看表象。
- 本文优先回答的是“先看什么、怎么分诊、下一步怎么落回 SSOT”，不替代 SSOT 对“应该是什么”的定义。

## 数据链路实例（2026-04-24）

| 实例 | 先看哪些角度 | 正确分类 | 当前处理结论 |
| --- | --- | --- | --- |
| `hotspot` 对 `2026-04-24` 核心池 `200` 只零命中 | `1 + 7 + 36 + 45` | 先修 writer honesty，再保留真实 coverage gap | 代码侧先过滤空 `stock_codes` 热点，并把 core pool 物化成 `ok / missing(no_hotspot_link)`；`0/200` 仍然说明上游主题与核心池无交集 |
| `market_state_input` 当日血缘缺失 | `1 + 9 + 36 + 39` | 直接修代码，再回放/补采 | 真相层在 `market_state_cache`，缺的是 derived batch + lineage + usage；现在应走独立的 `non-report truth materialize` 回放，而不是靠 report 生成时补写 |
| `northbound_summary` 最新 core pool `22/200 missing` | `1 + 12 + 36 + 37` | 部分代码缺口 + 剩余外部阻塞 | `fetcher_not_provided` 属入口 wiring，修代码；`northbound_data_unavailable` 属外部源真实缺口，保留 missing |
| `margin_financing` 全为 `proxy_ok` | `1 + 12 + 38 + 39` | 真相层质量问题，不是血缘 writer 问题 | 先保证 `proxy_ok` 被诚实落库并不再误判失败；只有远端历史恢复后才能把状态提升到 `ok/stale_ok` |
| `data_usage_fact / report_data_usage_link` 未闭环 | `1 + 34 + 36 + 37` | 分成“代码未双写”与“当前库无 report”两类 | 先让 replay 复用同一 `usage_id` 稳住 legacy→fact，再承认 `link=0` 若同时 `report=0` 仍属设计边界，不得伪造 link |

- 用角度 `36` 先分诊：是“直接修代码”、还是“先改代码再回放”、还是“外部接入/人工核验”。
- 用角度 `37` 标记状态：已修、仍存活、还是表述需收窄。
- 用角度 `39` 排顺序：先 truth-layer / lineage，再 scheduler wiring，再历史回放，最后才考虑下游 `report/settlement`。
- 用角度 `45` 做上限传导：当 `hotspot` 对核心池命中为 `0/200` 时，依赖该输入的下游闭环不得声称“已恢复”。

## 当前为什么仍值得优化（基于 `22_v7` 的必要性判断）

- 结论：**有必要优化，但不是因为角度不够多，而是因为旧版更像平铺清单，执行性不够强。**
- `22_v7` 已把当前高价值问题收敛到少数主战场：`真实性/血缘止血`、`坏批次/runtime anchor`、`FR-07 writer + rebuild`、`支付/认证/治理契约`。
- 旧版 `25` 虽然覆盖面足够，但没有把 `风险级别`、`问题状态`、`处理路径`、`推荐角度组合` 顶到前面，导致排查时仍容易平均用力。
- 页面/桥接/展示层问题并未消失，但 `22_v7` 已明确指出继续横向扫页面的收益在下降；除非出现直接用户误导、门禁失真或监控误判，否则排查顺序应后置。
- 因此，本次优化目标不是继续增加条目，而是把现有角度库改造成“先分诊，再套角度，最后回落 SSOT”的工作流。

> 背景：2026-04-17 `docs/core/22_全量功能进度总表_v16.md` 用 "活跃 205 / 活跃 205 = 100%" 得出"真实可用率 100%" 结论，被用户当场指控为欺骗。复盘结论是：前 39 个角度对 "分母被偷换"、"API 绿但 UI 空"、"软删除当修复"、"NOT_RUN 当进行中" 等手法**没有硬约束**，导致审计可以通过 cherry-pick 分母生成伪达标。本节 40–45 号角度作为强制前置检查，凡出现百分比的章节必须先过此六项。

## 最简使用方式

- 第一步先判 `P级别`：它是 `P0`、`P1` 还是可后置问题。
- 第二步再判 `状态`：它是 `仍存活`、`已过时`，还是 `表述需收窄`。
- 第三步确定 `处理路径`：`直接修代码`、`先改代码再重算或回放`、`人工核验`、`外部接入`、`冻结或隔离`。
- 第四步从上面的 `当前优先专题包` 或下方的 `问题家族组合` 选 `2-3` 个角度交叉看。
- 最后固定回答六件事：`是否有问题`、`属于哪一类`、`风险级别`、`当前是否仍真实存活`、`处理路径是什么`、`优先级是否值得现在修`。
