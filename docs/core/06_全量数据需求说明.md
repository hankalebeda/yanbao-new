# 06 全量数据需求说明

> 版本: v2.1  
> 最后更新: 2026-04-23  
> SSOT 地位: 本文档是系统“全部数据对象”的总地图，和 `04_数据治理与血缘.md`、`05_API与数据契约.md` 共同构成数据层三角基准。  
> 适用范围: 不只覆盖研报主链，也覆盖用户、支付、模拟交易、治理、调度、审计、通知等系统级数据。

---

## 1. 文档目的与本次修订结论

本文档回答四个问题：

1. 系统到底需要哪些数据，而不只是“生成研报要哪些数据”。
2. 每类数据来自哪里，由谁采集/计算，由谁消费。
3. 数据是持久化到 DB、写入 `report_data_usage` 快照，还是仅在查询时 read-through 组装。
4. 目前哪些链路已经补齐，哪些数据是“按设计不持久化”，避免把设计选择误判成缺漏。

本次修订的直接结论如下：

- 06 文档从“研报相关数据清单”扩展为“全系统数据地图”。
- 非研报补采链路已补齐到批量研报入口：`generate_reports_batch()` 每个股票在实际生成前，先走 `collect_non_report_usage_sync()`。
- `POST /api/v1/internal/reports/generate-batch` 现在固定执行“每次最多 3 个研报，A/B/C 每种 1 个”的规则。
- 复查后确认：`company_overview`、`valuation`、`industry_competition`、`news_policy`、`market_snapshot` 属于详情页 read-through 数据，不写入 `report_data_usage`，这是设计边界，不是数据缺漏。

阅读建议：

- 只关心研报主链：读第 3、4、6、7、10 节。
- 关心系统所有数据对象：读第 3、5、8、9 节。
- 关心“为什么有些数据不入库”：重点看第 6.2、6.3 节。

## 2. 数据分层总览

系统数据按 5 层理解最清晰：

| 层级 | 含义 | 典型数据 | 主要存储方式 |
|---|---|---|---|
| L0 外部输入层 | 外部 API、本地行情文件、第三方认证/支付回调、LLM 输出 | TDX `.day`、东方财富、AKShare、微博/雪球/财联社、OAuth 回调、Webhook | 不直接暴露，先进入服务层处理 |
| L1 标准化采集层 | 统一格式后的证券、行情、热点、批次、血缘、熔断状态 | `stock_master`、`kline_daily`、`market_hotspot_item*`、`data_batch*`、`report_data_usage` | DB 表 + 快照 |
| L2 研报上下文层 | 用于判定、补采、详情页展示的派生上下文 | `market_state_cache`、`stock_pool*`、`main_force_flow`、`stock_profile`、`company_overview`、`valuation` | DB 表 / `report_data_usage` / read-through |
| L3 产品业务层 | 对用户可见或对业务闭环有意义的产物 | `report*`、`instruction_card`、`sim_trade_instruction`、`settlement_result`、`sim_position` | DB 表 |
| L4 系统治理层 | 认证、支付、调度、审计、通知、清理、实验、熔断 | `app_user`、`billing_order`、`scheduler_task_run`、`audit_log`、`notification` | DB 表 |

系统采用三种数据落地模式：

| 模式 | 定义 | 典型对象 | 适用原因 |
|---|---|---|---|
| 独立业务表 | 每条数据有单独表结构与索引 | `report`、`sim_position`、`billing_order` | 需要查询、统计、审计、回放 |
| `report_data_usage` 快照 | 以 `dataset_name + status_reason` 存补采快照 | `main_force_flow`、`stock_profile` | 只需保留研报输入证据，不必建宽表 |
| read-through 组装 | 查询时临时调用服务组装返回，不落库 | `company_overview`、`valuation`、`news_policy` | 时效性强、只服务详情页、无需离线审计 |

## 4. 研报主链与核心市场数据

本节聚焦“股票池筛选 -> 行情/热点 -> 市场状态 -> 非研报补采 -> 研报生成”的主链数据。

### 4.1 证券主数据 (`stock_master`)

**用途**：统一股票代码、名称、交易所、行业和状态，是股票池、详情页、研报生成的基础主键域。  
**来源**：行情基础资料同步。  
**核心字段**：

| 字段 | 说明 |
|---|---|
| `stock_code` | 证券唯一标识，如 `600519.SH` |
| `stock_name` | 股票名称 |
| `exchange` | 交易所标识 |
| `industry` | 行业 |
| `list_date` | 上市日期 |
| `circulating_shares` | 流通股本 |
| `is_st / is_suspended / is_delisted` | 交易状态、风控过滤标志 |

### 4.2 K 线日线 (`kline_daily`)

**用途**：技术分析、市场状态、股票池打分、指令卡、结算、模拟交易的共同底座。  
**来源**：

- 主源：TDX 本地 `.day` 文件，`app/services/tdx_local_data.py`
- 备用：东方财富行情接口，`app/services/market_data.py`

**采集入口**：`_handler_fr04_data_collect -> ingest_market_data`。  
**存储**：`kline_daily`。  
**关键字段**：

| 字段 | 说明 |
|---|---|
| `trade_date` | 交易日 |
| `open / high / low / close` | 前复权 OHLC |
| `volume / amount` | 成交量、成交额 |
| `turnover_rate` | 换手率 |
| `ma5 / ma10 / ma20 / ma60` | 均线 |
| `atr_pct` | ATR 百分比 |
| `volatility_20d` | 20 日波动率 |
| `hs300_return_20d` | 同期沪深 300 涨跌幅 |
| `is_suspended` | 是否停牌 |
| `source_batch_id` | 来源批次 |

### 4.3 热点与事件 (`market_hotspot_item*` / legacy `hotspot_top50`)

**用途**：A 类策略判定、热点展示、研报证据引用。  
**来源**：东方财富、雪球、财联社、微博、抖音、快手、百度。  
**生产形态**：

- 现行主表：`market_hotspot_item`、`market_hotspot_item_source`、`market_hotspot_item_stock_link`
- 兼容旧表：`hotspot_top50`

**关键字段**：

| 字段 | 说明 |
|---|---|
| `topic_title` | 热点标题 |
| `source_name / source_url / fetch_time` | 证据三要素 |
| `news_event_type` | 事件标签，命中时影响 A 类策略判定 |
| `merged_rank / source_rank` | 聚合排名 / 单源排名 |
| `stock_code / match_confidence` | 个股关联结果 |

### 4.4 市场状态 (`market_state_cache`)

**用途**：市场大盘过滤、首页状态展示、研报发布门控。  
**来源**：`kline_daily + hotspot` 派生计算。  
**计算入口**：`calc_and_cache_market_state()`。  
**核心字段**：

| 字段 | 说明 |
|---|---|
| `trade_date / reference_date` | 状态服务日 / 引用基准日 |
| `market_state` | `BULL / NEUTRAL / BEAR` |
| `cache_status` | 缓存状态 |
| `market_state_degraded` | 是否降级 |
| `state_reason` | 判定或降级原因 |
| `a_type_pct / b_type_pct / c_type_pct` | 不同策略适配比例 |
| `kline_batch_id / hotspot_batch_id` | 来源批次 |

### 4.5 股票池与打分 (`stock_pool*`)

**用途**：限定研报生成对象；支持首页、列表页和历史守护。  
**来源**：`stock_master + kline_daily` 的 8 因子评分。  
**对象拆分**：

| 对象 | 用途 | 核心字段 |
|---|---|---|
| `stock_pool` | legacy 汇总结果 | `pool_date`、`core_pool`、`standby_pool`、`fallback_from` |
| `stock_pool_snapshot` | 单股快照行 | `trade_date`、`stock_code`、`pool_role`、`rank_no`、`score` |
| `stock_score` | 打分因子明细 | `factor_momentum`、`factor_liquidity`、`factor_ma20_slope` 等 |
| `stock_pool_refresh_task` | 刷新任务与状态 | `status`、`pool_version`、`evicted_stocks_json` |

### 4.6 研报补采数据 (`report_data_usage` 中的 dataset)

这部分是“非研报数据补齐”的核心。它们不是研报文本本身，但直接影响研报能否生成、能否解释、能否追溯。

| dataset_name | 来源 | 用途 | 存储方式 | 关键字段/快照 |
|---|---|---|---|---|
| `kline_daily` | TDX / 东方财富 | 主链必需 | `report_data_usage` 血缘记录 + `kline_daily` 实表 | 行情批次、状态 |
| `hotspot_top50` | 多源热点 | 主链必需 | 血缘记录 + 热点实表 | 来源、排名、状态 |
| `market_state_input` | 本地计算 | 主链必需 | 血缘记录 + `market_state_cache` | 市场状态、状态原因 |
| `main_force_flow` | 东方财富 `fflow_daykline` | 资金面增强 | `report_data_usage.status_reason` 前缀 `capital_snapshot:` | `net_inflow_*`、`streak_days` |
| `dragon_tiger_list` | 东方财富 LHB | 机构/游资行为 | 同上 | `lhb_count_*`、`net_buy_total` |
| `margin_financing` | 东方财富 `push2 rzrq` | 杠杆资金 | 同上 | `latest_rzye`、`latest_rqye`、`delta_*` |
| `northbound_summary` | AKShare | 北向概况 | `report_data_usage` 普通行 | `status`、`reason`、概况数值 |
| `etf_flow_summary` | ETF 申赎汇总 | 被动资金概况 | `report_data_usage` 普通行 | `status`、`reason`、净申赎值 |
| `stock_profile` | 东方财富 `stock/get` | 基本面增强 | `status_reason` 前缀 `profile_snapshot:` | `pe_ttm`、`pb`、`roe_pct`、`total_mv`、`industry` |

补充说明：

- `main_force_flow`、`dragon_tiger_list`、`margin_financing`、`stock_profile` 采用“快照前缀 + JSON”的轻量存储模式。
- `northbound_summary`、`etf_flow_summary` 保留整条状态记录即可，不需要再建宽表。
- 每条 `report_data_usage` 都必须具备有效 `batch_id`，对应 `data_batch` 中存在的批次。

#### 4.6.2 策略类型判定算法

`_determine_strategy_type()` 在研报生成时为每只股票确定策略分类（A/B/C），判定结果影响 LLM prompt 指导方向和指令卡止盈止损倍率。

判定优先级（从高到低）：

| 类型 | 判定条件 | 指令卡 ATR 倍率 | 适用场景 |
|---|---|---|---|
| **A（事件驱动）** | `market_hotspot_item_stock_link` 中该股在最近3天内有 `news_event_type` 非空的关联记录 | 1.5× | 热点/事件催化窗口期 |
| **C（低波套利）** | `atr_pct < 2%` 且 `volatility_20d ≤ 当日全市场 30th percentile` | 2.5× | 低波动、侧向震荡 |
| **B（趋势跟踪）** | `close > MA20` 且近5日涨幅 > 3%（查 `kline_daily` 中5天前收盘价） | 2.0× | 明确上升趋势 |
| **B（默认）** | 不满足以上任何条件 | 2.0× | 通用兜底 |

注意事项：

- A 类优先级最高，只要该股近3天有事件挂钩，就直接归 A，不再判断 B/C。
- C 类需要当日 `kline_daily.volatility_20d` 非空记录数足够；若无法计算30th percentile，使用 `0.02` 兜底阈值。
- B 类需要5天前的 `kline_daily` 记录存在；若历史数据缺失，默认返回 B。
- `market_state == BEAR` 时，B/C 类研报不发布（`publish_status = UNPUBLISHED`，`status_reason = BEAR_MARKET_FILTERED`）。
- A 类在 BEAR 市场仍可发布（事件驱动逻辑不受熊市过滤）。

### 4.8 v27 研报自包含快照（`content_json` 嵌入字段）

从 v27 起，每份研报的 `report.content_json` 除了面向展示的结构（`evidence_backing_points`、`direction_forecast`、`price_forecast` 等）外，还嵌入了5组数据快照，确保研报**自包含**：即使原始 `report_data_usage` 记录改变或被清理，研报仍能重现生成时的数据状态。

| 嵌入字段 | 数据来源 | 主要子字段 | 说明 |
|---|---|---|---|
| `capital_game_snapshot` | `capital_usage_collector` 快照解析 | `main_force.{net_inflow_1d/5d/10d/20d/streak_days}`、`dragon_tiger.{lhb_count_30d/90d/net_buy_total}`、`margin_financing.{latest_rzye/latest_rqye/rzye_delta_5d}`、`northbound.status`、`etf_flow.status` | 资金博弈全景快照；各子段均含 `_fmt` 格式化版本（如 `"2.34 亿"`） |
| `stock_profile_snapshot` | `stock_profile_collector` 快照解析 | `pe_ttm`、`pb`、`roe_pct`、`total_mv`（+`_fmt`）、`circulating_mv`、`industry`、`region`、`list_date` | 基本面快照；缺失时 `{status: "missing", reason: "stock_profile_not_collected"}` |
| `market_state_snapshot` | `market_state_cache` 表 | `market_state`、`state_reason`、`reference_date`、`market_state_degraded` | 生成时市场状态快照 |
| `kline_snapshot` | `kline_daily` 表 | `trade_date`、`open/high/low/close`、`ma5/ma20`、`atr_pct`、`volatility_20d` | 生成时K线快照（前复权） |
| `data_completeness` | 门控运算结果 | `total_required`、`total_ok`、`all_complete`、`checks[{dataset/status/source/has_data}]` | 仅覆盖 `_REPORT_REQUIRED_INPUT_DATASETS` 中的3个必需数据集 |

补充说明：

- 金额类字段同时提供 `_fmt` 版本，供前端展示使用，无需前端自行换算。
- `data_completeness.checks` 以 `_REPORT_REQUIRED_INPUT_DATASETS = {"kline_daily", "hotspot_top50", "market_state_input"}` 为基准，不包含可选增强数据集。
- 当子数据集缺失时，快照中写入 `{status: "missing", reason: "..."}` 占位，不静默置空。
- 嵌入快照通过 `_build_content_json_and_quality_issues()` 在 `generate_report_ssot()` 内构建，生成后与 `report` 记录一同持久化，不可修改。

### 6.1 哪些数据属于“非研报数据”

### 6.2 当前已打通的链路

#### A. 公开个股快照链路

- 路由：`GET /api/v1/stocks/{stock_code}/snapshot`
- 入口：`app/api/routes_business.py`
- 组装函数：`app/services/stock_snapshot_service.py::build_stock_snapshot_payload`
- 返回范围：`company_overview / market_snapshot / market_state / valuation / financial_analysis / industry_competition / capital_data / hotspot / news_policy / data_sources`

这条链路覆盖“详情页需要的全部非研报数据”。

#### B. 内部补采链路

- 路由：`POST /api/v1/internal/stocks/{stock_code}/non-report-data/collect`
- 入口：`collect_non_report_usage()` / `collect_non_report_usage_sync()`
- 批量 truth 回放入口：`materialize_non_report_usage_for_pool()` / `scheduler._handler_fr05_non_report_truth_materialize()`
- 落库范围：
  - 1 个市场状态输入血缘：`market_state_input`（来源 `market_state_cache` + `data_batch_lineage`）
  - 3 个资金类数据：`main_force_flow`、`dragon_tiger_list`、`margin_financing`
  - 1 个基本面快照：`stock_profile`
  - 2 个汇总数据：`northbound_summary`、`etf_flow_summary`

这条链路覆盖“研报生成前应落入 truth-layer 的全部非研报补采数据”。从 2026-04-24 代码修正起，写入语义是：

- 先落 `data_batch` / `data_batch_lineage` / `report_data_usage`
- 再同步镜像到 `data_usage_fact`
- 只有真正生成了研报，才会继续落 `report_data_usage_link`
- FR04 对 `hotspot_top50` 直接按 core pool 物化 usage：有 link 的股票写 `ok`，无 link 的股票写 `missing/no_hotspot_link`；不会再把空 `stock_codes` 的泛热点直接落成“可用 truth”

因此：`report_data_usage_link=0` 在 `report=0` 的库里是**设计边界内的真实状态**，不能把它误写成“writer 失效”。

补充说明：`collect_non_report_usage()` 除了真正写入 `report_data_usage` 外，还会为每一类补采建立 `data_batch` 记录，返回结构如下。

| 返回块 | 批次来源 | 返回字段 | 说明 |
|---|---|---|---|
| `capital_usage` | `source_name=supplemental_capital`、`batch_scope=stock_supplemental` | `batch_id`、`per_dataset`、各资金子项持久化状态 | 一次补采内同时覆盖 `main_force_flow`、`dragon_tiger_list`、`margin_financing` 3 个数据集 |
| `market_state_input` | `source_name=market_state_cache`、`batch_scope=market_state_derived` | `batch_id`、`usage_id`、`status`、`market_state_trade_date` | 由 `market_state_cache` 真相层派生，父批次来自 `kline_batch_id` / `hotspot_batch_id` |
| `stock_profile` | `source_name=stock_profile`、`batch_scope=stock_supplemental` | `batch_id`、`persisted_status`、`reason` | 基本面快照单独建批，便于与资金类补采分开追溯 |
| `northbound_summary` | `batch_scope=summary` | `status`、`reason`、`batch_id` | 以普通状态行写入 `report_data_usage`，`status_reason` 为 JSON 摘要或失败原因 |
| `etf_flow_summary` | `batch_scope=summary` | `status`、`reason`、`batch_id` | 同上；市场级汇总，不会生成个股级宽表 |

同步包装器 `collect_non_report_usage_sync()` 只负责在同步上下文里启动事件循环，不改变补采逻辑；批量研报入口与调度器入口最终复用的是同一套补采实现。

新增的 `materialize_non_report_usage_for_pool()` 只是在同一套 writer 之上做核心池批量回放：不生成研报，不补结算，只负责把最新 `market_state_input / capital_usage / stock_profile / northbound_summary / etf_flow_summary` 物化进 truth-layer。

#### C. 调度器日常生成链路

- 路径：`scheduler._run_one_stock`
- 顺序：先 `collect_non_report_usage(...)`，再 `generate_report_ssot(...)`

也就是说，日常批处理原本就已经具备“先补采，再生成”的闭环。

#### D. 批量研报生成链路（本次补齐）

- 路径：`app/services/report_generation_ssot.py::_gen_one_sync`
- 现状：每只股票在进入 `generate_report_ssot(...)` 前，先执行 `collect_non_report_usage_sync(...)`

这次补齐的重点就在这里。此前调度器路径是通的，但手工/内部批量生成路径没有把非研报补采串进去，导致“能力存在，但入口没接上”。现在两条路径行为对齐了。

### 6.3 为什么不是所有非研报数据都要入 `report_data_usage`

因为系统里有两种完全不同的数据诉求：

1. **发布审计诉求**：用于解释“这份研报生成时依赖了哪些输入、输入当时状态如何”。这种数据必须落地并能追溯，所以进入 `report_data_usage`。
2. **读模型诉求**：用于详情页查询时补充信息，如公司简介、最新新闻、同行列表。这类数据主要服务当前请求，不必为了审计专门建表。

因此：

- `stock_profile` 需要持久化，因为它可能进入研报高级区解释和完整性判定。
- `company_overview` 不持久化，因为它只影响详情页展示，不影响是否允许发布研报。
- `market_state_input` 必须持久化，因为它既是生产硬门控，又需要能追溯到 `kline/hotspot` 父批次；这不是 read-through 数据。
- `report` / `settlement_result` 属于下游业务产物；当上游 truth-layer 已修复但当前库仍为空时，应如实记为“下游尚未恢复”，不得通过补采接口伪造。

### 6.4 研报完整性门控口径

`generate_report_ssot()` 的门控有两层：

| 运行口径 | 必需数据集 | 说明 |
|---|---|---|
| 生产口径 (`settings.mock_llm=false`) | `kline_daily`、`hotspot_top50`、`market_state_input` | 这是生产硬门控 |
| 测试 / mock 口径 (`settings.mock_llm=true`) | 上述 3 项 + `northbound_summary`、`etf_flow_summary` | 测试环境更严格，用于防止补采链路回归 |

增强型但非生产硬门控的数据包括：

- `main_force_flow`
- `dragon_tiger_list`
- `margin_financing`
- `stock_profile`

这些数据缺失时，不一定阻止生产生成，但会降低可解释性和高级区信息密度。

#### 发布门控（Publish Gate）详细条件

研报数据完整性门控通过后，还有第二道**发布门控**决定研报是否对用户可见：

| 条件 | 行为 | `publish_status` | `review_flag` |
|---|---|---|---|
| `quality_flag == "degraded"` | **阻断发布** | `UNPUBLISHED` | `NONE` |
| `quality_flag == "stale_ok"` 且 `quality_reason` 含 `fetcher_not_provided / northbound_not_ok / etf_flow_not_ok / no_etf_data_available` | **阻断发布，标记人工审核** | `UNPUBLISHED` | `PENDING_REVIEW` |
| `llm_fallback_level` 为 `failed` 或 `rule_based` | 不发布，标记待审 | `UNPUBLISHED` | `PENDING_REVIEW` |
| `quality_flag == "ok"` 且 `llm_fallback_level == "primary"` | **正常发布** | `PUBLISHED` | `APPROVED` |
| 停牌股票（`is_suspended=True`） | 不生成实质内容，跳过 | `UNPUBLISHED` | `NONE`，`quality_flag=degraded` |
| `market_state == BEAR` 且 `strategy_type` 为 B 或 C | 生成但不发布 | `UNPUBLISHED` | `NONE`，`status_reason=BEAR_MARKET_FILTERED` |

注意：`quality_flag` 和 `publish_status` 是两个独立字段。`quality_flag` 反映数据质量，`publish_status` 反映是否对用户展示。两者可以组合出多种中间态（如数据 ok 但 LLM 降级 → `quality_flag=ok, publish_status=UNPUBLISHED`）。

## 7. 批量研报生成规则

### 7.1 规则本身

系统级规则是：**每次最多生成 3 个研报，且 A / B / C 每种策略类型各 1 个。**

实现方式：

1. 对输入股票列表做预判：根据 `kline_daily + market_hotspot_item` 判定候选的 `strategy_type`。
2. 仅保留每种策略类型的第一个股票。
3. 最终进入生成的股票数上限为 3。

### 7.2 现在到底在哪一层强制执行

| 层级 | 现状 |
|---|---|
| `generate_reports_batch(..., one_per_strategy_type=...)` 服务函数 | 保留开关，便于维护型任务按需全量生成 |
| `POST /api/v1/internal/reports/generate-batch` 内部 API | **固定强制 `one_per_strategy_type=True`** |

这样设计的原因是：

- 内部 API 面向常规批量生成，必须受“最多 3 个、每种 1 个”约束。
- 守护修复类任务（如 `history_guardian`）可能需要直接调用服务函数做缺失补齐，不应被 API 的常规限流规则绑死。

### 7.3 当前 API 语义

`POST /api/v1/internal/reports/generate-batch` 的实际语义已经收口为：

- 接收一个候选 `stock_codes` 列表
- 先执行 pre-cleanup（默认开启）
- 再以 “A/B/C 各 1 个，最多 3 个” 的规则预筛选
- 每个入选股票先补采非研报数据，再生成研报

## 9. 数据缺漏复查结论

本次对 `docs/core`、`app/`、`tests/` 的交叉复查，结论如下。

### 9.1 已确认补齐

1. 文档层：06 已覆盖全系统数据，而不再只覆盖研报主链。
2. 批量生成层：内部批量 API 固定执行“最多 3 个、A/B/C 各 1 个”。
3. 非研报补采层：批量生成入口已经补上 `collect_non_report_usage_sync()`，与调度器链路对齐。
4. 详情页层：`build_stock_snapshot_payload()` 已覆盖公司概况、估值、行业竞争、新闻政策、热点、资本数据、市场状态、技术快照。

### 9.2 复查后不判定为缺漏的项目

1. `company_overview`、`valuation`、`industry_competition`、`news_policy` 未持久化：这是 read-through 设计，不是漏采。
2. `financial_analysis` 没有单独表：它是展示层派生对象，不是采集事实表。
3. 内部服务函数仍保留 `one_per_strategy_type` 开关：这是维护任务需要，不影响内部 API 的固定约束。

### 9.3 需要持续关注但不应伪造补齐的项目

1. `northbound_summary`、`etf_flow_summary`、资金面数据受外部数据源可用性影响，允许 `stale_ok` / `missing`，禁止伪造数值。
2. LLM、外部热点、支付回调、Cookie 平台探测都属于运行时波动数据，遇到失败要显式降级，不得静默补空。

复查结论：**当前代码与文档口径下，没有再发现“系统需要但 06 完全未声明”的数据大类；剩余问题主要是外部数据源可用性，而不是本地数据模型缺项。**

## 10. 数据状态码枚举（`report_data_usage.status`）

| 状态码 | 含义 | 允许研报生成 |
|---|---|---|
| `ok` | 数据完整且新鲜 | 是 |
| `stale_ok` | 数据稍旧但仍可接受 | 是，需显式降级说明 |
| `realtime_only` | 仅有实时快照、无完整历史 | 是，仅适用于部分资金类快照 |
| `degraded` | 数据质量下降或计算链异常 | 否 |
| `missing` | 数据缺失 | 生产硬门控数据集缺失时否 |

补充规则：

- `status != ok` 时，`status_reason` 必须可解释，不能写空字符串。
- 快照型数据若 `status=ok`，`status_reason` 可以承载 JSON 快照内容；否则应承载可读失败原因。
- `report_data_usage` 的状态码用于“输入数据真实状态”，不能把页面展示层的空值处理伪装成 `ok`。
