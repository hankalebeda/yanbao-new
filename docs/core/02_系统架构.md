## 2. 行动 07：分层架构与调用规则

### 2.1 架构原则（十二条执行铁律）

> 这些原则不是指导意见，是代码准入检查项——不满足就不合入。

1. **业务决策集中在 L2**：买卖建议、信号类型判定（A/B/C）、会员权益裁剪（含 Free 后端摘要裁剪 ≤200 字符）、清理策略、先验数据注入、BEAR 短路拦截、停牌跳过判定、K 线悲观撮合结果判定、术语情境化填充等决策只在 L2 业务编排层做出，L1/L3/L4 不做业务结论判定。
2. **治理能力集中在 L3**：锁、幂等、熔断、限速、频控、事务边界、Outbox 派发、审计、告警、降级标记持久化与告警传播、断点续传统一由 L3 治理控制层管理，不散落在各业务模块；L3 负责执行治理动作（含将 L2 已给出的降级结论持久化），不改写 L2 已给出的业务判定结果。
3. **外部差异封装在 L4**：数据库、LLM、第三方平台（通达信/OAuth/支付/SMTP）的差异由 L4 资源适配层统一封装，上层通过契约对象（`Protocol/Port`）调用，不直接耦合外部实现；L4 适配器默认采用结构化子类型实现，避免跨层静态 import。
4. **依赖驱动禁止时钟驱动**：上下游依赖用"上游完成信号/事件"驱动下游触发（DAG），禁止用固定时刻替代依赖完成（如禁止"16:30 强行触发 FR-07"）；仅允许在异常治理路径使用"次日开盘前兜底终止时间"作为**停止条件**，不得作为下游启动条件。
5. **降级必须可见可追溯**：任何降级路径必须显式标注原因字段（`quality_flag`、`status_reason`、`state_reason`）并触发 NFR-13 告警，前台必须展示降级提示横幅，不允许静默伪成功。
6. **前复权动态折算**：涉及持仓绩效计算（FR-07/FR-08）时，禁止直接用历史冻结价格与当日价格比对；必须在同一前复权序列上下文中提取 entry/exit 价格。
7. **口径隔离**：FR-07（信号结算，1 手标准单位）与 FR-08（资金模拟，按挡位实际手数）口径严格隔离，同名字段含义不同，禁止混用。
8. **日内先平后开**：同一交易日批处理必须先执行全部平仓释放资金，再按 confidence 降序执行新开仓（FR-08）。
9. **事务后派发**：凡依赖持久化结果的外发通知（邮件/Webhook）必须走 Outbox 模式，在事务 commit 成功后派发，事务回滚则取消派发（FR-13/NFR-11）。
10. **资金挡位全局常量驱动**：`capital_tier` 枚举（`10k|100k|500k`）由全局配置源（如 `config.CAPITAL_TIERS`）统一定义；FR-06/07/08/10/12 等涉及挡位的模块必须动态遍历该常量，禁止散落硬编码字符串，新增挡位时不得要求多处业务逻辑手改。
11. **幂等键统一治理**：业务主键去重（`idempotency_key`/`event_id`/`trade_date+position_id`/`trade_date+task_name`）统一由 L3 幂等拦截层管理；L2 仅声明 `key + duplicate_policy`（如 `return_existing|skip|reject`）。L2 允许在"去重后有效事件集合"上执行业务聚合（如 FR-11 负反馈阈值触发），但不得自行实现幂等存储或并发去重器。
12. **鉴权/权益前置、业务后置**：L1 统一完成身份识别与前置校验，外部用户链路使用 JWT（claims 至少含 `sub/role/tier/exp`，且 `access_token` 必含 `jti` 以支持撤销黑名单），内部链路使用 `X-Internal-Token`/签名校验；L2 可信赖 L1 已验证的调用身份，不重复鉴权。

### 2.3 各层职责、技术基线、禁止事项

| 层级 | 必做职责 | 技术基线 | 本层禁止 |
|---|---|---|---|
| **L0 触发层** | 发起请求或触发任务（时间信号、外部回调、事件完成信号） | 浏览器、API Client、系统时钟信号、内部事件总线 | 写业务规则、写持久化逻辑、发起跨层静态 import |
| **L1 接入层** | 路由接入、JWT 鉴权（claims 含 sub/role/tier/exp/**jti**；jti 用于黑名单撤销）、jti 黑名单校验（通过 `TokenRevocationPort`，命中则 401 拒绝）、内部接口 `X-Internal-Token` 校验、Webhook 签名校验、RBAC 前置、入参 Pydantic 校验、NFR-14 统一响应封装（`{success, request_id, data}`）、NFR-07 未授权拦截、频控拦截入口（429 响应，通过 `RateLimitPort`） | FastAPI（JSON API）、Jinja2（HTML 页面）、Pydantic v2（入参校验） | 直连数据库、直连 LLM、直连支付/OAuth、做业务决策（如买卖建议/信号类型判定）、维护频控状态、直接 import governance/* 具体模块 |
| **L2 业务编排层** | 按 FR 编排链路、做业务判断与分支（信号类型判定 A/B/C、先验数据注入、辩证审计编排、仓位策略、对照组统计、会员权益裁剪含 Free 后端摘要裁剪、清理策略判定、BEAR 短路拦截、停牌跳过判定、K 线悲观撮合判定、复权动态折算、术语情境化填充、数据未就绪状态管理）；只读查询通过 Port 契约对象发起（运行时注入 L4 适配器） | UseCase/Service 模块（每个 FR 对应一个或多个 Service） | 绕过 L3 直接外发消息或落库；在 L2 内自行实现锁/事务/熔断/重试等治理逻辑；散落硬编码挡位字符串；实现展示层截断/样式逻辑 |
| **L3 治理控制层** | 调度与 DAG（含全局兜底超时与跨日一致性）、分布式互斥锁（含 TTL/fencing token/续约心跳）、回调注册表（handler_id→callable，运行时分发）、幂等拦截（含 Webhook event_id 去重）、事务边界与原子提交、Outbox 派发（事务后外发）、熔断器（含半开探测与冷却期）、限速器（数据源采集防封）、频控/防刷（原子计数+阈值判定，含 FR-09 登录频控/FR-11 反馈防刷/FR-13 告警抑制）、审计日志（before/after 可追溯）、告警（日志+Webhook+同类抑制窗口）、断点续传（任务游标/恢复窗口/僵尸过期）、降级标记持久化 | Scheduler、Handler Registry、Lock Manager、Idempotency Guard、Transaction Manager、Circuit Breaker、Rate Limiter、Outbox Consumer、Audit Logger | 输出业务结论（如买卖建议、会员裁剪、研报内容）；静态 import L2 业务模块 |
| **L4 资源适配层** | 统一封装所有内部存储与外部依赖：数据库读写（WAL 模式）、通达信本地行情、多源热搜爬取（7 平台优先级适配）、LLM 多级降级调用（主→备→CLI→本地→failed）、OAuth Mock 适配（QQ/微信）、支付渠道适配、SMTP 邮件适配、Webhook 外发适配 | SQLAlchemy+SQLite（WAL 模式）、HTTP Adapter（含重试）、LLM Router Adapter、OAuth/Pay Adapter、SMTP Adapter | 承担业务决策、重写 FR 口径、修改业务字段语义 |

**组合根（Composition Root）约束**：
1. `app/main.py` 作为启动装配层，职责仅限依赖注入、handler 注册、应用启动；不承载 FR 业务规则与治理算法。
2. 组合根可 import L1/L2/L3/L4 进行装配，但不得替代任一层做跨层透传逻辑。
3. 代码审查时将组合根视为"装配例外区"：允许装配 import，不允许业务分支与数据写入语义。

### 2.4 调用约束

#### 2.4.1 静态依赖矩阵（代码 import 方向）

> 本矩阵仅描述**静态 import 方向**，不等于运行时调度链方向。  
> 只允许打 ✅ 的方向 import，打 ❌ 的方向连类型引用都禁止。  
> 说明：L0 属于触发语义层（时间信号/外部回调），不承载可被 import 的业务代码，不进入静态矩阵。

| 调用方 ╲ 被调用方 | L1 | L2 | L3 | L4 |
|---|:---:|:---:|:---:|:---:|
| **L1** | ✅ | ✅ | ❌ | ❌ |
| **L2** | ❌ | ✅ | ✅ | ❌ |
| **L3** | ❌ | ❌ | ✅ | ✅ |
| **L4** | ❌ | ❌ | ❌ | ✅ |

**解读**：
- L1 只能调 L2，不可直连 L3 或 L4。
- L2 只能调 L3，不可直连 L4。
- L3 可下调 L4（做落库/外发）；L3 触发 L2 的运行时执行必须通过 `handler_id` 回调注册表分发，禁止 L3 直接 import L2 具体业务模块。
- L2 的只读数据访问通过 Port 契约对象完成：L2 仅依赖契约类型，L4 提供实现并由组合根注入；L2/L4 之间不存在静态互相 import。
- L4 仅暴露经组合根注入的契约实现，不被上层直接静态 import。

**治理端口例外（L1→L3 两种许可路径，此外均禁止）**：
> **路径①（频控）**：L1 接入层的频控拦截入口（429 响应）允许通过 `RateLimitPort`（契约接口，由组合根注入 L3 实现）调用 L3 频控/限流治理能力——**仅限原子计数查询与写入，不得触达任何业务逻辑**。  
> **路径②（jti 鉴权）**：L1 每次请求鉴权时需检查 access_token 的 jti 是否在 L3 黑名单中，允许通过 `TokenRevocationPort`（契约接口，由组合根注入 L3 实现）向 L3 查询 jti 黑名单——**仅限只读查询 jti 有效性，命中则 401 拒绝，不得触达其他 L3 治理能力**。  
> 以上两条路径均须通过 Port 契约而非直接 `import governance/*` 具体模块实现；L1 禁止通过这两个 Port 访问锁/事务/熔断等其他 L3 能力。**其余所有 L1→L3 路径均禁止。**

#### 2.4.2 运行时标准路径（九类典型场景）

| 场景 | 标准路径 | 关键约束 |
|---|---|---|
| **用户/API 请求** | L0→L1(鉴权+校验)→L2(业务编排)→L3(幂等+事务)→L4(读写)→逐层返回 | L1 不可直连 L4；Free 用户高级区摘要由 L2 裁剪后下发，禁止 LLM 二次总结 |
| **只读查询链**（无治理副作用） | L0→L1→L2(查询编排+ReadPort 抽象调用)→L4(ReadPort 实现)→L2(聚合)→逐层返回 | 仅限只读；不得在此链路执行事务写入、外发、幂等写操作；该路径表示运行时端口分发，不代表允许 L2→L4 静态 import；**缓存例外**：若查询涉及 L3 管理的 TTL 缓存（如 FR-05 市场状态缓存 TTL 至次日 09:00、FR-10 首页数据 TTL 5 分钟），路径为 L2→L3(缓存读取/TTL 判定/回退)→L4(缓存未命中时读库)→L2(聚合)，此场景仍属只读但经过 L3 |
| **日批主链路**（研报→结算→模拟） | L0(DAG 触发)→L3(调度/互斥锁/handler_id 分发)→L2(业务编排)→L3(事务/幂等/断点续传)→L4(落库) | 正常路径仅由"上游完成事件"推进；禁止固定时刻强行触发下游；异常路径允许按 `config.dag_cascade_timeout_before_open` 执行全局兜底终止并链式标记下游 |
| **LLM 调用链** | L2(strategy_type 前置判定→构建 Prompt/先验注入/BEAR 短路/停牌跳过判断)→L3(熔断/限速/全局挂起)→L4(LLM 多级降级适配)→L2(质量判定/辩证审计/参数倒挂校验/降级结论赋值)→L3(事务/降级标记持久化) | strategy_type 由 L2 规则引擎在 LLM 调用前判定（非 LLM 自由发挥），判定结果注入 Prompt 上下文；LLM 熔断由 L3 控制；BEAR B/C 短路与停牌跳过在 L2 的 LLM 调用前执行；业务异常（JSON 解析）重试上限 2 次由 L3 频控；**降级职责分工（03 必须在 FR-06 链路中显式区分）**：L4 返回 LLM 原始结果后**先回到 L2** 完成全部业务判定（质量判定、辩证审计、参数倒挂校验），L2 给出 `quality_flag`/`llm_fallback_level` 等降级结论的最终值，**然后**由 L3 在事务范围内将 L2 已给出的降级标记统一持久化到数据库；L3 不得自行修改或推翻 L2 的降级判定 |
| **OAuth 回调链** | L0(第三方 OAuth 重定向)→L1(state/参数校验)→L2(账号绑定与 Token 编排)→L3(幂等+事务)→L4(OAuth 适配+落库) | 禁止把 OAuth 回调当普通 JWT 用户请求处理；邮箱缺失走 FR-09 回退策略 |
| **支付 Webhook 回调链** | L0(支付渠道回调)→L1(签名校验)→L2(订单/权益编排)→L3(幂等 event_id + 原子事务)→L4(落库/补偿) | 鉴权方式为签名而非 JWT；支付订单与权益发放必须同事务提交（HC-16） |
| **事件推送** | L2(声明推送意图)→L3(Outbox 写入+事务后派发+重试抑制+告警防雪崩)→L4(渠道发送) | 事务提交前禁止外发；回滚时取消派发；Outbox 物理写入由 L3 在事务边界内负责；DRAWDOWN_ALERT 同类抑制 4h 窗口由 L3 控制 |
| **管理写操作** | L0→L1(JWT+RBAC admin/super_admin)→L2(业务编排+级联检查)→L3(审计门禁+事务)→L4(落库) | 无审计记录则写入失败；强制重建须 super_admin 且无 FR-08 持仓引用（级联阻断） |
| **清理归档** | L0(定时)→L3(调度/隔离/handler_id 分发)→L2(清理策略判定：永久保留 vs 定期清理)→L3(事务控制)→L4(删除/归档) | 不阻塞当日交易主链路；关键研报/行情/持仓永不删除；僵尸任务先转 Expired 再清理；L3→L2 必须走注册表回调，不得静态调用 |

#### 2.4.3 L3→L2 运行时回调模式（消除“三明治调用”歧义）

> 目标：保持 `L3 不静态 import L2`，同时允许 `调度(L3)→业务(L2)→治理(L3)` 的运行时链路。

1. 组合根（`main.py`/启动装配）将 L2 handler 注册到 L3：`handler_id -> callable`。
2. L3 调度器只持有 `handler_id` 与回调注册表，不引用 L2 具体模块类型。
3. L3 执行任务时按 `handler_id` 取回 callable 并调用；清理归档等链路中的 `L3→L2` 也必须沿用同一注册表机制。
4. handler 内部如需写操作，只能调用 L3 暴露的治理能力（事务、幂等、锁、熔断）；禁止在 L2 handler 内自建锁表/重试器/熔断器。
5. 代码审查以静态 import 为准：`governance/*` 不得 import `services/*`；`main.py` 仅允许装配 import，不允许写业务分支。

### 4.6 行动 09 写作模板约束（03 必须遵守）

> 来源：99 行动 09。02 在此仅定义“03 必须交付的结构模板”，不定义字段细节。

**03 编写顺序（强制）**：
1. 先写 `03 §0` 全局错误/降级表示标准（错误码优先回链 05；若 05 尚未完成，仅允许引用 01/99 已出现错误名作临时占位并标注“待 05 回链”，禁止新增自定义错误码）。
2. 再按 `§4.5` 清单逐条写 FR 执行链路/状态机。
3. 最后做跨 FR 一致性复检（统一错误结构、统一降级结构、统一终态口径）。
4. 03 章节命名固定为 `§FR-xx`（含 `§FR-09-b`）；02 中所有“状态迁移→03 §FR-xx”回链均以此命名为准。

**每条 FR 链路至少包含七要素**：
1. 步骤序列（步骤名 + 调用对象 + 起点触发事件 + 终态收敛）
2. 输入来源与输出去向（优先回链 05/04；若 05/04 尚未完成，则临时回链 01 对应 FR 条款并标注“待 05/04 回链”，禁止在 03 自创字段结构）
3. 失败处理策略（抛错/降级/跳过/重试）
4. 超时阈值与超时后动作
5. 步骤关系（串行/并行/前置条件）
6. 全链路失败返回结构（引用 `03 §0` 统一标准）与“是否阻塞其他功能/下游链路”判定
7. 观测点（至少一个状态落库点 + 一个告警触发点）

**03 禁止项（来自 99 行动 09 + 行动 11）**：
- 各 FR 各自定义互不兼容的错误/降级结构
