## 4. 执行步骤（数据采集链路）

### 4.1 T日全量数据采集流程（FR-04 触发）

```text
步骤 1：触发（FR-02 调度器 09:35 或手工触发）
  → 创建 data_batch（trigger_type=scheduled/manual，status=RUNNING）
  → 写 scheduler_task（task_name=hotspot_collect，status=RUNNING）

步骤 2：行情采集（并行 or 顺序，L4 数据适配层）
  → tdx_local：读 .day 文件 → 写 kline_daily（adjust_type=front_adjusted）
  → eastmoney：httpx 补充/校验 → 若 tdx_local 失败则用 eastmoney 兜底
  → 采集成功 → data_usage_fact（status=ok）
  → 采集失败 → data_usage_fact（status=degraded，status_reason=超时/文件不存在）

步骤 3：热搜采集（L4，按 source_name 逐一执行）
  → 成功 → hotspot_top50（quality_flag=ok）+ data_usage_fact（status=ok）
  → Cookie 失效 → 跳过，不写 hotspot_top50，data_usage_fact（status=missing）
  → 格式错误 → data_usage_fact（status=degraded）

步骤 4：质量融合（L3，采集完成后）
  → 按 trade_date 聚合各 source 的 quality_flag
  → 融合规则：`degraded > missing > stale_ok > ok`（取最差级）
  → 融合后设置 data_batch.status：
    全部 ok → COMPLETED
    部分失败 → PARTIAL
    全部失败 → FAILED

步骤 5：血缘关联（研报生成时 FR-06 触发）
  → 研报生成时读取本批次的 data_usage_fact
  → 写 report_data_usage_link（report_id, usage_id）
  → 写 report.quality_flag（继承采集层最差 quality_flag）
```

### 4.2 冷启动处理规则

| 场景 | 处理方式 |
|------|---------|
| `kline_daily` T-1 数据缺失 | 回退至最近可用日期（最多 T-5），写 `status_reason=stale_fallback` |
| 无任何历史行情 | `scheduler_task.status=SKIPPED`，`status_reason=cold_start_no_data` |
| `stock_pool` 无历史记录 | FR-01 降级为全市场 Top200 流动性池（cold_start_blocked=true） |

## 5. 降级规则与质量融合

### 5.1 quality_flag 枚举与语义

| 枚举值 | 语义 | 研报 published 规则 |
|--------|------|-------------------|
| `ok` | 数据完整、时效正常 | 可发布（`published=1`） |
| `stale_ok` | 数据使用 T-1 缓存，业务可接受 | 可发布，需标注数据时效 |
| `missing` | 某些数据源缺失，但保留基础数据 | 可发布，置信度下调 |
| `degraded` | 数据严重降级，行情/热搜数据不可靠 | 降级发布（`published=0` 或仅内部），`llm_fallback_level` 记录降级程度 |

### 5.2 多源质量融合规则（数据采集层）

**融合优先级（取最差级）**：`degraded > missing > stale_ok > ok`

```text
inputs: List[quality_flag per source_name]
output: consolidated quality_flag

if any source is "degraded":   return "degraded"
elif any source is "missing":  return "missing"
elif any source is "stale_ok": return "stale_ok"
else:                          return "ok"
```

> 此规则适用于：`data_batch.status` 计算、`report.quality_flag` 继承

### 5.3 LLM 降级层级与 llm_fallback_level

| 层级 | 含义 | 触发条件 |
|------|------|---------|
| `primary` | 主 LLM（ai-api 优先模型） | 正常响应 |
| `secondary` | 备用 LLM（ai-api 备选模型） | 主 LLM 超时或失败 |
| `cli` | CLI 工具模式 | 前两层全失败 |
| `local` | 本地 Ollama 模型 | 前三层全失败 |
| `failed` | 全部失败 | 全链路不可用 → 研报不发布 |

## 6. 阈值与 SLO

### 6.1 数据质量阈值

| 指标 | 阈值 | 说明 |
|------|------|------|
| 核心池行情覆盖率 | ≥ 90%（180/200只） | 低于此值整批标记 `degraded` |
| 热搜数据源最低存活数 | ≥ 2 个 source 正常返回数据 | 全失败时降级为仅行情模式 |
| Cookie 刷新周期 | 每 4 小时探测一次 | 连续 2 次失败 → 状态转 `EXPIRING`；3 次失败 → `EXPIRED` |
| kline_daily 最大容忍 stale 天数 | 5 个交易日 | 超出则拒绝使用历史缓存 |

### 6.2 研报生成 SLO

| 指标 | 目标值 | 监控方式 |
|------|--------|---------|
| 研报生成成功率 | ≥ 95%（每交易日） | scheduler_task 统计 |
| LLM 主通道响应时间 | P99 ≤ 30s | NFR-13 告警 |
| 单股研报端到端延迟 | ≤ 60s | scheduler_task + report 时间差 |
| 模拟结算 SLO | 各 FR-07 结算任务在 window_expire_date T+1 09:00 前完成 | scheduler_task 监控 |

### 6.3 模拟盘 SLO

| 指标 | 目标值 | 说明 |
|------|--------|------|
| sim_account 更新延迟 | T+1 09:00 前完成 | FR-08 开盘前结算上一日 |
| 最大持仓数硬限制 | 见 sim_account.sim_max_positions | 超出则拒绝新指令（skip_reason=max_positions_reached） |
| 回撤保护触发阈值 | max_drawdown_pct ≥ 0.20 | 触发 drawdown_state=HALT |

## 7. 失败处理规则

### 7.1 数据采集失败

| 失败场景 | 处理方式 | 记录字段 |
|---------|---------|---------|
| tdx_local .day 文件缺失 | 回退 T-1 缓存（最多5日） | `data_usage_fact.status=stale_ok` |
| eastmoney API 超时（3次重试后） | 标记 degraded | `data_usage_fact.status=degraded` |
| Cookie 平台数据失败 | 跳过该 source，不阻塞其他 source | `data_usage_fact.status=missing` |
| 全部数据源采集失败 | data_batch.status=FAILED；当日研报不生成 | `scheduler_task.status=FAILED` |

### 7.2 研报生成失败

| 失败场景 | 处理方式 | 记录字段 |
|---------|---------|---------|
| LLM 全链路失败 | report 不写 published，report_generation_task.status=FAILED | `llm_fallback_level=failed` |
| 风控审计阻断 | report.risk_audit_status=blocked，不发布 | `published=0`，`status_reason` 填拦截原因 |
| 股票停牌 | report_generation_task.status=SKIPPED | `skip_reason=suspended` |
| 幂等键冲突（T日已有研报） | 跳过生成，复用已有 report_id | `skip_reason=idempotency_exists` |
| 最大重试次数（3次）耗尽 | report_generation_task.status=FAILED | `error_message` 记录最后一次错误 |

### 7.3 结算失败

| 失败场景 | 处理方式 |
|---------|---------|
| 结算窗口到期但数据缺失 | settlement_status=SUSPENDED_EXIT，exit_reason=suspended_exit |
| 行情数据无法获取（停牌） | 延迟结算至复牌后第一个交易日 |
| 结算计算异常（除零等） | settlement_status=FAILED，error 写 audit_log |

### 7.4 模拟盘失败

| 失败场景 | 处理方式 |
|---------|---------|
| drawdown_state=HALT | 拒绝所有新开仓指令（skip_reason=drawdown_halt） |
| 可用现金不足 | sim_trade_instruction.status=SKIPPED，skip_reason=insufficient_cash |
| 最大持仓数已满 | skip_reason=max_positions_reached |

## 8. 特殊数据治理规则

### 8.1 安全敏感字段规则

| 字段类型 | 存储规范 | 禁止行为 |
|---------|---------|---------|
| password_hash | bcrypt/argon2哈希，不可逆 | 禁止明文存储、禁止日志输出 |
| cookie_session | 加密存储（AES-256-GCM，密钥来自环境变量） | 禁止明文、禁止日志 |
| refresh_token | SHA-256 哈希值存储 | 禁止明文、禁止日志 |
| access_token_hint | 仅最后4位或哈希 | 禁止完整 token 存储 |
| JWT jti | 原值存储（本身非敏感） | 不得与 user PII 关联展示 |

### 8.2 数据保留策略

| 数据类型 | 保留周期 | 清理方式 |
|---------|---------|---------|
| `jti_blacklist` | JWT exp + 1日（到期自动清理） | FR-09-b 定时任务 |
| `audit_log` | 永久保留（Append-Only） | 不清理 |
| `outbox_event`（DISPATCHED） | 7日后可归档 | FR-09-b 定时任务 |
| `kline_daily` | 永久保留（历史因子计算需要） | 不清理 |
| `experiment_log` | 永久保留（质量回溯需要） | 不清理 |
| `scheduler_task`（SUCCESS > 30日） | 可归档 | FR-09-b 可选清理 |

### 8.3 幂等性保障规则

| 表 | 幂等键 | 冲突处理 |
|----|--------|---------|
| `report` | `idempotency_key`（trade_date:stock_code） | 已存在则返回已有 report_id，不重复写 |
| `settlement_result` | `(report_id, window_days)` | 已存在则跳过，不覆盖 |
| `order` | `event_id`（支付回调唯一ID） | UPSERT：已有则比较 status，冲突记录审计日志 |
| `data_batch` | `batch_id`（UUID，调用方生成） | 已存在则返回现有状态，不重复创建 |

## 9. 责任分工

| 层 | 职责 | 代表模块 |
|----|------|---------|
| **L0 触发层** | 定时/事件触发，不直接读写业务表 | `apscheduler`、`webhooks` |

| **L4 资源适配层** | 数据采集（kline/hotspot）、LLM 调用、ORM 操作 | `tdx_adapter.py`、`llm_client.py`、SQLAlchemy models |

| 表分类 | 写操作归属层 | 读操作归属层 |
