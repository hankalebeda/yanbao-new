## 1. 产品范围

### 1.1 系统边界

- **目标**：A 股个股研报平台，每日对全量A股进行基础信息与日线行情采集监控，但仅精确对**200只核心池股票**生成专门研报（含买卖建议、实操指令卡），通过模拟持仓验证收益。
- **核心能力**：完整站点（首页/列表/研报详情/统计看板/模拟收益看板）；数据与推理透明；四维度绩效（真实胜率≥55%、盈亏比≥1.5、年化Alpha≥10%、最大回撤≤20%）；绩效统计必须同时展示样本数与覆盖率。**核心池需针对 1W、10W、50W 三个资金挡位分别出具持仓建议，并分挡位进行真实佣金约束下的双边模拟与独立统计卡片追踪。**
- **范围**：针对全量A股（约5000+）的基础数据处理；核心池(200只)的日级主链（单股单日仅一次正式报告）；研报与预测评估；**OAuth/第三方登录（QQ、微信）全功能实现（首版即完成标准授权码流程，如缺真实参数可挂载高仿Mock端，严禁只留501占位）**（见 FR-09）；**完整订单支付及Webhook回调链路实现（首版即闭环，严禁只留501占位）**。
- 系统支持研报质量与收益的持续改进，方法论与实现设计见 03_详细设计 §10 研报质量与收益提升设计。

### 1.2 用户角色

| 角色 | 能力 |
|------|------|
| **普通用户（Free）** | 浏览研报、查看结论与实操指令、查看模拟收益看板 |
| **付费用户（Pro/Enterprise）** | 完整研报、高级区推理链、更多历史数据（权益差异见 1.3） |
| **管理员** | 系统配置、告警接收、运营看板 |

### 1.3 权益能力矩阵（Free / Pro / Enterprise）

> 详细权益约束见 FR-09；接口字段见 05_API 对应 FR-09 章节（01文档为权威来源，FR-09/05_API为实现规格细化）。

| 能力 | Free | Pro | Enterprise |
|------|------|-----|------------|
| 研报结论与实操指令 | ✓ | ✓ | ✓ |
| 高级区推理链 | 摘要/隐藏（后端裁剪≤200字符，详见 FR-10 边界） | 完整 | 完整 |

- **资金挡位 SSOT 约束**：`capital_tier` 规范枚举固定为 `10k|100k|500k`（接口/数据库/测试统一使用）；前端中文展示映射为 `1W|10W|50W`。系统实现须以全局常量/配置源（如 `config.CAPITAL_TIERS`）动态遍历；FR-06/08/10/12 等涉及挡位的字段解析与分支逻辑禁止散落硬编码字符串，新增挡位时不得要求多处业务逻辑手改。

### 1.4 股票池与全量A股监控

- **来源**：股票池由系统**自行生成**。基础监控池覆盖全量A股（剔除退市）；基于严密的选入规则与排序机制，动态筛选出**精确的 200 只核心池股票**纳入研报生成。不以环境变量或外部脚本为主要来源。
- **更新频率**：每个交易日开盘前更新；长尾采集任务覆盖全量股票，但研报生成任务与列表页默认范围均以当日**核心池 (200只)** 为准。
- **规则维度**：选入（全量全扫描计算指标）、数量（核心池名额严格锁定为 200，依据市值/流动性/特定强信号打分排序取 Top 200）、退出（核心池末位淘汰、跌停/停牌/ST踢出并由候补填补）；规则见 FR-01，执行规格见 03_详细设计。
- **候补机制**：打分排序 Top 201～250 为候补池；核心池出现空缺时，从候补池按分数从高到低依序填补；填补后立即更新 `pool_version`；**被淘汰（踢出）的股票代码记录于 `evicted_stocks: List[str]` 字段**（`fallback_from` 字段含义为上一有效 pool_date 日期字符串，详见 FR-01 边界）。

### 1.5 核心使用场景

1. 用户打开首页 → 进入列表（数据来自股票池）→ 选择股票查看研报详情（含结论、实操指令卡、推理过程）。
2. 定时任务每日生成研报（按股票池顺序）→ BUY 强信号自动触发模拟开仓 → 持仓追踪直到止盈/止损/超时。
3. 用户查看统计看板、模拟收益看板：**Free 仅可查看 10W 档位；Pro/Enterprise 可切换查看 1W、10W、50W 三个资金挡位的独立绩效曲线与风险暴露情况**，验证系统在不同资金规模下的兼容性。资金挡位底层统一使用 `10k/100k/500k` 枚举，展示层映射为 `1W/10W/50W`。

### 1.6 研报准确率与后续优化

研报是系统的基础产出物，准确率直接决定四维度绩效的达成。系统支持研报质量的持续改进，路径如下：

1. **发现问题**：FR-07 结算误判、FR-11 用户反馈、FR-12 管理员标记产生误判与负样本。
2. **归因分类**：信号错、执行错、数据错、事件误读、市场状态错（见 03_详细设计 §10.5）。
3. **改进动作**：Prompt 修改、Few-shot 样本库注入、阈值/参数调整、市场状态机调优。
4. **验证上线**：回测沙盒验证、实验门禁（walk-forward、泄漏检查），满足绩效门槛或至少不劣化才可上线；上线后 7 日内劣化则回滚（见 03_详细设计 §10.6）。

Few-shot 样本库、信号增强（多模型投票/自我批评）、回测沙盒等机制详见 03_详细设计 §10。参数/Prompt/模型变更须通过 03 定义的实验门禁后上线（NFR-15）。

### FR-00 真实性红线（强制）

| 维度 | 内容 |
|------|------|
| **功能域** | 数据与展示真实性 |
| **触发** | 无单独触发；所有数据采集、研报生成、展示链路均受约束 |
| **输入** | 无（约束性规则） |
| **输出** | 无直接输出 |
| **术语** | `citations`（引用列表：每条证据须有 `source_name`、`source_url`、`fetch_time`；`fetch_time` 为 ISO8601；`source_url` 为可访问 http(s) URL） |
| **边界** | 禁止伪造行情/热搜/新闻/政策 → 违反则不允许发布；研报每条证据须含 `source_name`、`source_url`、`fetch_time`；**核心来源最小门槛**：至少包含「日线行情（kline_daily）」+「外部证据（citation）」；若任一核心来源不可用 → 显式报错或降级标记（`quality_flag!=ok` 且 `status_reason` 非空），不允许静默填充假数据；前台展示 → 须为中文业务语义，不直接暴露英文字段键；研报页或页脚 → 须有显式风险提示与免责声明；**已发布研报只读约束**：`published=true` 的研报内容（结论/citations/推理链）禁止事后覆盖写入；`PATCH /api/v1/admin/reports/{id}` 仅允许修改 `review_flag`/`published`（下架），不允许修改业务内容字段；**强制重建例外（Force Regenerate Issue）**：仅 `role=super_admin` 可通过受审计专用流程触发同日同股物理重建（软删除旧脏数据并生成新 report_id），用于已发布脏数据纠错；该流程必须显式携带 `force_regenerate=true` 与 `reason_code`，并记录完整审计链路；**关键操作审计**：研报发布（`published` 从 false→true）、下架（true→false）、降级恢复、强制重建等操作均须记录 `{ actor_user_id, action_type, timestamp, request_id, before, after }`，可追溯 |
| **验收** | pytest 抽检 `citations` 非空且每条含 `source_name`、`source_url`、`fetch_time`（ISO8601 可解析，`source_url` 非空且为 http(s)）；人工检查页脚含免责文案；**只读断言**：对已发布研报 `PUT /api/v1/reports/{id}` 尝试修改 `recommendation` 字段 → 返回 405 或 403 且内容不变；**审计断言**：管理员调用 `PATCH /api/v1/admin/reports/{id}` 修改 `published` 后，审计表中可查到含 `actor_user_id` 和 `before`/`after` 的记录 |
| **优先级** | P0 |
| **相关** | 05_API、04_数据治理 |

### FR-01 股票池筛选

| 维度 | 内容 |
|------|------|
| **功能域** | 股票池管理（自生成与全量监控分级） |
| **上下游** | 下游：FR-04（全量采集）、FR-06（仅限核心池研报）、FR-10（列表） |
| **触发** | 定时任务（每日开盘前）/ 手动刷新 / 系统启动时加载 |
| **接口** | 管理端手动刷新：`POST /api/v1/admin/pool/refresh`（请求体：`{ trade_date?: str, force_rebuild?: bool }`；默认 `trade_date=今日交易日`、`force_rebuild=false`） |
| **输入** | `trade_date: str`（YYYY-MM-DD）；`filter_params: dict`（必含 `min_market_cap_cny: float`、`min_listing_days: int`、`min_avg_amount_20d_cny: float`、`exclude_st: bool`、`exclude_limit_down_days: int`、`target_pool_size: int`；可选 `preferred_industries?: List[str]`、`excluded_industries?: List[str]`、`max_single_industry_weight?: float`（0~1））；`universe_stocks: List[BaseInfo]`（全A基础信息）；`universe_kline_20d: Dict[str, List[KlineDaily]]`（全A近20日行情） |
| **规则维度** | **基准覆盖**：全量A股（约5000+），排除退市；**选入核心池**：市值下限 **≥50亿**、上市时长>1年、日均成交额>**3000万**，并应用行业约束（`preferred_industries` 白名单优先、`excluded_industries` 黑名单剔除、`max_single_industry_weight` 控制单行业占比上限）；排除 ST/*ST/退市、近 **5** 日连续跌停；**数量要求**：核心池精确打分排序提取 **200** 只；**退出**：ST、连续跌停、停牌超 **10** 日、打分跌出 Top 200 触发末位淘汰 |
| **打分因子** | **八因子加权评分（总分=各因子得分×权重之和，满分100）**，打分前先对各因子进行**全市场百分位标准化（normalize_rank）**，使每因子输出0~1之间的分位值，再按权重合成：<br>1. **近20日价格动量**（权重20%）：近20交易日累计收益率，越高越好，线性映射至[0,1]；<br>2. **市值适中度**（权重15%）：偏好中等市值（50亿～1000亿流通市值），太小流动性差降分，太大弹性不足降分，用倒U型分段计分（50亿以下=0，50~100亿线性升至0.8，100~1000亿固定0.9，1000亿以上线性降至0.3）；<br>3. **流动性（日均成交额分位）**（权重20%）：过去20日日均成交额在全市场的分位，越高越好，直接用分位数；<br>4. **MA20斜率（趋势强度）**（权重15%）：用`(MA20今日 - MA20五日前) / MA20五日前`计算均线斜率百分比，斜率>0加分，做全市场分位排名；<br>5. **近60日业绩预期改善**（权重10%）：最近一个季报/年报期与预期相比是否超预期或业绩正向修正，有超预期=1，无=0.5，业绩下修=0；当未配置此专业项数据源流入时，永远采用 0.5 作为缺失状态中立回退；<br>6. **换手率舒适度**（权重10%）：偏好适中换手率（日换手率1%~5%为佳），过低（<0.3%）=0.3，过高（>10%投机气氛）=0.2，正常范围线性计分；<br>7. **近5日相对强弱（RSI缓和区）**（权重5%）：RSI在40~70之间为佳（既非超买也非超卖），RSI>80或RSI<20=0，40~70线性=1；<br>8. **股价距52周高点回撤比**（权重5%）：`(当前收盘价 / 52周最高价 - 1)`，回撤越小说明强势股，线性映射；<br>**合成公式**：`score = Σ(normalize_rank(factor_i) × weight_i) × 100`；所有因子均基于最新交易日数据，因子数据缺失时该因子取0.5（中性，不因缺失而惩罚或奖励） |
| **输出** | **管道落库输出**：全量股票字典/列表 `Dict[str, BaseInfo]`；核心池 `core_pool: List[str]`（精确200只）；候补池 `standby_pool: List[str]`（Top201~250，按分数排序）；`pool_date: str`（YYYY-MM-DD）；`pool_version: int`；`scores: Dict[str, float]`；`evicted_stocks: List[str]`（本次从核心池淘汰的股票代码，可为空）。**HTTP 同步响应（NFR-14）**：`HTTP 200 + { success: true, request_id: str, data: { pool_date: str, pool_version: int, core_pool_size: int, standby_pool_size: int, fallback_from?: str, evicted_stocks?: List[str] } }` |
| **边界** | 规则执行失败或数据不可用 → 回退到上一交易日池 + 告警（`fallback_from: str` 必填，含义为上一有效 pool_date 的日期字符串 YYYY-MM-DD，非被替换股票代码；被淘汰股票代码记录于 `evicted_stocks: List[str]` 字段）；筛选结果为空 → 告警，不阻塞，且同样回退到上一交易日池（`fallback_from` 必填）；ST/*ST/退市股 → 自动排除；**冷启动兜底**：若需回退但数据库不存在任何历史池（首次部署/清库后首日），必须阻断下游依赖（FR-04/FR-06），返回 `error_code=COLD_START_ERROR`，并按 NFR-13 发送最高级告警；**系统重启当日如已执行过筛选（`pool_date` == 今日）→ 直接加载缓存，不重复计算，保持幂等**；**并发刷新互斥**：同一 `trade_date` 的 pool 刷新执行期间必须持有排他锁（Redis 锁或 DB 行锁等）；并发手动/定时请求必须拒绝（`HTTP 409` 或 `429`），避免同日双写与版本竞争；**候补池填补时机：仅在每日开盘前 pool 刷新时执行（非盘中实时）；当日研报生成期间核心池不动态调整，避免研报中途切换目标股票** |
| **验收** | pytest 断言：1. 给定规则下 `core_pool` 不含ST；2. `len(core_pool) == 200`（精确200）；3. `len(standby_pool) >= 1`（候补池非空）；4. `standby_pool` 中所有股票 score < min(core_pool scores)（候补分数低于核心池最低分，排序正确）；5.  筛选失败时回退到上一交易日池且 `fallback_from` 非空；6.  **并发互斥断言**：同一 `trade_date` 下并发触发定时+手动刷新时，仅一个请求成功，另一个返回 `409/429` 且不产生第二份 `pool_version`。 |
| **优先级** | P0 |
| **相关** | 05_API（内部/配置）、04_数据治理 |

### FR-02 定时调度

| 维度 | 内容 |
|------|------|
| **功能域** | 任务调度 |
| **上下游** | 下游：FR-05（09:00 市场状态）、FR-01（开盘前股票池）、FR-04（热搜）、FR-06（研报）、FR-07/FR-08（结算） |
| **触发** | 系统启动加载配置 / Cron 表达式触发 |
| **输入** | 调度配置（时区 `Asia/Shanghai`；时间触发：市场状态 09:00、热搜按小时；**事件依赖触发（DAG）**：FR-06 等待 `FR-04核心池采集完成` 事件，FR-07/FR-08 等待 `FR-06批次完成` 事件；FR-06 可配置日内软超时告警阈值如 16:00；补充配置项 `config.dag_cascade_timeout_before_open: str`（HH:MM 格式，默认 `"08:30"`，表示次日开盘前 30 分钟为 DAG 全局兜底超时死线）；**FR-07/FR-08 不设 16:30 日内硬死线**，最晚与 NFR-04 对齐至次日开盘前） |

| **触发** | POST /api/v1/reports/generate (参数校验拦截非核心池) 或定时任务(遍历核心池200只) |
| **输入（HTTP Request）** | `stock_code: str`（须在当日200只核心池白名单内，格式如 `600519.SH`）；`trade_date: str`（YYYY-MM-DD，默认当日交易日） |

| **输出** | 对 200 只核心股票产出：`report_id: str`（UUID）；`idempotency_key: str`（格式：`daily:{stock_code}:{trade_date}`，如 `daily:600519.SH:2026-03-03`）；`published: bool`（成功=true；失败仍入库但 published=false，quality_flag=degraded，不对外展示）；`recommendation` 枚举 `BUY|SELL|HOLD`；`confidence: float`（0.0~1.0）；`quality_flag: str` 枚举 `ok|stale_ok|degraded`；`llm_fallback_level: str` 枚举 `primary|backup|cli|local|failed`；`citations: list`；`strategy_type: str` 枚举 `A|B|C`；`instruction_card: { signal_entry_price: float, atr_pct: float, stop_loss: float, target_price: float }`；`sim_trade_instruction: Dict[str, TradeInstruction]`（固定含 `10k|100k|500k` 三个key，且每挡结构为 `{ status: "EXECUTE|SKIPPED", position_ratio: float, skip_reason?: str }`：当 `BUY and confidence>=0.65` 时按挡位给出 `EXECUTE|SKIPPED` 与对应 `position_ratio`；其余情况三挡统一 `status="SKIPPED"`、`position_ratio=0.0` 且 `skip_reason="LOW_CONFIDENCE_OR_NOT_BUY"`）；（非核心池传入时）拒绝处理返回 400 error_code=NOT_IN_CORE_POOL |
| **质量口径补充（冻结）** | `quality_flag` 采用**分域口径**：采集数据域（FR-04）允许 `missing`；研报输出域（FR-06）使用子集 `ok|stale_ok|degraded`。跨源融合遵循“向下对齐”原则：任一核心来源不可用且无回退时最终为 `degraded`；存在可追溯回退时可为 `stale_ok`；全部核心来源可用才为 `ok`。 |
| **LLM 降级** | 主 API → 备 API → CLI 网关（如 Codex/Gemini CLI）→ 本地模型（Ollama）分级降级；每次调用必须记录 `llm_fallback_level`（`primary|backup|cli|local|failed`）。当进入 `local` 级别时，研报仍可发布但必须显式标注降级来源；若所有层级均不可用则该股当日研报生成失败，`llm_fallback_level="failed"`、`quality_flag=degraded` 且 `status_reason` 必填，不静默跳过。由于 AI 生成不支持大量并行，强制采用**单线程串行或低配并发队列（Queue）机制**缓慢出具，应对 200 只核心池 |
| **强信号定义** | **`recommendation=BUY AND confidence≥0.65`** 为强信号 → 产出 `sim_trade_instruction` 并触发 FR-08 开仓；`BUY + confidence<0.65` → 仅发布研报，不产出指令卡，不触发 FR-08；`SELL/HOLD` → 不触发开仓 |
| **先验数据注入规则** | 研报引擎在构建 Prompt 前，必须从 FR-07 **已结算历史数据**中计算 `prior_stats` 并注入，具体规则：1. **数据来源**：只能来自 FR-07 `settlement_status=settled` 的历史研报结算记录，**禁止使用任何独立回测或参数优化产生的数据**，防止回测嵌套污染；2. **时序切割（Time-Series Split）**：仅使用 `signal_date < data_cutoff` 的已结算记录（`data_cutoff = trade_date.replace(day=1)`，即 trade_date 所在月的1日；前月及更早），确保先验数据与当前研报在时间上严格隔离，不含任何未来信息；3. **按信号类型分组**：`prior_stats` 按 `strategy_type`（A/B/C）独立统计，当前研报注入与其 `strategy_type` 对应的先验数据；4. **最小样本门槛**：若某 `strategy_type` 的已结算样本数 < 30 笔 → `prior_stats=null`，Prompt 中不注入该字段，LLM 须在无先验数据的条件下独立推理，**禁止用小样本先验数据**；5. **先验数据结构**：`{ strategy_type: str, sample_count: int, win_rate_historical: float, avg_profit_loss_ratio: float, data_cutoff: str（YYYY-MM-DD，本月1日）}`；6. **Prompt 措辞约束**：先验数据在 Prompt 中必须以"以下为本系统已结算历史研报统计（截至{data_cutoff}），**仅供参考，不代表未来**"开头，LLM 只能将其用作量级参照，不得直接以历史胜率作为当前置信度依据。 |

| **边界补充（辩证审计）** | 当主模型输出 `recommendation=BUY AND confidence≥0.65` 时，研报引擎须用同一数据集调用第二个低成本 LLM 端点（`config.llm_audit_provider`，`config.llm_audit_enabled=true` 默认）执行风险审计。副模型只输出风险点，不输出买卖建议。处理规则：1. 若副模型识别风险点中有任意一项未被主模型推理链正面讨论，则在高级区追加"⚠️ 风险补充审计"章节；2. 若副模型失败（超时/API不可用）则跳过审计，主结论照常发布，并记录 `risk_audit_status` 枚举 `completed\|skipped\|not_triggered`，且 `risk_audit_status=skipped` 时必须写 `risk_audit_skip_reason`。设计原则固定为辩证式（主推多/副审风险），禁止投票式通过门槛。 |

### FR-07 预测结算与回灌

| 维度 | 内容 |
|------|------|
| **功能域** | 预测结果结算与绩效统计 |
| **上下游** | 上游：FR-06；下游：FR-10 |
| **触发** | 定时任务（每日）/ 人工触发（管理员） |
| **接口** | 内部批量结算任务提交：`POST /api/v1/internal/settlement/run`（异步受理，返回 `task_id`）；管理员手工结算任务提交：`POST /api/v1/admin/settlement/run`（请求体：`{ trade_date: str, window_days: int, target_scope: str, target_report_id?: str, target_stock_code?: str, force?: bool }`，默认 `force=false`）；任务查询：`GET /api/v1/internal/settlement/tasks/{task_id}` |
| **输入** | `trade_date: str`（YYYY-MM-DD）；`window_days: int`（枚举：1/7/14/30/60）；`target_scope: str`（枚举：`all|report_id|stock_code`）；`target_report_id?: str`；`target_stock_code?: str`；`force?: bool`（默认 false）；结算数据源为 `published=true AND quality_flag=ok` 的已发布研报（含10k/100k/500k指令） |
| **输出** | **HTTP 异步受理响应**：`HTTP 202 + { success: true, request_id: str, data: { task_id: str, status: "QUEUED", force: bool } }`；任务完成后通过任务查询接口返回：`{ processed_count: int, skipped_count: int, failed_count: int, force: bool }`。结算统计入库输出：1/7/14/30/60 日**信号结果对比**（含 `return_n1/return_n7/...`）；**费用扣除口径（务必精确，追求真实）**：以 `signal_date` 收盘价 `P_entry` 买入 → 到 `signal_date+N` 交易日收盘价 `P_exit` 卖出（**前复权一致性提取约束（防除权假摔污染）**：`P_entry` 与 `P_exit` 必须在**同一次**全息前复权查询的序列快照上下文中提取；严禁用 T 日的静态买入价记录穿越去比对 T+N 日复权后的大跌离场价，防止拆股等除权行为 100% 误计为巨亏），净收益率计算公式如下：<br>`买入实付 = P_entry × shares × (1 + 0.05%) + max(P_entry × shares × 0.025%, 5元)`（含买入滑点0.05%+买入佣金万2.5最低5元）；<br>`卖出实得 = P_exit × shares × (1 - 0.05%) - max(P_exit × shares × 0.025%, 5元) - P_exit × shares × 印花税（口径见 NFR-05）`（含卖出滑点0.05%+卖出佣金万2.5最低5元+印花税口径见 NFR-05）；<br>`return_nN = (卖出实得 - 买入实付) / 买入实付`；shares 统一以 100 股（1 手）计算（结算为信号层不关心具体资金量，以1手为标准单位）；**`return=0` 判定口径：`abs(net_return_pct) < 0.0001`（万分之一以下视为零收益，如停牌未变动等场景）**；**数据须入库字段 `cost_breakdown_fr07: { buy_price, sell_price, buy_commission, sell_commission, stamp_duty, buy_slippage_cost, sell_slippage_cost, gross_return_pct, net_return_pct }`**；<br>信号统计**按 strategy_type（A/B/C）×结算窗口（1/7/14/30/60日）独立统计**（不按 capital_tier 分挡）；各维度输出：`win_rate`（净收益>0的比率，**排除 return=0 的样本**（口径见上方判定））、`profit_loss_ratio`（平均盈利净收益率 / 平均亏损净收益率绝对值，**排除 return=0 的样本**；样本中盈利笔数=0或亏损笔数=0时置 `null`）、`alpha_annual`（相对沪深300年化超额收益；精确口径：**持仓交易日数 = 该 strategy_type 第一笔 BUY 信号日到最近一笔结算日之间的交易日总数（自然窗口，含无持仓日，非各笔持仓天数之和）**；策略年化收益 = (1+累计净收益率)^(252/持仓交易日数)-1；沪深300年化收益同口径计算；alpha_annual = 策略年化收益 - 沪深300年化收益；**若沪深300同期数据缺失 → alpha_annual 置 null 且 display_hint 标注原因**）、`max_drawdown_pct`；**`coverage_pct` 口径：有效结算信号数 / 当前累计发出BUY信号总数（同strategy_type维度）**；同时输出 `sample_size: int`、`coverage_pct: float`（0~1）、`display_hint: str|null` |
| **与 FR-08 的口径区别** | FR-07 为**信号结算回灌**（定期收盘价静态计算，扣除双程摩擦，以1手为标准单位）；FR-08 为**实际资金模拟**（动态净值复利，含止盈/止损/超时，按实际资金分挡）；口径差异：FR-07 各信号独立、不关心挡位与资金约束；FR-08 受挡位资金额度与净值波动控制；**shares计算基准严格隔离：FR-07 固定 100 股（1手，不关联任何资金挡位）；FR-08 由各挡位 `position_ratio × 挡位资金 ÷ 股价` 取整计算实际手数，与 FR-07 完全独立，两套口径禁止混用** |

### FR-13 业务事件推送

| 维度 | 内容 |
|------|------|
| **功能域** | 业务事件通知（区别于§4范围外的"营销类主推"） |
| **上下游** | 上游：FR-08（模拟结算触发）、FR-06（BUY强信号触发）、FR-11（复审触发）；下游：外部推送渠道 |
| **触发** | 监听「FR-06批次完成的事件（Event）」触发（不局限于特定时刻，适配异步环境）；FR-08 仓位状态变更时触发；FR-11 复审状态切换时触发 |
| **触发条件四类** | **1. 仓位状态变更**：FR-08 模拟盘仓位 `position_status` 从 `OPEN` 变更为 `TAKE_PROFIT`/`STOP_LOSS`/`TIMEOUT` → 触发推送（结算后立即发出，让用户次日开盘前可参考执行真实操作）；**2. 当日BUY强信号产生**：FR-06 每日研报生成结束后，统计当日 `recommendation=BUY AND confidence≥0.65` 的研报列表，汇总推送一次（不对每只股票单独推送，避免骚扰）；**3. 模拟账户熔断告警**：FR-08 `drawdown_state` 进入 `REDUCE` 或 `HALT` 状态时推送管理员；**4. 复审告警**：FR-11 触发 `review_flag=PENDING_REVIEW` 时推送管理员 |
| **输入** | 推送事件类型 `event_type: str` 枚举 `POSITION_CLOSED\|BUY_SIGNAL_DAILY\|DRAWDOWN_ALERT\|REPORT_PENDING_REVIEW`；`payload: dict`（各类型的结构化数据，详见下方）；推送渠道配置 `config.notification_channels` |
| **输出** | 推送记录入库字段：`notification_id: str`（UUID）；`event_type: str`；`triggered_at: str`（ISO8601）；`channel: str`（email/webhook）；`status: str`（sent/failed/skipped）；`payload_summary: str`（内容摘要，供管理员查阅）；推送实体内容按渠道格式适配 |
| **Payload 结构** | **POSITION_CLOSED**：`{ stock_code, stock_name, position_status, capital_tier, actual_entry_price, exit_price, net_return_pct（口径来自 FR-08 模拟持仓，非 FR-07 信号结算）, holding_days, signal_date, close_date }`；**BUY_SIGNAL_DAILY**：`{ trade_date, signal_count: int, signals: [{ stock_code, stock_name, confidence, strategy_type }]（按 confidence 倒序，最多5条）}`；**DRAWDOWN_ALERT**：`{ capital_tier, drawdown_pct, drawdown_state, triggered_at }`；**REPORT_PENDING_REVIEW**：`{ report_id, stock_code, trade_date, negative_count, review_flag, triggered_at }` |
| **推送渠道** | **管理员通道**（必须实现）：Webhook（复用 NFR-13 已定义的 Webhook 机制，`alert_type` 扩展以上枚举）+ 邮件（复用 FR-09 SMTP 基础设施）；**Pro/Enterprise 用户通道**（可选，首版按配置开关控制）：仅 POSITION_CLOSED 和 BUY_SIGNAL_DAILY 两类，推送至用户注册邮箱；`REPORT_PENDING_REVIEW` 仅发管理员通道。**推送渠道配置 `config.notification_channels`**：`{ admin_webhook_url: str\|null, admin_email: str\|null, user_email_enabled: bool（默认false） }` |
| **边界** | **区分说明**：POSITION_CLOSED 推送属于"模拟结算事件通知"，是业务闭环的一部分，**不属于§4范围外的营销类主推**；§4 仍保持排除每日股票推荐等营销行为不变。推送失败（网络异常/SMTP失败）→ 重试一次，失败后仅记录 `status=failed`，**不允许因推送失败阻塞 FR-08 结算主链路**；推送幂等：同一 `（event_type, stock_code/trade_date, capital_tier）` 组合当日只推送一次（重复触发则幂等跳过，`status=skipped`）；用户未配置邮箱或 Pro 权益不符合 → `status=skipped`，记录原因；**事务后派发硬约束（实现固定为 Outbox）**：凡依赖资金/仓位落库结果的事件（尤其 `POSITION_CLOSED`），必须采用 Outbox 模式在数据库事务 `commit` 成功后派发；绝对禁止在内存状态已变更但主体事务未提交时提前发送邮件/Webhook。若事务回滚，则对应事件必须取消且不得产生 sent 记录。**告警防雪崩**：对 `DRAWDOWN_ALERT` 实施同类型抑制策略，同一 `capital_tier` 在 `drawdown_state in (REDUCE, HALT)` 持续期间每 4 小时仅发送首报一次，后续同态告警抑制，直至状态恢复后才重新放行首报；推送内容**禁止包含用户 email/user_id/JWT 等个人身份信息**（遵循 NFR-16）。 |
| **验收** | pytest 断言：1. mock FR-08 仓位变更为 TAKE_PROFIT → `notification` 表有 `event_type=POSITION_CLOSED AND status in (sent/failed)` 记录；2. 同一仓位变更事件二次触发 → 第二条 `status=skipped`（幂等去重）；3. 推送失败不阻塞结算：mock Webhook 超时，FR-08 结算记录仍正常入库，`notification.status=failed`；4. BUY_SIGNAL_DAILY 推送内容中 `signals` 按 confidence 倒序且最多5条；5.  `config.notification_channels.user_email_enabled=false` 时，用户通道所有记录 `status=skipped`；6.  **事务回滚不派发断言**：mock FR-08 在 `sim_account` 落库阶段抛错并触发事务回滚时，`notification` 表不得出现该仓位的 `POSITION_CLOSED/sent` 记录；**事务后派发一致性断言**：事件落库后仅在事务提交成功后进入可发送状态；模拟派发进程中断并恢复后，事件可重试成功且不会重复发送；7.  **DRAWDOWN_ALERT 抑制断言**：同一 `capital_tier` 在 4 小时抑制窗口内重复触发 REDUCE/HALT 时仅首条为 sent，后续为 skipped；状态恢复后再次触发可重新 sent；8.  **REPORT_PENDING_REVIEW 断言**：负反馈触发复审时，`notification` 表产生 `event_type=REPORT_PENDING_REVIEW` 且仅走管理员通道；人工验证：Webhook 收到 POSITION_CLOSED 事件时 payload 含 `net_return_pct` 且格式正确 |
| **优先级** | P1 |
| **相关** | 05_API、04_数据治理、NFR-13 |

## 4. 范围外说明（不做什么）

- **自动下单（绝对红线）**：绝不向券商发送真实委托或尝试自动化交易执行；模拟持仓仅作为系统内部胜率验证的闭环
- **分钟级高频**：仅日级研报
- **非 A 股**：仅支持 A 股
- **非研报功能**：不做社交、论坛、聊天
- **未经约定数据源**：新增数据源须在 01/04/05 显式约定
- **邮件/短信/推送**：不做**营销类**用户主动通知（如每日股票推荐、运营活动推送）；**事务性业务事件通知（如仓位平仓通知、每日BUY强信号汇总、回撤告警、复审告警）属于 FR-13 定义的业务事件推送，不在此范围外**；事务性邮件（注册激活、密码重置）属 FR-09 必要功能，不在此范围外，失败处理见 FR-09 边界。
- **（安全约束）发送给 LLM 的数据**：Prompt 中只含行情/技术指标/热搜标题等业务数据，**禁止包含用户 email/user_id/JWT 等个人身份信息**；API Key 仅存于 `.env`（不入 git），通过 pydantic-settings 读取（见 NFR-16）。
- **研报导出**：不支持 PDF/Excel/Word 导出

### 4.1 实施策略（去除分期，首版即终态全量交付）

- **不再区分首版占位**：FR-09 的 OAuth 与 支付订单闭环等机制必须在本次开发中实现。缺秘钥可接本地全真Mock引擎，必须跑通真实闭环。
- **全量监控与核心研报隔离**：数据源必须能够承载全量A股采集的数据量，而 LLM 算力消耗必须严格把控在 200 只核心池以内。

### 6.1 FR 编号追加规则（必须遵守）

| 场景 | 规则 |
|------|------|
| 新增 FR | 追加到末尾（FR-13、FR-14……），**禁止插入已有编号中间** |
| 废弃 FR | 标记为 `[REMOVED] FR-xx：[原名]（已移除）`，**禁止删除编号** |
| 拆分 FR | 原编号保留为父条目，子条目用 FR-xx-a / FR-xx-b 扩展（如 FR-09-b） |
| 重命名 FR | 只改功能域名称，编号不变 |

> 原因：测试函数名（`test_fr01_*`）、差距清单（`FR-01 ❌`）、追溯表均依赖编号；重排编号会导致所有引用同时失效。
