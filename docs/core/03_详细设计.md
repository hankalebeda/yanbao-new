## §0 全局错误/降级表示标准

> 来源：99 行动 09 要求在 03 §0 先定义全局标准，所有执行链路引用此标准。  
> 具体错误码枚举值在 05_API与数据契约.md 的「错误情况」章节定义，本节仅引用，不重复定义（SSOT 原则）。

### §0.1 错误码枚举（所有接口的 HTTP 500 error_code 必须来自此表）

| 错误码 | 含义 | 典型触发场景 |
|---|---|---|
| `DATA_SOURCE_UNAVAILABLE` | 数据源不可用（采集失败/超时） | FR-04 全量采集失败、FR-05 行情读取失败 |
| `LLM_ALL_FAILED` | 所有 LLM 降级路径均失败 | FR-06 主→备→CLI→本地全部不可用 |
| `VALIDATION_FAILED` | 内部数据校验失败 | FR-06 参数倒挂、FR-07 复权数据异常 |
| `IDEMPOTENCY_CONFLICT` | 幂等键冲突 | FR-06 同日同股重复生成 |
| `DEPENDENCY_NOT_READY` | 依赖的上游数据未就绪 | FR-06 等待 FR-04 核心池采集完成 |
| `COLD_START_ERROR` | 冷启动无历史数据可回退 | FR-01 首次部署无历史池、FR-05 无历史状态缓存 |
| `CONCURRENT_CONFLICT` | 并发互斥锁冲突 | FR-01/07 并发刷新/结算被拒 |
| `NOT_IN_CORE_POOL` | 请求标的不在当日核心池 | FR-06 非核心池股票被拒 |
| `REPORT_ALREADY_REFERENCED_BY_SIM` | 研报已被模拟持仓引用，禁止重建 | FR-12 强制重建级联阻断 |
| `STALE_TASK_EXPIRED` | 僵尸任务超窗过期 | FR-06 超 72h 任务转 Expired |
| `UPSTREAM_TIMEOUT` | 上游任务全局兜底超时 | FR-02 DAG 跨日兜底终止 |
| `CIRCUIT_BREAKER_OPEN` | 数据源/LLM 熔断器已开启 | FR-04/FR-06 连续失败触发熔断 |

### §0.2 降级字段标准结构（凡是降级响应，必须包含以下字段）

```json
{
  "success": false,
  "request_id": "<唯一请求ID，与 NFR-14 对齐>",
  "error_code": "<§0.1 枚举之一>",
  "error_message": "<人类可读说明>",
  "degraded": true,
  "degraded_reason": "<说明，degraded=true 时必填>",
  "data": null
}
```

> 与 01 NFR-14 对齐：降级/错误响应同样必须携带 `request_id`，禁止出现无请求追踪ID的异常返回。

**降级标记字段约定**（贯穿所有 FR，由 L2 判定、L3 事务内持久化）：

| 字段 | 适用范围 | 含义 |
|---|---|---|
| `quality_flag` | FR-04/06/07 | 数据/研报质量标记（全局枚举），FR-04 可用 `ok|stale_ok|missing|degraded`；FR-06/07 使用子集 `ok|stale_ok|degraded` |
| `status_reason` | FR-01/04/06/07/08/09-b | 当非正常终态时的原因文本，必须非空 |
| `state_reason` | FR-05 | 市场状态降级原因，含 `market_state_degraded=true` 标识 |
| `llm_fallback_level` | FR-06 | LLM 降级层级，枚举：`primary|backup|cli|local|failed` |
| `data_status` | FR-10 | 前端数据就绪状态，枚举：`READY|COMPUTING|DEGRADED` |
| `risk_audit_status` | FR-06 | 辩证审计状态，枚举：`completed|skipped|not_triggered` |

### §0.3 全局观测点约定

每条 FR 执行链路至少包含：
1. **一个状态落库点**：关键中间状态或终态写入数据库（字段定义见 04）
2. **一个告警触发点**：异常/降级事件触发 NFR-13 告警（日志+可配置 Webhook）

告警字段结构：`{ alert_type: str, fr_id: str, message: str, timestamp: str(ISO8601) }`

## §FR-00 真实性红线（跨 FR 全局约束）

### 执行位置与落点

> FR-00 不是独立链路，而是贯穿所有 FR 执行链路的强制约束。以下定义各约束在链路中的执行位置。

#### 约束 1：已发布研报只读保护

| 要素 | 内容 |
|---|---|
| **执行位置** | FR-06 研报落库后、FR-12 管理员 PATCH 操作前 |
| **责任层** | L2（业务判定）→ L3（事务保护） |
| **规则** | `published=true` 的研报内容（结论/citations/推理链）禁止事后覆盖写入；`PATCH /api/v1/admin/reports/{id}` 仅允许修改 `review_flag`/`published`（下架），不允许修改业务内容字段 |
| **失败处理** | 尝试修改已发布内容 → 返回 405 或 403 且内容不变（HTTP 语义见 `05_API §15 错误码词典`） |
| **观测点** | 审计日志记录被拒写入尝试（`action_type=PUBLISH_WRITE_REJECTED`） |

#### 约束 2：强制重建审计通道

| 要素 | 内容 |
|---|---|
| **执行位置** | FR-12 `POST /api/v1/admin/reports/{id}/force-regenerate` 链路 |
| **责任层** | L1（RBAC `super_admin` 校验）→ L2（级联检查+软删除+重建编排）→ L3（审计写入+事务） |
| **规则** | 仅 `role=super_admin` 可触发；必须携带 `force_regenerate=true` + `reason_code`；执行前检查无 FR-08 持仓/结算引用（有引用返回 409 `REPORT_ALREADY_REFERENCED_BY_SIM`）；旧研报软删除并生成新 `report_id` |
| **审计记录** | `{ actor_user_id, action_type="FORCE_REGENERATE", before, after, reason_code, request_id, timestamp }` |
| **失败处理** | 级联阻断 → 409；非 super_admin → 403；审计写入失败 → 整链路回滚 |
| **观测点** | 审计表落库（状态） + NFR-13 告警（`alert_type=FORCE_REGENERATE`） |

#### 约束 3：降级显式可见

| 要素 | 内容 |
|---|---|
| **执行位置** | 所有 FR 的降级分支终态 |
| **责任层** | L2（赋值 `quality_flag`/`status_reason`/`state_reason`）→ L3（持久化+告警） |
| **规则** | 任何降级路径必须显式标注原因字段并触发 NFR-13 告警，前台基于降级数据的研报须展示醒目警告横幅（"⚠️ 当前研报包含降级数据..."），不允许静默伪成功 |
| **失败处理** | 降级标记缺失的研报不允许 `published=true` |
| **观测点** | `quality_flag != ok` 落库 + 告警触发 |

#### 约束 4：关键操作审计

| 要素 | 内容 |
|---|---|
| **执行位置** | 研报发布（`published` false→true）、下架（true→false）、降级恢复、强制重建 |
| **责任层** | L3（审计门禁，无审计记录则写入失败） |
| **记录结构** | `{ actor_user_id, action_type, timestamp, request_id, before, after }` |

## §FR-01 股票池筛选

### 链路起点

定时刷新（每日开盘前 FR-02 调度触发）或管理端手动刷新请求 `POST /api/v1/admin/pool/refresh`

### 刷新生命周期状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `IDLE` | 无刷新任务进行中 | 初始态 / 刷新完成后回归 |
| `REFRESHING` | 正在执行筛选计算 | 获得互斥锁后进入 |
| `COMPLETED` | 刷新成功，产出新 `pool_version` | 筛选计算完成且落库成功 |
| `FALLBACK` | 筛选失败，已回退到上一交易日池 | 筛选异常且存在历史池 |
| `COLD_START_BLOCKED` | 冷启动阻断，无历史池可回退 | 首次部署/清库后首日无历史数据 |

### 执行链路

**步骤 1：并发互斥锁获取**
- 调用：L3 分布式互斥锁（`lock_key=pool_refresh:{trade_date}`）
- 输入：`trade_date`
- 输出：锁句柄或拒绝
- 失败处理：已有锁 → 返回 HTTP 409/429（`CONCURRENT_CONFLICT`），不排队直接拒绝
- 超时：锁 TTL 由 L3 配置（含续约心跳），遵循 HC-15

**步骤 2：幂等检查**
- 调用：L3 幂等拦截
- 输入：`trade_date`
- 规则：若当日已有 `pool_date == trade_date` 的成功记录且非 `force_rebuild=true` → 直接返回已有结果（`duplicate_policy=return_existing`）
- 输出：通过或返回已有数据

**步骤 3：全量数据加载**
- 调用：L2 编排 → L4 通达信适配器 / 数据库读取
- 输入：`universe_stocks`（全A基础信息）、`universe_kline_20d`（全A近20日行情）
- 输出：结构化的全量股票数据集
- 失败处理：数据加载失败 → 进入步骤 6 回退分支
- 超时：由 L3 配置（建议 300s）

**步骤 4：筛选计算（八因子打分+排序+截取）**
- 调用：L2 业务编排（`stock_pool.py`）
- 输入：全量数据集 + `filter_params`（含 `min_market_cap_cny`、`min_listing_days` 等，参数口径以 01 FR-01 为准）
- 处理：
  1. 排除 ST/*ST/退市/近5日连续跌停
  2. 八因子加权评分（因子定义以 01 FR-01 打分因子为权威准据）
  3. 按 score 降序取 Top 200 为 `core_pool`，Top 201~250 为 `standby_pool`
  4. 对比上一版 `core_pool` 计算 `evicted_stocks`
- 输出：`core_pool`、`standby_pool`、`scores`、`evicted_stocks`、`pool_version`
- 失败处理：计算异常 → 进入步骤 6 回退分支

**步骤 5：结果落库**
- 调用：L3（事务边界）→ L4（数据库写入）
- 输入：步骤 4 产出
- 输出：`pool_version` 递增并持久化
- 失败处理：落库失败 → 进入步骤 6 回退分支
- 观测点：`pool_version` 落库成功为状态落库点

**步骤 6：回退分支（筛选失败时）**
- 调用：L2 判定 → L4 读取历史池
- 规则：
  - 若存在上一交易日池 → 加载为当日池，`fallback_from` 必填（上一有效 `pool_date` 日期字符串）
  - 若不存在任何历史池（冷启动）→ 返回 `error_code=COLD_START_ERROR`，阻断下游 FR-04/FR-06，触发最高级告警（NFR-13）
- 观测点：告警触发（`alert_type=POOL_REFRESH_FALLBACK` 或 `COLD_START_ERROR`）

**步骤 7：释放互斥锁**
- 调用：L3 释放锁
- 无论成功或失败均执行

### 步骤关系

串行执行：1→2→3→4→5→7（正常路径）；任一步骤失败跳转 6→7

### 全链路失败返回

引用 §0.2 标准结构，`error_code` 按失败原因从 §0.1 选取。不阻塞其他功能（除冷启动阻断 FR-04/FR-06 外）。

## §FR-02 定时调度

### 链路起点

Cron 或上游完成事件触发 DAG 节点

### DAG 任务状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `PENDING` | 任务已注册，等待触发条件 | 调度器注册时初始化 |
| `WAITING_UPSTREAM` | 等待上游依赖完成信号 | DAG 依赖尚未满足 |
| `RUNNING` | 任务正在执行 | 触发条件满足或事件到达 |
| `SUCCESS` | 任务执行成功 | handler 返回成功 |
| `FAILED` | 任务执行失败（已重试） | 重试耗尽（默认3次） |
| `SKIPPED` | 非交易日跳过 | `is_trade_date()=false` |

### 状态迁移规则

```text
PENDING → WAITING_UPSTREAM（有 DAG 依赖时）
PENDING → RUNNING（无依赖/Cron 直接触发）
WAITING_UPSTREAM → RUNNING（上游完成事件到达）
WAITING_UPSTREAM → FAILED（超过次日开盘前死线，且 `status_reason="upstream_timeout_next_open"`）
RUNNING → SUCCESS
RUNNING → FAILED（重试耗尽后）
RUNNING → SKIPPED（非交易日检测）
```

### 执行链路

**步骤 1：交易日判定**
- 调用：L2 → `is_trade_date(date)`
- 输入：当日日期、`task_name`
- 输出：`{ is_trade_date: bool, task_policy: "TRADING_DAY_REQUIRED"|"ALWAYS_RUN" }`
- 规则：主策略（通达信 vipdoc 目录）→ 备策略（内置日历 CSV）→ 兜底（周一至五非法定假日）
- 非交易日分流（与 01 FR-02 边界对齐）：
  - `task_policy=TRADING_DAY_REQUIRED`（FR-01/05/06/07/08）且 `is_trade_date=false` → 直接收敛 `SKIPPED`
  - `task_policy=ALWAYS_RUN`（FR-04 热搜采集、FR-03 会话探活、FR-09-b 清理）即使 `is_trade_date=false` 也继续后续步骤
- 失败处理：判定失败 → 兜底策略

**步骤 2：重启幂等检查**
- 调用：L3 幂等拦截
- 规则：若当日该任务已有 `SUCCESS` 记录（以 `triggered_at` 最新为准）→ 跳过不重复触发
- 观测点：幂等跳过日志记录

**步骤 3：多实例互斥锁**
- 调用：L3 分布式互斥锁（`lock_key=cron:{trade_date}:{task_name}`）
- 规则：基于 `trade_date + task_name` 唯一键抢占；锁含 `lock_ttl_seconds`（默认300s，= 任务预计最长执行时长×2）+ 续约心跳；持锁崩溃后 TTL 到期允许接管；接管使用 fencing token（HC-15）
- 失败处理：抢锁失败 → 退出不执行

**步骤 4：DAG 依赖检查**
- 调用：L3 DAG 调度器
- 规则（HC-03）：
  - FR-06 必须等待 `FR-04核心池采集完成` 事件才可启动
  - FR-07/FR-08 必须等待 `FR-06批次完成` 事件才可启动
  - 禁止用固定时刻替代上游完成信号（如禁止"16:30 强行触发 FR-07"）
- 输出：依赖满足 → 触发执行；不满足 → 保持 `WAITING_UPSTREAM`

**步骤 5：任务执行**
- 调用：L3 通过 `handler_id` → L2 handler（02 §2.4.3 回调模式）
- 输入：`task_name`、`trade_date`
- 输出：任务执行日志（`task_name, triggered_at, status, retry_count, error_message`）
- 失败处理：重试默认3次；连续失败 → 告警（NFR-13）
- 超时：任务级超时由 L3 配置

**步骤 6：跨日一致性与全局兜底超时**
- 规则（HC-03）：
  - 若 FR-06 过夜执行（NFR-04 允许），FR-07/FR-08 必须在 FR-06 完成后继续执行
  - 超过 `config.dag_cascade_timeout_before_open`（默认次日 08:30）仍未获上游完成信号 → 触发 DAG 全局防链死兜底超时
  - 当前节点及所有下游节点链式标记为 `status=FAILED` + `status_reason="upstream_timeout_next_open"` 并终止本交易日链路
  - 绝对禁止带着过期挂起态跨越到新交易日
- 观测点：`status=FAILED AND status_reason=upstream_timeout_next_open` 落库 + 最高级告警

### 步骤关系

1→2→3→4→5 串行；步骤 6 为异常治理旁路，与步骤 4/5 并行监控

### 全链路失败返回

引用 §0.2 标准结构。调度失败不阻塞其他已完成任务，仅阻塞该任务的下游 DAG 节点。

## §FR-03 Cookie 与会话管理

### 链路起点

Cookie 初始化（首次人工登录）/ 健康探测定时任务触发（每5分钟）

### 会话健康状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `ACTIVE` | 会话有效，可正常使用 | 初始写入或探测成功续期 |
| `EXPIRING` | 会话即将过期（`expires_at` 距当前 < 30分钟） | 定时检测到临近过期 |
| `EXPIRED` | 会话已过期，须刷新 | `expires_at` < 当前时间 |
| `REFRESH_FAILED` | 自动刷新失败，需人工介入 | 连续刷新失败 |
| `SKIPPED` | 探活被互斥锁跳过（多实例防风暴） | 另一实例正在探活 |

### 执行链路

**步骤 1：分布式探活互斥锁**
- 调用：L3 分布式互斥锁（`lock_key=cookie_probe:{login_source}`）
- 规则（01 FR-03 边界）：同一探测窗口/同一任务键仅允许单节点发起；抢锁失败 → 直接 `status="skipped"`，禁止并发探活风暴
- 失败处理：抢锁失败节点返回 `skipped` 并拦截外部并发请求
- 超时：锁 TTL = 30s

**步骤 2：会话健康探测**
- 调用：L2 → L4（使用当前 `cookie_session` 发起轻量验证请求）
- 输入：`session_id`、`login_source`
- 输出：`status: "ok"|"fail"`、`last_refresh_at`（ok时）、`status_reason`（fail时）
- 失败处理：探测失败 → 进入步骤 3 自动刷新

**步骤 3：自动刷新尝试**
- 调用：L2 → L4（平台特定的 cookie 续期逻辑）
- 输入：当前 `cookie_session`
- 输出：新 `cookie_session` + `expires_at`
- 失败处理：刷新失败 → 告警（NFR-13），状态转 `REFRESH_FAILED`，需人工重新登录
- 超时：15分钟内恢复（失效恢复时长目标）
- 观测点：`last_refresh_at` 更新为状态落库点

**步骤 4：安全约束执行**
- `cookie_session` 禁止明文写入日志/前端页面，日志中脱敏或完全不打印

**步骤 5：释放互斥锁**
- 调用：L3 释放锁

### 步骤关系

1→2→(成功则结束，失败则)3→5 串行

### 全链路失败返回

引用 §0.2 标准结构。Cookie 失效不阻塞系统启动，相关功能模块进入 degraded 状态。

## §FR-04 多源数据采集

### 链路起点

全量采集批次启动（硬依赖 FR-01 股票池产出；FR-03 仅约束登录态数据源）

### 数据源熔断状态机定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `CLOSED` | 正常状态，允许请求 | 初始态 / half-open 探测成功 |
| `OPEN` | 熔断开启，冷却期内禁止请求 | 连续失败 ≥ 3 次（含超时、403/429/503） |
| `HALF_OPEN` | 冷却期结束，允许 1 次探测请求 | 冷却期 300 秒到期 |

**状态迁移**：

```text
CLOSED → OPEN（连续失败 ≥ 3 次）
OPEN → HALF_OPEN（冷却 300 秒后）
HALF_OPEN → CLOSED（探测成功，重置失败计数）
HALF_OPEN → OPEN（探测失败，重新进入 300 秒冷却，刷新 circuit_open_at）
```

### 多源优先级降级规则

- 采集按优先级顺序尝试：eastmoney→xueqiu→cls→baidu_hot→weibo→douyin→kuaishou
- ≥ 3 个可用 → `quality_flag=ok`
- < 3 个可用 → `quality_flag=stale_ok`
- 全部不可用 → `quality_flag=degraded`

### 执行链路

**步骤 1：加载采集目标**
- 调用：L2 → 读取 FR-01 产出的全量股票列表 + 核心池
- 输入：`trade_date`、FR-01 `core_pool`（200只）+ 全量列表（5000+）
- 输出：采集任务清单（分核心池与长尾两组）

**步骤 2：日线行情采集（通达信）**
- 调用：L2 → L3（限速器）→ L4（通达信适配器）
- 输入：`config.tdx_install_dir`、`stock_codes`
- 输出：`kline_daily[]`（前复权口径，`adjust_type=front_adjusted`）
- 限速策略：极度保守的 Rate Limit + 长周期分散采集（HC-20 要求队列防封IP）
- 失败处理：
  - **核心池单股失败**：优先回退 T-1 日缓存（`quality_flag=stale_ok`，研报标注「行情数据基于T-1日」）；无缓存则 `published=false`、`quality_flag=degraded`、`status_reason` 必填
  - **长尾股失败**：Partial Commit 原子隔离（HC-20），记录失败明细，不导致整批回滚
- 产出含 `is_suspended` 标记（供 FR-06 停牌跳过判定）

**步骤 3：热搜采集（多源并发）**
- 调用：L2 → L3（熔断器+限速器）→ L4（7 平台适配器）
- 输入：平台优先级列表
- 输出：`hotspot_top50[]`（去重合并 rank 后保留 Top 50 条）
- 熔断规则：见上方数据源熔断状态机，由 L3 `circuit_breaker.py` 统一管理
- 失败处理：按多源优先级降级规则赋 `quality_flag`
- 观测点：熔断开启/关闭事件写入告警日志（`alert_type=CIRCUIT_BREAKER_OPEN/CLOSE`）

**步骤 4：数据标准化与血缘登记**
- 调用：L2（标准化/字段组装）→ L3（事务）→ L4（入库）
- 输入：步骤 2/3 原始数据
- 输出：标准化数据集 + 血缘记录 + `quality_flag`
- 规则：核心池数据深度标注；非核心池提供基础统计信号

**步骤 5：批次提交（核心池 vs 长尾隔离）**
- 调用：L3（事务边界）→ L4
- 规则（HC-20）：非核心池长尾股票允许 Partial Commit；单股解析/落库失败原子隔离并记录错误，不导致整批 5000+ 股票事务整体回滚
- 核心池关键数据按单股失败降级规则执行
- 观测点：批次完成状态落库；核心池覆盖率统计

### 步骤关系

1→(2 和 3 并行)→4→5 串行

### 全链路失败返回

引用 §0.2 标准结构。核心池 200 只全覆盖时终态为成功；部分失败时以降级终态收敛（逐股可判定：成功或显式降级 `stale_ok/degraded`）。长尾部分提交可追溯。不阻塞其他功能（但核心池覆盖不足时会影响 FR-06 研报质量）。

## §FR-05 市场状态机

### 链路起点

市场状态计算触发（每日09:00定时任务，依赖 FR-04 行情数据）

### 市场状态缓存/回退状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `FRESH` | 当日09:00计算完成的新鲜状态 | 09:00任务成功执行 |
| `CACHED` | 使用上一交易日缓存（幽灵时段00:00-08:59） | 当日09:00任务未运行时 |
| `DEGRADED_NEUTRAL` | 数据故障降级为NEUTRAL | 数据源故障致无法计算 |

> `COLD_START_FALLBACK` 为 `state_reason` 值，不是状态枚举；绝对冷启动统一表达为 `market_state=NEUTRAL` + `state_reason="COLD_START_FALLBACK"` + `reference_date=null`。

### 执行链路

**步骤 1：数据准备（T-1 口径严守）**
- 调用：L2 → L4（行情读取）
- 输入：`trade_date`（T日）、`reference_date`（T-1有效交易日）
- 输出：`sh_index_close`、`hs300_close`、`hs300_ma5`、`hs300_ma20`、`hs300_ma20_5d_ago`、`hs300_return_20d`、`market_volume_ratio`
- 规则：09:00 任务禁止读取 T 日盘中/收盘未完成数据；若 `reference_date` 缺失则继续回退到更早有效交易日（禁止返回空值，除绝对冷启动外）
- 失败处理：数据获取失败 → 步骤 3 降级分支
- 超时阈值与动作：单次行情读取超时阈值由 `config.market_state_fetch_timeout_seconds` 控制（默认 10s）；超时按“数据获取失败”处理并进入步骤 3 降级分支

**步骤 2：状态判定（优先级：BEAR > BULL > NEUTRAL）**
- 调用：L2（`market_state.py`）
- 判定规则（以 01 FR-05 为权威准据）：
  - **BULL**：`hs300_ma20 > hs300_ma20_5d_ago`（MA20斜率向上）AND `hs300_return_20d > 0.03`
  - **BEAR**：`hs300_ma5 < hs300_ma20` AND `hs300_return_20d < -0.05`
  - **NEUTRAL**：其余
  - 同时满足多条件时以最高优先级判定（BEAR > BULL > NEUTRAL），不存在模糊态
- 输出：`market_state`（BULL/NEUTRAL/BEAR）、`market_state_date`（T日）、`reference_date`

**步骤 3：缓存写入与降级标记**
- 调用：L3（缓存写入/TTL 管理）→ L4（持久化）
- 缓存 TTL 至次日 09:00
- 降级标记约束：若因数据获取故障降级为 NEUTRAL → `state_reason` 必须显式含 `market_state_degraded=true`；FR-06 此状态下须在研报中标注「市场状态数据异常」
- 超时阈值与动作：缓存写入超过 `config.market_state_write_timeout_seconds`（默认 3s）时，返回降级结果并触发 `alert_type=MARKET_STATE_WRITE_TIMEOUT`
- 观测点：状态落库 + 降级时触发告警

**步骤 4：幽灵时段回退（00:00-08:59:59）**
- 触发：当日09:00任务未运行时有查询请求
- 规则：返回上一有效交易日缓存状态，`market_state_date` 为该交易日
- 不得返回空状态或抛 500

**步骤 5：绝对冷启动兜底**
- 触发：幽灵时段且数据库无任何历史状态缓存
- 返回：`HTTP 200` + `market_state=NEUTRAL`、`state_reason="COLD_START_FALLBACK"`、`reference_date=null`

### 步骤关系

1→2→3 串行（正常路径）；4/5 为查询时旁路分支

### 全链路失败返回

引用 §0.2 标准结构。市场状态失败不阻塞其他功能，但会影响 FR-06 的 BEAR 短路判定质量。

## §FR-06 研报生成

### 链路起点

研报任务入队启动（依赖 FR-01 核心池/FR-04 采集数据/FR-05 市场状态，且读取 FR-07 已结算历史作为 `prior_stats`，仅历史快照、非当日阻塞依赖）

### 输入来源与输出去向

- 输入来源：`FR-01.core_pool`、`FR-04` 标准化行情与热搜、`FR-05.market_state`、`FR-07` 历史结算快照（仅 `signal_date < data_cutoff`）
- 输出去向：`report/report_generation_task/sim_trade_instruction` 写入 L4；`quality_flag`/`llm_fallback_level`/`risk_audit_status` 等降级字段按 §0.2 持久化，并供 FR-10/FR-12/FR-13 消费
- 回链说明：字段级契约待 04/05 最终回链，当前以 01 FR-06 口径为准

### 任务状态机定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `Pending` | 任务已入队，等待执行 | 任务创建时初始化 |
| `Processing` | 正在执行 LLM 调用/规则引擎 | 从队列取出开始处理 |
| `Completed` | 研报生成成功 | 全部步骤通过并落库 |
| `Failed` | 生成失败（已重试） | LLM/校验失败且重试耗尽 |
| `Suspended` | LLM 全局熔断挂起 | L3 熔断器连续 N 次失败触发 `llm_circuit_open=true` |
| `Expired` | 僵尸任务超窗过期 | `trade_date < T-1` 或 `updated_at < now()-72h` |

**状态迁移**：

```text
Pending → Processing（队列取出，开始处理）
Pending → Suspended（LLM 全局熔断时 Pending 批量转 Suspended）
Pending → Expired（超窗僵尸任务过期清理）
Processing → Completed（成功落库）
Processing → Failed（LLM 失败且重试耗尽 / 校验失败）
Failed → Pending（手动重试或断点续传时恢复，需满足恢复窗口条件）
Suspended → Pending（LLM 熔断恢复后探活成功，批量恢复）
Suspended → Expired（超窗清理）
```

### 降级职责分工约束（02 §2.4.2 LLM 调用链）

L4 返回 LLM 原始结果后**先回到 L2** 完成全部业务判定（质量判定、辩证审计、参数倒挂校验），L2 给出 `quality_flag`/`llm_fallback_level` 等降级结论的最终值，**然后**由 L3 在事务范围内将 L2 已给出的降级标记统一持久化到数据库；L3 不得自行修改或推翻 L2 的降级判定。

### 执行链路

**步骤 1：核心池白名单校验**
- 调用：L2
- 规则：`stock_code` 须在当日 200 只核心池白名单内；非核心池 → 返回 400 `NOT_IN_CORE_POOL`
- 停牌检查：FR-04 标记 `is_suspended=true` → 跳过 LLM，直接生成静态降级研报（`recommendation=HOLD`、`confidence=0.0`、`sim_trade_instruction[*].status="SKIPPED"`、`skip_reason="SUSPENDED"`、`status_reason="SUSPENDED_SKIPPED"`）

**步骤 2：幂等检查**
- 调用：L3 幂等拦截
- 幂等键：`idempotency_key = daily:{stock_code}:{trade_date}`
- 规则（HC-01）：重复触发 → 返回已有 `report_id`，不重新生成

**步骤 3：strategy_type 规则引擎前置判定（非 LLM 自由发挥）**
- 调用：L2（`report_engine.py`）
- 时序：在 LLM 调用**之前**完成判定
- 规则（以 01 FR-06 为权威准据）：
  - **A（事件驱动）**：FR-04 输入数据中存在政策/公告/重大事件标签命中
  - **B（趋势跟踪）**：MA20 向上且近 5 日涨幅 > 3%
  - **C（低波套利）**：ATR_pct < 2% 且近 20 日波动率处于市场后 30% 分位
  - 未命中任何规则 → 默认 B
- 输出：`strategy_type_pre_llm`（A/B/C）
- LLM 不得自行决定 strategy_type；最终 `report.strategy_type` 以规则引擎结果为准

**步骤 4：BEAR 市场短路拦截（HC-09）**
- 调用：L2
- 规则：BEAR 市场状态 + strategy_type = B/C → 在 LLM 调用**之前**直接短路跳过
- 产出：静态降级记录（`published=false`、`quality_flag=degraded`、`status_reason="BEAR_MARKET_FILTERED"`）
- 绝对禁止向外部 LLM 发起无效请求消耗资源

**步骤 5：先验数据注入**
- 调用：L2
- 数据来源：FR-07 `settlement_status=settled` 的已结算历史记录
- 时序切割：`data_cutoff = trade_date.replace(day=1)`（本月1日），仅使用 `signal_date < data_cutoff` 的记录
- 按 `strategy_type` 分组独立统计
- 最小样本门槛：样本数 < 30 → `prior_stats=null`，Prompt 中不注入
- Prompt 措辞约束：以"以下为本系统已结算历史研报统计（截至{data_cutoff}），仅供参考，不代表未来"开头

**步骤 6：LLM 调用（含熔断/降级）**
- 调用：L2（构建 Prompt）→ L3（熔断/限速/全局挂起）→ L4（LLM 多级降级适配器）
- LLM 降级路径：主 API → 备 API → CLI → 本地（Ollama）→ failed
- 熔断规则：连续 N 次（默认5）网络/鉴权失败 → `llm_circuit_open=true`，`Pending` 批量转 `Suspended`；冷却期仅低频探活，成功后 `Suspended→Pending` 恢复
- 业务异常（JSON Parse/Schema）重试上限 2 次（L3 频控控制）；超限 → `Failed/degraded`，记录 `llm_bad_output_format`
- 强制串行或低配并发队列（200只核心池缓慢出具）

**步骤 7：质量判定与辩证审计**
- 调用：L2（LLM 返回后，先回到 L2）
- 7a. 参数倒挂校验：`stop_loss < entry_price < target_price`，倒挂 → `sim_trade_instruction[*].status=SKIPPED`、`skip_reason="logic_inversion_fallback"`
- 7b. 辩证审计（仅 BUY + confidence ≥ 0.65 触发）：
  - 用同一数据集调用 `config.llm_audit_provider`（低成本 LLM）执行风险审计
  - 副模型只输出风险点，不输出买卖建议（辩证式，非投票式）
  - 若副模型识别风险点未被主模型推理链正面讨论 → 追加"⚠️ 风险补充审计"章节
  - 副模型失败/超时 → 跳过审计，主结论照常发布，记录 `risk_audit_status=skipped`、`risk_audit_skip_reason` 必填
  - SELL/HOLD → `risk_audit_status=not_triggered`
- 7c. L2 给出 `quality_flag`/`llm_fallback_level`/`risk_audit_status` 等降级结论最终值
- 输出：`risk_audit_status` 收敛为 `completed|skipped|not_triggered` 三态之一

**步骤 8：资金挡位指令组装**
- 调用：L2
- 规则：BUY + confidence ≥ 0.65 → 按 `config.CAPITAL_TIERS`（10k/100k/500k）分挡产出 `sim_trade_instruction`
- 各挡位 `position_ratio` 遵循 01 FR-06 资金分挡策略约束
- 买不起 1 手（100股）→ 该挡位 `status=SKIPPED`
- BUY + confidence < 0.65 / SELL / HOLD → 三挡统一 `status=SKIPPED`、`position_ratio=0.0`

**步骤 9：事务落库（L3 事务+降级标记持久化）**
- 调用：L3（事务边界+降级标记持久化）→ L4（写入）
- 规则：L3 将 L2 步骤 7c 已给出的降级结论统一持久化；不得修改 L2 判定结果
- 观测点：`report_id` + `quality_flag` 落库

**步骤 10：断点续传恢复（服务重启时）**
- 调用：L3（`task_queue.py`）
- 规则（HC-10）：自动扫描并拉起 `Pending|Failed|Suspended` 任务
- 恢复窗口约束：仅允许 `trade_date >= T-1` 且 `updated_at >= now()-72h`
- 超窗任务 → 转 `Expired`、`status_reason="stale_task_expired"`

### 步骤关系

1→2→3→4（短路则结束）→5→6→7→8→9 串行；步骤 10 为重启旁路

### 全链路失败返回

引用 §0.2 标准结构。单只研报失败不阻塞其他核心池股票的生成。BUY 强信号触发 FR-08 开仓（见 §FR-08 起点）。

## §FR-07 预测结算与回灌

### 链路起点

结算任务提交（依赖 FR-06 批次完成）。两个入口：
1. **内部定时批量入口**（`POST /api/v1/internal/settlement/run`）：异步受理，返回 `task_id`
2. **管理端手工入口**（`POST /api/v1/admin/settlement/run`）：含 `force` 参数影响幂等行为

**对照组起点**：主体结算任务提交时同步创建对照组独立子任务（`baseline_random` + `baseline_ma_cross`），子任务拥有独立异步生命周期（HC-22）

### 输入来源与输出去向

- 输入来源：`FR-06` 已发布研报与交易指令、`FR-04` 行情快照、费用参数（01 FR-07/NFR-05）
- 输出去向：`settlement_result` 与 `baseline_result` 落库，`signal_validity_warning` 供 FR-10 前端预警展示
- 回链说明：字段枚举与表结构以后续 04/05 为准，03 仅定义链路与状态收敛

### 异步结算任务状态定义（受理链）

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `QUEUED` | 任务已提交入列 | HTTP 202 受理 |
| `PROCESSING` | 正在执行结算计算 | 从队列取出 |
| `COMPLETED` | 结算任务完成 | 全部目标结算完毕 |
| `FAILED` | 结算任务失败 | 执行异常 |

### 结算结果态（settlement_status，与受理链严格隔离）

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `pending` | 等待结算（行情未就绪） | 研报发布但结算窗口未到/行情缺失 |
| `settled` | 已结算 | 计算完成且数据有效 |
| `skipped` | 跳过（已 settled 且 force=false） | 手工重跑幂等跳过 |
| `degraded` | 降级结算（数据不完整） | 行情部分缺失 |

### 对照组独立异步生命周期状态（HC-22）

| 状态 | 含义 |
|---|---|
| `BASELINE_QUEUED` | 对照组子任务已创建 |
| `BASELINE_PROCESSING` | 蒙特卡洛模拟/MA金叉计算中 |
| `BASELINE_COMPLETED` | 对照组计算完成 |
| `BASELINE_FAILED` | 对照组计算失败 |

对照组结果未就绪时主体结算可独立返回 `READY`，对照组字段为 `null` + `display_hint="基线对照数据计算中"`

### 执行链路

**步骤 1：任务受理与鉴权**
- 内部入口：L1 校验 `X-Internal-Token`
- 管理端入口：L1 JWT `role=admin` 校验
- 受理后立即返回 `HTTP 202 + { task_id, status: "QUEUED", force }`
- 超时阈值与动作：受理阶段超过 `config.settlement_accept_timeout_seconds`（默认 5s）时返回 503，且不创建任务

**步骤 2：并发结算互斥锁**
- 调用：L3 分布式互斥锁（`lock_key=settlement:{trade_date}:{target_scope}`）
- 规则：定时批处理与管理员手工强制结算执行同一日期范围任务时须获取互斥锁
- 失败处理：抢锁失败 → 立即返回 409（`CONCURRENT_CONFLICT`），不排队防死锁

**步骤 3：手工结算幂等判定**
- 规则：`force=false` 且目标记录已 `settled` → `skipped`（不覆盖）
- `force=true` 且 `role=admin` → 允许重算覆盖，写审计日志

**步骤 4：信号结算计算**
- 调用：L2（`settlement.py`）
- 费用扣除口径（以 01 FR-07 为权威准据）：
  - 买入实付 = P_entry × shares × (1 + 0.05%) + max(P_entry × shares × 0.025%, 5元)
  - 卖出实得 = P_exit × shares × (1 - 0.05%) - max(P_exit × shares × 0.025%, 5元) - 印花税（NFR-05 口径：卖出金额×0.5‰）
  - shares 固定 100 股（1手，HC-07 口径隔离，不关联挡位）
  - `return=0` 判定：`abs(net_return_pct) < 0.0001`
- 前复权一致性提取约束（HC-06）：`P_entry` 与 `P_exit` 必须在同一次前复权序列快照上下文中提取
- 按 strategy_type（A/B/C）× 结算窗口（1/7/14/30/60日）独立统计
- `win_rate`/`profit_loss_ratio` 排除 return=0 的样本
- `coverage_pct` = 有效结算信号数 / 当前累计发出BUY信号总数
- 产出 `cost_breakdown_fr07` 全字段
- 超时阈值与动作：单任务计算超过 `config.settlement_task_timeout_seconds`（默认 900s）时，任务受理态收敛为 `FAILED`，并写 `status_reason="settlement_timeout"` + NFR-13 告警
- 观测点：`settlement_status=settled` 落库

**步骤 5：对照组基线独立异步计算（HC-22）**
- 调用：L2 创建独立子任务 → L3（异步调度）
- 5a. **随机买入基线（baseline_random）**：500 次蒙特卡洛模拟，取累计净收益率中位数；拆分为独立异步执行生命周期，HTTP 接口仅提交任务，不在请求连接内同步执行（防网关 504）
- 5b. **纯技术 MA 金叉基线（baseline_ma_cross）**：MA5 上穿 MA20 为买入信号，同费用模型
- 样本 < 30 笔时对照组同样不展示（与四维度统计门槛一致）
- 对照组与主指标可独立就绪、独立查询
- 超时阈值与动作：基线子任务超过 `config.baseline_task_timeout_seconds`（默认 1800s）收敛为 `BASELINE_FAILED`，主体结算结果保持可查询并返回 `display_hint`

**步骤 6：信号有效性预警判定**
- 规则：当 AI 研报累计净收益率 < baseline_random 累计净收益率 → `signal_validity_warning=true`
- FR-10 看板须展示"⚠️ 当前信号有效性低于随机水平"
- 观测点：`signal_validity_warning` 结果落库；连续预警触发 `alert_type=SIGNAL_VALIDITY_WARNING`

### 步骤关系

1→2→3→4 串行（主结算路径）；5 与 4 并行（对照组独立异步）；6 在 4+5 均完成后执行

### 全链路失败返回

引用 §0.2 标准结构。结算失败不阻塞其他 FR。

## §FR-08 模拟实盘追踪

### 链路起点

**两阶段起点**：
- **阶段①入队**：FR-06 每生成一个 BUY 强信号（`recommendation=BUY AND confidence≥0.65`）即将 `sim_trade_instruction` 写入当日待执行批次队列（按 `trade_date` 聚合）
- **阶段②截断与开仓**：当日 FR-06 **首轮** 200 只核心池任务全部结束后（含断点续传恢复完成），L2 按挡位以 confidence 降序（同分以 `report_id` 为 tiebreaker）做全局排序与 `sim_max_positions` 截断，三挡分别执行开仓

### 输入来源与输出去向

- 输入来源：`FR-06.sim_trade_instruction`、`FR-04` 行情与停牌/流动性信息、账户当前资产与持仓快照
- 输出去向：`sim_position/sim_account` 原子更新；仓位状态变更事件回链 FR-13（`POSITION_CLOSED`/`DRAWDOWN_ALERT`）
- 回链说明：资金字段/仓位字段在 04 定义，接口展示在 05 定义

### 仓位状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `OPEN` | 持仓中 | 开仓成功 |
| `TAKE_PROFIT` | 止盈平仓 | High ≥ dynamic_target_price 且满足 T+1/流动性条件 |
| `STOP_LOSS` | 止损平仓 | Low ≤ dynamic_stop_loss 且满足条件（含悲观撮合） |
| `TIMEOUT` | 超时平仓 | 持仓 ≥ 180 日 |
| `DELISTED_LIQUIDATED` | 退市核销 | 持仓期间标的退市 |
| `SKIPPED` | 跳过未开仓 | 资金不足/碎股/满仓/回撤HALT等 |

### 回撤状态定义

| 状态 | 条件 | `drawdown_state_factor` | 行为 |
|---|---|---|---|
| `NORMAL` | `max_drawdown_pct > -0.12` | 1.0 | 正常开仓 |
| `REDUCE` | `-0.20 < max_drawdown_pct <= -0.12` | 0.5 | 仓位减半 |
| `HALT` | `max_drawdown_pct <= -0.20` | 0.0 | 禁止新开仓，仅允许平仓 |

### 执行链路

**步骤 1：日内预处理——先平后开（HC-08）**
- 调用：L2（`sim_position.py`）
- 规则：同一交易日批处理必须先执行全部平仓（止盈/止损/超时/退市）释放 `cash_available`，再按 confidence 降序执行新 BUY 信号开仓
- 超时：无硬超时，随日批链路

**步骤 2：止盈/止损评估**
- 调用：L2
- 前复权动态价格折算护城河（HC-06）：每次盘后评估必须从最新前复权序列提取 `entry_date` 对应的 `adj_entry_price`，并按原始 `atr_pct` 重算 `dynamic_stop_loss` 与 `dynamic_target_price`
- **止盈**：仅当 `entry_date < trade_date`（T+1可卖）且 `is_suspended=false` 且 `volume>0` 且当日 High ≥ `dynamic_target_price` → `TAKE_PROFIT`
- **止损**：仅当 `entry_date < trade_date` 且 `is_suspended=false` 且 `volume>0` 且当日 Low ≤ `dynamic_stop_loss` → `STOP_LOSS`
- **K线悲观撮合（HC-13）**：High ≥ TP AND Low ≤ SL 同日极值碰撞 → 强制判定为先触发止损（`STOP_LOSS`），坚决拦截套利幻觉
- **流动性顺延**：停牌/一字涨跌停/`volume=0` → 保持 `OPEN`，设 `suspended_pending=true`/`limit_locked_pending=true` 顺延次日
- **超时**：持仓 ≥ 180 日 → `TIMEOUT` 强制平仓
- **退市核销**：持仓期间标的退市 → `DELISTED_LIQUIDATED`、`exit_price=0.0`、`net_return_pct=-1.0`

**步骤 3：新信号开仓**
- 调用：L2（从批次队列取信号）
- 排序：按挡位以 confidence 降序（同分以 `report_id` 为 tiebreaker）
- 截断：各挡位 `sim_max_positions`（1W=2只、10W=5只、50W=10只，精确配置值）
- 回撤检查：`drawdown_state=HALT` → 禁止新开仓
- 开仓价格：`actual_entry_price` = T+1 开盘价（与 FR-06 `signal_entry_price` 信号日收盘价口径不同，HC-07 禁止混用）
- 碎股截断：`shares = floor(raw_shares / 100) * 100`，不足 100 股 → `SKIPPED`
- 资金检查：买不起 1 手 → `SKIPPED`、`skip_reason="INSUFFICIENT_FUNDS"`

**步骤 4：账户净值更新**
- 调用：L2 → L3（事务）→ L4（原子写入）
- `sim_position` 与 `sim_account` 必须原子提交（SQLite WAL + 队列 + 重试）
- 结算重放幂等：去重键 `(trade_date, position_id)`
- 费用扣除口径（NFR-05）：佣金万2.5最低5元 + 印花税卖出0.5‰ + 滑点0.05%
- `net_return_pct` 平仓时计算写入（OPEN 状态为 null）
- `max_drawdown_pct` 与 `drawdown_state` 同步更新
- 观测点：`sim_account.total_asset` + `position_status` 落库

**步骤 5：事件触发（Outbox 模式，HC-02）**
- 仓位状态变更为 TAKE_PROFIT/STOP_LOSS/TIMEOUT → 需触发 FR-13 `POSITION_CLOSED`
- 回撤进入 REDUCE/HALT → 需触发 FR-13 `DRAWDOWN_ALERT`
- 事务后派发（L3 Outbox），回滚时取消派发

### 步骤关系

1→2→(平仓完成后)→3→4→5 严格串行（先平后开时序约束）

### 全链路失败返回

引用 §0.2 标准结构。模拟持仓失败不阻塞研报发布（NFR-11 两阶段约束）。

## §FR-09 商业化与权益（含登录/鉴权）

### 链路起点

注册/登录/刷新/OAuth/Webhook 请求进入认证链路

### 输入来源与输出去向

- 输入来源：用户凭据、OAuth 回调参数、支付 Webhook 事件、refresh_token/jti 上下文
- 输出去向：`user/oauth_identity/refresh_token/jti_blacklist/order` 等状态落库并供 FR-10/FR-12 查询消费
- 回链说明：状态枚举与字段明细以 04/05 为准；03 聚焦状态迁移和事务边界

### 会话状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `ACTIVE` | 会话有效 | 登录成功/Token 刷新成功 |
| `EXPIRED` | access_token 过期 | `exp` 到期 |
| `REVOKED` | Token 已撤销 | `/auth/logout` 或密码重置触发 |
| `BLACKLISTED` | jti 进入黑名单 | `/auth/logout` 或密码重置调用 L3 写入 |

### 订单状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `CREATED` | 订单已创建 | `POST /billing/create_order` |
| `PENDING` | 等待支付 | 用户已跳转支付 |
| `PAID` | 已支付（权益已发放） | Webhook 验证成功且原子事务提交 |
| `EXPIRED` | 超时未支付 | PENDING 超过安全时限（15分钟） |
| `REFUNDED` | 已退款 | 手动退款处理 |

### 执行链路

**步骤 1：注册链路**
- L1 参数校验 → L2 编排 → L3（幂等+事务）→ L4（落库+SMTP）
- 创建用户（`email_verified=false`）→ 发送激活邮件
- 邮件失败 → `HTTP 201 + data.email_status="send_failed"`，引导 `POST /auth/resend-activation`
- 未激活占坑 24h 释放（HC-18，由 FR-09-b 清理任务执行）

**步骤 2：登录链路**
- L1 速率限制前置（同一 IP/email 10分钟5次失败 → 429，L1 通过 RateLimitPort 调用 L3）
- L2 密码验证（bcrypt/argon2）→ 未激活账号 → 401 `EMAIL_NOT_VERIFIED`
- 成功：签发 `access_token`（含 jti/sub/role/tier/exp）+ `refresh_token`

**步骤 3：Token 刷新链路（refresh_token 宽限期 60s）**
- 调用：L2（`membership.py`）
- 规则（HC-17）：
  - 正常刷新：签发新 token 对，旧 refresh_token 即时作废
  - 并发刷新宽限：同一旧 refresh_token 在首次成功刷新后 60 秒内的并发重复请求，不触发全设备登出（返回同一新 token 或 401 但保留会话）
  - 超过 60 秒的旧 token 重放 → 按攻击处理，触发全设备登出
- 响应须符合 NFR-14 封装（含 `success/request_id/data`）

**步骤 4：忘记密码发信链路**
- 调用：L2（`forgot_password` 编排）→ L4（SMTP）
- 规则：无论邮箱是否存在均返回 HTTP 200；仅对存在账号发送重置链接邮件（token 一次性、有效期 1 小时）
- 超时阈值与动作：SMTP 超过 `config.auth_mail_timeout_seconds`（默认 10s）时记录 `status_reason="reset_mail_send_failed"` 并触发告警，不泄露邮箱存在性

**步骤 5：重置密码提交链路（全设备登出）**
- 调用：L2 编排 → L3 原子事务 → L4 落库
- 规则（HC-17）：验证 reset_token 后更新密码，并在同一事务内执行：
  - ① 撤销该用户所有存活 refresh_token（批量失效标记）
  - ② 所有关联 access_token 的 jti 写入 L3 黑名单（TTL ≥ access_token 最大有效期）
- 任一步失败 → 整链路回滚
- L1 每次请求鉴权时通过 `TokenRevocationPort` 查询 jti 黑名单，命中 → 401

**步骤 6：OAuth 回调链路**
- L0（第三方 OAuth 重定向）→ L1（state/参数校验）→ L2（账号绑定与 Token 编排）→ L3（幂等+事务）→ L4（OAuth 适配+落库）
- 邮箱缺失防碰撞（方案B）：允许 `user.email` 可空，以 `oauth_identity(provider, provider_user_id)` 联合唯一键为幂等身份键
- 同一 `(provider, provider_user_id)` 二次登录不重复建户
- 超时阈值与动作：第三方 OAuth 资料换取超过 `config.oauth_exchange_timeout_seconds`（默认 8s）时返回 401/400 并触发 `alert_type=OAUTH_CALLBACK_TIMEOUT`

**步骤 7：支付 Webhook 回调链路**
- L0→L1（签名校验）→L2（订单/权益编排）→L3（幂等 event_id + 原子事务）→L4（落库）
- HC-16：`event_id` 幂等主键，重复回调不重复发放权益
- 订单置 PAID 与权益发放同一原子事务提交（SQLite 禁止 SELECT FOR UPDATE，用唯一约束/版本号/写锁）
- 乱序到达防降级：禁止低级套餐覆盖已有高级权益
- 失败 → 全量回滚，返回 500 触发支付渠道重试
- 超时阈值与动作：Webhook 编排超过 `config.billing_webhook_timeout_seconds`（默认 10s）按内部失败处理返回 500，触发渠道重试
- 观测点：订单状态 + 权益落库；签名失败/事务回滚触发 `alert_type=PAYMENT_WEBHOOK_ERROR`

**步骤 8：掉单防御守护轮询**
- 调用：L2（守护任务）→ L4（主动查询支付状态）
- 规则：订单 `PENDING` 超过 15 分钟 → 主动轮询支付渠道反查状态 → 闭环到 `PAID/EXPIRED`
- 同时提供 FR-12 管理端补单接口
- 超时阈值与动作：单次反查超过 `config.billing_reconcile_timeout_seconds`（默认 8s）则记录失败并按下一轮守护任务重试，不改变订单终态

### 步骤关系

各子链路独立；步骤 8 为后台守护异步任务

### 全链路失败返回

引用 §0.2 标准结构。会话与订单权益状态一致收敛（成功、拒绝或回滚）。

## §FR-09-b 系统维护清理任务

### 链路起点

Cron 定时触发（避开交易时段；建议每周或每月执行一次，可配置为 02:00 等低峰时段）

### 输入来源与输出去向

- 输入来源：`report_generation_task/order/user/audit_log` 等可清理域数据快照 + 保护域清单（HC-19）
- 输出去向：僵尸任务转终态（`Expired`）与归档清理结果落库；清理指标写入观测日志/告警
- 回链说明：保留周期与归档策略以 01 FR-09-b、04 数据治理章最终口径为准

### 清理任务状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `SCHEDULED` | 已计划执行 | Cron 触发写入任务表 |
| `RUNNING` | 正在执行清理 | 调度器取出并获取互斥锁 |
| `COMPLETED` | 清理完成 | 所有清理子步骤执行完毕（含部分失败） |
| `FAILED` | 清理失败 | 关键异常导致整体中断（如数据库不可用） |

> 清理任务采用 `cleanup_id` 作为幂等键，同一 `cleanup_date` 仅执行一次。

### 执行链路

**步骤 0：永久保留域保护检查（HC-19）**
- 调用：L2（清理策略判定）→ L3（策略快照固化）→ L4（执行前校验）
- 规则：以下域一律列入 `cleanup_protected_domains`，禁止物理删除：
  - 正式研报与证据源（report/citations）
  - 全市场历史日线行情
  - 核心池买入/持仓/结算记录（含 sim_position/sim_account/sim_settlement）
  - 用户正式配置与关键审计日志
- 失败处理：若清理计划触及保护域，记录子项 `result=failed` + `status_reason="protected_domain_forbidden"`，触发告警并阻断该子项执行（不扩展任务状态枚举）

**步骤 1：未激活占坑清理（HC-18）**
- 调用：L2 → L3（事务）→ L4
- 规则：`email_verified=false AND created_at < now()-24h` → 物理删除
- 观测点：清理数量日志记录

**步骤 2：僵尸任务转终态与归档**
- 调用：L2 → L3（事务）→ L4
- 规则（对齐 01 FR-09-b）：
  - 仅处理 `report_generation_task.status in (Pending, Suspended)` 且超过 `retention_days.stale_pending_suspend`（默认 3 天）且无状态变更的记录
  - 先原子转 `Expired`，并写 `status_reason="stale_task_expired"`（禁止继续拉起补算）
  - 转终态后按 `retention_days.stale_pending_suspend_archive`（默认 7 天）执行清理/归档
  - `status in (Completed, Failed, Expired)` 的历史中间态记录按 `retention_days.task_history`（默认 30 天）清理
- 观测点：`expired_stale_task_count` 与 `deleted_report_generation_task_count` 计数落库

**步骤 3：过期订单清理**
- 调用：L2 → L3 → L4
- 规则：FR-09 `PENDING` 超 15 分钟 → `EXPIRED`（若守护轮询已处理过则跳过）

**步骤 4：审计日志归档（可选）**
- 审计日志超 90 天 → 归档压缩（不删除，审计要求保留）

### 超时阈值与动作

- 单轮清理任务超过 `config.cleanup_task_timeout_seconds`（默认 1800s）→ 任务状态收敛为 `FAILED`，`status_reason="cleanup_timeout"`，触发 NFR-13 告警
- 任一子步骤超时仅影响该子步骤结果，不允许破坏步骤 0 的永久保留域保护约束

### 步骤关系

步骤 0 先执行；步骤 1~4 在保护清单固化后可并行执行、互不阻塞

### 全链路失败返回

引用 §0.2 标准结构。清理任务失败仅记录日志+告警，不阻塞任何业务链路（失败子项允许下一轮重试）。

## §FR-10 数据看板与前端消费

### 链路起点

前端 SPA 请求（已鉴权/未鉴权 → 不同 `tier` 数据深度）

### 数据就绪状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `READY` | 数据已就绪 | 后端查询成功且缓存有效 |
| `COMPUTING` | 后端异步计算中 | 对照组/结算异步任务未完成 |
| `DEGRADED` | 数据可用但含降级标记 | `quality_flag != ok` 或 `signal_validity_warning=true` |

> 冷启动/无该周期数据不得新增 `EMPTY` 枚举，统一以 `COMPUTING` 或 `DEGRADED` + `status_reason` 表达（与 01 FR-10、02 §4.5.1 对齐）。

### 执行链路

**步骤 1：鉴权与权限降级**
- 调用：L1（JWT 鉴权 → tier 提取）
- 规则：
  - `tier=Free`：可访问首页/列表/详情结论区；`GET /api/v1/reports/{id}/advanced` 返回后端裁剪摘要（≤200字符，含省略标记）；`sim-dashboard` 仅允许 `capital_tier=100k`（未传默认100k，非100k返回403）
  - `tier=Pro|Enterprise`：完整研报内容、完整统计拆分、对照组
  - `role=admin`（任意权益挡位）：同 Pro/Enterprise + 管理端
- 超时阈值与动作：鉴权链路超过 `config.frontend_auth_timeout_seconds`（默认 3s）返回 503，前端展示“服务繁忙，请稍后重试”

**步骤 2：研报列表查询**
- 调用：L2 → L4（分页查询）
- 输入：`trade_date`（可选）、`page`、`page_size`（默认20，上限100）
- 输出：按 `trade_date DESC, confidence DESC` 排序的研报摘要列表
- 降级数据标记：响应中 `quality_flag != ok` → `degraded_banner="⚠️ 当前研报包含降级数据，投资决策请谨慎"`
- 超时阈值与动作：查询超过 `config.reports_list_timeout_seconds`（默认 5s）返回空 `items` + `data_status=COMPUTING` + `status_reason="list_query_timeout"`，不修改业务数据

**步骤 3：研报详情查询**
- 调用：L2 → L4
- 输入：`report_id`
- 输出：完整研报（含推理链/citations/sim_trade_instruction）
- 权限控制：详情接口对未登录与 Free 均返回基础可见字段；高级区由 `GET /api/v1/reports/{id}/advanced` 控制（未登录 401，Free 200 摘要≤200字符，Pro/Enterprise 200 完整）
- 超时阈值与动作：详情查询超过 `config.report_detail_timeout_seconds`（默认 5s）返回 503，前端保留上次渲染并提示重试

**步骤 4：统计看板查询**
- 调用：L2 → L4（聚合查询+缓存）
- 输出：四维度统计指标（`win_rate`/`profit_loss_ratio`/`cumulative_return_pct`/`coverage_pct`）× strategy_type × 结算窗口
- 样本 < 30 → 不展示该粒度
- 对照组数据（baseline_random/baseline_ma_cross）独立字段
- `signal_validity_warning=true` → 展示"⚠️ 当前信号有效性低于随机水平"
- `data_status` 字段标注就绪状态
- **对照组独立就绪分级分支（HC-22 对齐）**：
  - 主要绩效指标（四维度+净值曲线）已就绪 **且** 对照组（baseline_random/baseline_ma_cross）尚在异步计算中 → `data_status=READY`，对照组相关字段为 `null`，`display_hint="基线对照数据计算中"`
  - 主要绩效指标已就绪 **且** 对照组已就绪 → `data_status=READY`，对照组字段正常返回
  - 主要绩效指标自身未就绪 → `data_status=COMPUTING`，`status_reason` 必填
  - 数据源降级或异常 → `data_status=DEGRADED`，`status_reason` 必填
- 超时阈值与动作：聚合查询超过 `config.dashboard_stats_timeout_seconds`（默认 8s）返回 `data_status=COMPUTING` + `status_reason="stats_aggregating_timeout"`

**步骤 5：模拟持仓看板查询**
- 调用：L2 → L4
- 输出：三挡独立净值曲线 + 仓位明细 + 回撤状态
- 各挡位 `total_asset`/`cash_available`/`drawdown_state` 独立展示
- `COMPUTING` 状态 → 轮询提示
- `is_simulated_only` 标记（01 FR-10 免责申明约束）
- 超时阈值与动作：`sim-dashboard` 查询超过 `config.sim_dashboard_timeout_seconds`（默认 8s）返回 `data_status=COMPUTING` + `status_reason="sim_dashboard_timeout"`

**步骤 6：前端消费约束**
- 所有涉及金额/收益率的展示须加 `is_simulated_only` 标记（模拟非实盘）
- 免责声明文案固定由后端返回，前端不自行编辑

### 观测点

- 状态落库点（上游复用）：FR-07 `settlement_status`、FR-08 `position_status/drawdown_state` 为 FR-10 的权威状态来源，FR-10 不另建业务状态枚举
- 查询侧观测：L3 记录 `request_id/data_status/status_reason` 访问日志；当 `data_status=DEGRADED` 连续触发阈值时发送 NFR-13 告警（`alert_type=DASHBOARD_DEGRADED`）

### 步骤关系

各查询接口独立、无依赖

### 全链路失败返回

引用 §0.2 标准结构。查询接口不修改业务状态，降级标记透传到前端展示；查询失败不阻塞写链路。

  - `quality_flag=degraded` → `published=false`，需管理员确认
  - `quality_flag=missing` → 不允许发布
  - 降级标记字段缺失 → 阻断发布（FR-00 约束 3）

**步骤 2：自动发布 / 人工确认**
- 调用：L2 → L3（事务）→ L4

**步骤 3：阶段 2 模拟延迟解耦**
- 规则：FR-08 模拟实盘结果不阻塞 FR-06 发布子链路；模拟延迟交付（T+1 开盘后）
- 发布完成后 FR-08 开仓信号异步入队

**步骤 4：下架操作**
- 调用：FR-12 `PATCH /api/v1/admin/reports/{id}` 设 `published=false`
- 审计记录：`action_type="UNPUBLISH"`
- FR-00 约束：已发布内容不可修改，仅可下架

### 步骤关系

1→2 串行；3 异步；4 由管理端触发

### 全链路失败返回

引用 §0.2 标准结构。发布失败不影响研报已落库状态。

## §FR-11 用户反馈

### 链路起点

用户前端点击「研报有帮助/有问题」（携带 JWT 鉴权）

### 输入来源与输出去向

- 输入来源：前端 `report_id + feedback_type` 请求体、JWT 身份上下文
- 输出去向：`report_feedback` 落库、`review_flag` 状态流转、阈值命中后回链 FR-13 事件
- 回链说明：反馈与复审字段清单以 04/05 为准

### 反馈复审状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `NONE` | 无需复审 | 默认状态，负反馈未达阈值 |
| `PENDING_REVIEW` | 待人工复审 | 负反馈去重后 ≥ 3 次 |
| `APPROVED` | 复审通过（研报无误） | FR-12 管理端人工确认 |
| `REJECTED` | 复审拒绝（研报有误） | FR-12 管理端人工下架 |

### 执行链路

**步骤 1：L1 结构防御校验**
- 调用：L1（Controller层）
- 规则：请求体仅允许 `{ report_id, feedback_type }`，包含任意未声明字段（如 `reason` 等）直接返回 `422` 或 `400 + INVALID_PAYLOAD` 并拒绝解析。未登录返回 `401`。

**步骤 2：并发防刷限流（L3 原子计数）**
- 调用：L3（限流器/防刷Redis层）
- 规则：同一用户 `feedback_type=negative` 每日最多 20 次。
- 实现要求：必须使用原子递增机制（如 Redis `INCR`），超出配额返回 `429 Too Many Requests`，拒绝落库。
- 超时阈值与动作：限流存储超过 `config.feedback_rate_limit_timeout_seconds`（默认 2s）返回 503，拒绝落库并触发告警

**步骤 3：幂等去重与落库**
- 调用：L2 → L3（事务）→ L4
- 规则：
  - 校验 `report_id` 是否存在且 `published=true`，否则返回 `404` 或 `400 + REPORT_NOT_AVAILABLE`，拒绝落库。
  - 对于 `positive` 反馈：仅插入记录，不引发状态流转。
  - 对于 `negative` 反馈：按 `user_id + report_id` 去重；重复提交则幂等返回 HTTP 200，但反馈计数不增加。
- 超时阈值与动作：反馈落库超过 `config.feedback_write_timeout_seconds`（默认 3s）返回 503，不改变复审状态

**步骤 4：复审阈值评估**
- 调用：L2 → L4
- 规则：单份研报累计去重有效 `negative` 反馈 ≥ 3 次时，原子更新研报记录 `review_flag = PENDING_REVIEW`。
- 超时阈值与动作：阈值评估超过 `config.feedback_review_eval_timeout_seconds`（默认 3s）时终止本次流转并触发告警，下次请求重试评估

**步骤 5：复审告警联动（Outbox模式）**
- 规则：当且仅当 `review_flag` **首次**变更为 `PENDING_REVIEW` 时，在步骤 4 的同一数据库事务内写入 FR-13 事件外发记录 `event_type=REPORT_PENDING_REVIEW`。
- 观测点：FR-13 `notification` 表有新增记录（限定管理员通道）。

### 步骤关系

1→2→3→(针对negative且去重有效)4→(触阈值)5 严格串行。

### 全链路失败返回

引用 §0.2 标准结构。频控/拦截不产生服务端内部错误。

## §FR-12 管理端面板

### 链路起点

管理员（`role=admin`/`role=super_admin`）通过前端管理界面或 API 发起操作

### 管理操作状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `PENDING` | 操作已受理 | 管理员发起请求并通过 RBAC 校验 |
| `EXECUTING` | 操作执行中 | 进入事务处理（含级联检查） |
| `COMPLETED` | 操作成功 | 事务提交 + 审计日志写入成功 |
| `REJECTED` | 操作被拒绝 | 门禁校验失败（如级联阻断 409、权限不足 403） |
| `FAILED` | 操作异常 | 事务执行异常回滚 |

> 所有管理端写操作必须在 `COMPLETED` 终态同步产生审计记录（`actor_user_id/before/after/timestamp/request_id`）。`REJECTED` 状态同样记录审计（记录拒绝原因）。

### 执行链路

**步骤 1：RBAC 鉴权与参数校验**
- 调用：L1（JWT + RBAC + 入参校验）
- 输入：`role/sub/request_id` 与管理端请求体
- 输出：标准化 `admin_command`
- 失败处理：未登录 401；非 `admin/super_admin` 403；参数非法 422
- 超时阈值与动作：鉴权链路超过 `config.admin_auth_timeout_seconds`（默认 3s）返回 503，`status_reason="admin_auth_timeout"` 并触发告警

**步骤 2：操作路由编排**
- 调用：L2（`admin_orchestrator.py`）
- 输入：`admin_command`（`report|pool|user|billing|overview|audit`）
- 输出：子链路执行计划（写路径或读路径）
- 失败处理：未知操作类型返回 400（`INVALID_ADMIN_ACTION`）
- 超时阈值与动作：编排超过 `config.admin_route_timeout_seconds`（默认 3s）返回 503，不进入后续步骤

**步骤 3：门禁与幂等检查（写路径）**
- 调用：L2 → L3（预检）
- 规则：
  - 强制重建必须 `role=super_admin` 且通过 FR-00 级联检查（已有 FR-08 引用则 `REJECTED + 409`）
  - 补单操作以 `event_id/order_id` 幂等去重（重复请求不重复发放权益）
  - 已发布只读保护：禁止修改研报业务内容字段
- 输出：`precheck_passed=true|false`
- 失败处理：门禁不通过收敛为 `REJECTED`，保留原状态不写业务变更
- 超时阈值与动作：预检超过 `config.admin_precheck_timeout_seconds`（默认 5s）收敛为 `FAILED`，`status_reason="admin_precheck_timeout"`

**步骤 4：事务执行与状态收敛（写路径）**
- 调用：L3（事务边界）→ L4（持久化）
- 输入：通过预检的写操作计划
- 输出：`COMPLETED|FAILED`；业务实体状态与管理操作状态一致收敛
- 失败处理：任一步异常回滚并收敛为 `FAILED`
- 超时阈值与动作：事务超过 `config.admin_write_timeout_seconds`（默认 8s）回滚并返回 500

**步骤 5：审计写入与告警（写路径）**
- 调用：L3（审计门禁）
- 规则：`COMPLETED` 与 `REJECTED` 均必须写审计 `{ actor_user_id, before, after, request_id, timestamp }`
- 失败处理：审计写入失败视为操作失败，回滚业务写入并返回 500
- 观测点：审计记录落库；`FAILED/REJECTED` 触发 NFR-13 告警（如 `alert_type=ADMIN_OP_FAILED/FORCE_REGENERATE_BLOCKED`）

**步骤 6：看板与审计查询（读路径）**
- 调用：L2 → L4（聚合查询/分页查询）
- 输出：管理员概览、审计日志列表、系统健康视图（允许最多 5 分钟缓存）
- 失败处理：查询失败时返回标准错误结构，不修改任何业务状态
- 超时阈值与动作：读路径超过 `config.admin_read_timeout_seconds`（默认 5s）优先返回缓存快照；无缓存则返回 503

### 观测点

- 状态落库点：管理操作状态（`PENDING/EXECUTING/COMPLETED/REJECTED/FAILED`）与审计记录落库
- 告警触发点：门禁拒绝、事务回滚、审计写入失败均触发 NFR-13 告警

### 步骤关系

写路径：1→2→3→4→5 严格串行；读路径：1→2→6

### 全链路失败返回

引用 §0.2 标准结构。管理端失败不影响交易主链路；被拒绝操作必须保持原业务状态不变。

## §FR-13 通知与事件子系统（Outbox 模式）

### 链路起点

业务事件产生（事务内写入 Outbox 表，HC-02）

### 输入来源与输出去向

- 输入来源：FR-06/08/11 产生的业务事件与对应业务主键（`event_type + payload`）
- 输出去向：`outbox_event` 派发状态落库 + `notification` 发送记录落库；结果供 FR-10/FR-12 查询与审计
- 回链说明：Payload 字段结构以 01 FR-13 与 05 契约为准，03 不扩展新字段枚举

### 事件处理状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `CREATED` | 业务事件已产生，待幂等投影 | 业务链路触发事件 |
| `DEDUP_SKIPPED` | 命中幂等去重，事件不入派发队列 | 投影键命中当日重复 |
| `ENQUEUED` | 事件已写入 Outbox，等待派发 | 去重通过并成功入队 |

### Outbox 派发状态定义

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `PENDING` | 已入 Outbox 表，待投递 | 事务提交时写入 |
| `DISPATCHING` | 正在执行投递 | 轮询器取出 |
| `DISPATCHED` | 投递成功 | 所有通知渠道处理完成 |
| `DISPATCH_FAILED` | 投递失败（已重试） | 重试耗尽 |

### 通知记录状态定义（01 FR-13 对齐）

> Outbox 派发状态管理消息**投递链路**（内部技术状态），通知记录状态管理**业务通知结果**（对外可观测状态）。两套状态分层独立。

| 状态 | 含义 | 迁移条件 |
|---|---|---|
| `sent` | 通知成功发送 | 渠道（Webhook/邮件）返回成功 |
| `failed` | 通知发送失败 | 重试耗尽仍未成功 |
| `skipped` | 通知被跳过 | 幂等去重 / 用户未配置渠道 / 权益不符 / 抑制窗口命中 |

### 事件类型枚举（01 FR-13 对齐，四类业务事件）

| 事件类型 | 触发源 | Payload 结构（01 FR-13 定义） |
|---|---|---|
| `POSITION_CLOSED` | FR-08 仓位 `position_status` 从 `OPEN` 变更为 `TAKE_PROFIT/STOP_LOSS/TIMEOUT` | `{ stock_code, stock_name, position_status, capital_tier, actual_entry_price, exit_price, net_return_pct, holding_days, signal_date, close_date }` |
| `BUY_SIGNAL_DAILY` | FR-06 每日研报批次完成后，汇总当日 `recommendation=BUY AND confidence>=0.65` 的研报列表 | `{ trade_date, signal_count: int, signals: [{ stock_code, stock_name, confidence, strategy_type }]（按 confidence 倒序，最多5条）}` |
| `DRAWDOWN_ALERT` | FR-08 `drawdown_state` 进入 `REDUCE` 或 `HALT` | `{ capital_tier, drawdown_pct, drawdown_state, triggered_at }` |
| `REPORT_PENDING_REVIEW` | FR-11 触发 `review_flag=PENDING_REVIEW` | `{ report_id, stock_code, trade_date, negative_count, review_flag, triggered_at }` |

### 幂等键投影规则（HC-01 对齐）

> 口径以 01 §FR-13 与 02 HC-01 为权威准据。所有四类事件当日仅推送一次，重复触发幂等跳过（`status=skipped`）。

| 事件类型 | 幂等键维度 | 去重粒度说明 |
|---|---|---|
| `POSITION_CLOSED` | `(event_type, stock_code, trade_date, capital_tier)` | 同一持仓同日同挡仅推一次 |
| `BUY_SIGNAL_DAILY` | `(event_type, trade_date, capital_tier=NULL)` | 每日汇总推送仅一次（非每只股票单独推送）；无挡位语义时 `capital_tier` 维度固定为空 |
| `DRAWDOWN_ALERT` | `(event_type, capital_tier, trade_date)` + **4h 抑制窗口** | 同一 `capital_tier` 在 `drawdown_state in (REDUCE, HALT)` 持续期间每 4 小时仅发送首报一次，后续同态告警抑制，直至状态恢复后才重新放行首报 |
| `REPORT_PENDING_REVIEW` | `(event_type, stock_code, trade_date, capital_tier=NULL)` | 与 01/02 统一按 `stock_code/trade_date` 投影去重；`report_id` 仅用于追溯，不进入幂等维度 |

### 推送渠道路由规则

| 事件类型 | 管理员通道（Webhook+邮件） | Pro/Enterprise 用户通道（邮件） |
|---|---|---|
| `POSITION_CLOSED` | ✅ | ✅（`config.notification_channels.user_email_enabled=true` 时） |
| `BUY_SIGNAL_DAILY` | ✅ | ✅（`config.notification_channels.user_email_enabled=true` 时） |
| `DRAWDOWN_ALERT` | ✅ | ❌（仅管理员） |
| `REPORT_PENDING_REVIEW` | ✅ | ❌（仅管理员） |

### 执行链路

**步骤 1：事务内事件写入（Outbox 表）**
- 调用：L3（与主业务共享事务边界）
- 输入：主业务事务上下文 + `event_type` + 规范化 `payload`
- 输出：Outbox 记录（`status=PENDING` 或去重后 `status=skipped`）
- 规则（HC-02）：
  - 事件与主业务数据在同一事务内写入 Outbox 表
  - 事务回滚 → Outbox 记录同步回滚，不发送通知
  - 保证 at-least-once 语义
  - FR-06 批次完成但 `BUY_SIGNAL_DAILY.signal_count=0` 时：仅记录“无强信号”运行日志，不创建该事件 Outbox 记录
  - 写入时同步执行幂等键去重检查（按上方投影规则），命中则直接 `status=skipped` 不入队

**步骤 2：异步轮询派发**
- 调用：L3（`outbox_dispatcher.py`）
- 输入：`status=PENDING` 的 Outbox 事件批次
- 输出：派发中的事件集（`DISPATCHING`）与幂等抑制后 `skipped` 结果
- 规则：
  - 后台轮询器定期（间隔2秒）扫描 `status=PENDING` 事件
  - 批量取出并标记 `DISPATCHING`（乐观锁防并发重复取）
  - 按事件类型路由到对应 handler
  - **DRAWDOWN_ALERT 4h 抑制**：派发前检查同一 `capital_tier` 在最近 4h 内是否已有 `sent` 记录且 `drawdown_state` 未恢复，命中则直接标记 `skipped`
- 观测点：Outbox 状态迁移 `PENDING→DISPATCHING` 落库

**步骤 3：通知渠道执行**
- 调用：L3 → L4（通知适配器）
- 输入：步骤 2 路由后的事件与渠道配置
- 输出：分渠道发送结果（`sent/failed/skipped`）
- 渠道路由：按上方"推送渠道路由规则"表分发
- 各渠道独立执行，单渠道失败不阻塞其他渠道
- 推送内容禁止包含用户 email/user_id/JWT 等个人身份信息（NFR-16）

**步骤 4：重试与终态收敛**
- 输入：步骤 3 的分渠道执行结果
- 输出：Outbox 终态（`DISPATCHED/DISPATCH_FAILED`）与通知记录终态（`sent/failed/skipped`）
- 重试策略：失败后重试一次（01 FR-13 口径），失败后记录 `status=failed`
- 推送失败不阻塞主业务链路（FR-08 结算/FR-06 研报批次不因推送失败回滚）
- 观测点：`status=failed` 触发 NFR-13 告警
- 通知记录入库字段：`notification_id(UUID), event_type, triggered_at(ISO8601), channel(email/webhook), status(sent/failed/skipped), payload_summary`

### 超时阈值与动作

- 出队声明超时：`DISPATCHING` 状态持有超过 `config.outbox_dispatch_claim_timeout_seconds`（默认 5s）未进入渠道执行时，回滚为 `PENDING` 并写 `status_reason="outbox_claim_timeout"`
- 单渠道发送超时：超过 `config.notification_channel_timeout_seconds`（默认 10s）时标记该渠道 `failed`，进入步骤 4 重试判定
- 单事件派发总超时：超过 `config.outbox_dispatch_timeout_seconds`（默认 60s）收敛为 `DISPATCH_FAILED` + `status_reason="outbox_dispatch_timeout"`，触发 NFR-13 告警

### 步骤关系

1 在业务事务内同步；2→3→4 异步后台轮询

### 全链路失败返回

引用 §0.2 标准结构。事件派发失败不影响主业务链路结果（at-least-once，业务已提交），仅影响通知可达性并进入重试/告警闭环。

## §10 研报质量与收益提升设计

> **来源**：`01_需求基线.md` §1.6（研报准确率与后续优化）、FR-06（先验数据注入规则）  
> **SSOT 角色**：研报质量改进的完整闭环链路定义（唯一来源）  
> **边界**：本章不定义数据库字段（→ 04），不定义接口（→ 05）；只定义改进流程与门禁标准

### §10.1 问题发现路径

> 三条信号触发研报质量改进闭环，各路径独立可追溯。

| 发现来源 | 触发条件 | 产出信号 | 责任层 |
|---------|---------|---------|-------|
| **FR-07 结算误判** | `settlement_status=settled AND net_return_pct < -0.05`（策略性亏损超5%）持续出现 | 潜在误判样本，按 `strategy_type × market_state` 分组统计 | L2（settlement.py）|
| **FR-11 用户负反馈** | 去重后 `negative` ≥ 3 次触发 `review_flag=PENDING_REVIEW` | 研报需人工复审 | L2（feedback.py）→ FR-12 |
| **FR-12 管理员标记** | 管理员 PATCH `review_flag=REJECTED` 下架研报 | 正式负样本，可纳入 Few-shot 库 | L2（admin.py）|

### §10.2 归因分类（五维度）

| 类别 | 识别特征 | 典型场景 | 改进动作 |
|------|---------|---------|--------|
| **信号错** | 规则引擎 `strategy_type` 判定与实际走势不符（A 判成 B 等） | 事件驱动型股票被误判为趋势跟踪 | 调整 A/B/C 判定阈值参数（需更新 01 FR-06）|
| **执行错** | 止损/止盈触发时机不当，机械执行导致过早离场 | ATR 倍数偏低导致正常波动触发止损 | 调整 ATR 止损倍数（A×1.5/B×2.0/C×2.5）|
| **数据错** | `citations` 来源质量差（`quality_flag=stale_ok/degraded`）或热搜数据不准确 | 采集降级时仍触发强信号 | 提升数据质量门槛（可追加熔断阈值调整）|
| **事件误读** | LLM 对热搜标题/政策新闻理解有偏差，推理链与实际相悖 | 利空新闻被误判为利好 | Few-shot 负样本注入（见 §10.3 步骤3）|
| **市场状态错** | `market_state=NEUTRAL` 期间 BUY 信号大量亏损（应当偏空或观望） | 震荡市中系统持续发出 BUY 信号 | 调优 BULL/BEAR 判定参数（需更新 01 FR-05 边界）|

**归因标注规则（辅助自动+人工确认）**：
- `market_state=NEUTRAL AND net_return_pct < -0.08` → 初始标注建议 `market_state_error`
- `quality_flag=stale_ok/degraded AND net_return_pct < -0.05` → 初始标注建议 `data_error`
- `stop_loss` 触发后当日持续涨超3% → 初始标注建议 `execution_error`
- 其余由 FR-12 管理员在复审界面手工确认 `failure_category`

### §10.3 改进动作链路

**步骤 1：负样本聚合（L2 自动）**
- 调用：L2（`settlement.py` + `admin.py`）→ L4（数据库读取）
- 数据来源：
  - FR-07 `settlement_status=settled AND net_return_pct < -0.05` 的记录
  - FR-11/FR-12 触发 `review_flag=REJECTED` 的研报
- 聚合维度：按 `strategy_type × market_state` 分组，计算失败率
- 触发门槛：负样本聚合在 FR-09-b 清理任务周期内（每周）自动执行统计，不实时触发
- 观测点：统计结果记录到 `experiment_log` 表（字段见 04）

**步骤 2：归因标注（L2 辅助 + FR-12 人工确认）**
- 调用：L2（`admin.py`）→ FR-12 管理界面
- 规则辅助标注（见 §10.2 归因标注规则），输出建议 `failure_category`
- 管理员在 FR-12 复审界面确认或修正 `failure_category`
- 确认后状态流转：`PENDING_REVIEW → REJECTED`（含 `failure_category` + `reason_code` 写审计日志）

**步骤 3：改进动作执行**

```text
Prompt 修改 / 阈值调整流程：
  → 改 01 文档（FR-05 边界 / FR-06 边界）→ 同步 02/03/04/05
  → 更新 config.llm_system_prompt（对应归因类型的修正指令）
  → 执行 §10.4 实验门禁
  → 通过后 → 上线生效

Few-shot 样本注入流程：
  → 正样本来源：settlement net_return_pct > 0.05 AND quality_flag=ok AND published=true
  → 负样本来源：review_flag=REJECTED 且 failure_category=event_misread 的研报
  → 样本格式：{ strategy_type, market_state, input_summary: str, reasoning_chain_key: str, outcome: str }
  → 注入位置：FR-06 步骤 5（先验数据注入）的 Prompt 构建中，附在 prior_stats 之后
  → 注入门槛（硬约束）：
      每种 strategy_type 至少 5 个正样本 + 3 个负样本方可注入
      样本 signal_date 必须 < data_cutoff（与 §10.4 时序切割对齐，防数据泄漏）
      不满足门槛时跳过注入，prior_stats 照常注入（若满足其门槛）
  → 注入控制开关：config.few_shot_enabled（默认 false，明确配置后才启用）
```

**步骤 4：回滚机制**
- 上线后 7 日内若满足回滚条件（见 §10.4）→ 恢复上一版 Prompt/参数
- 回滚操作记录审计日志（`action_type=PROMPT_ROLLBACK`，纳入 FR-12 审计范围）

### §10.4 实验门禁（防回测污染）

> **强制约束**：任何影响研报结论的参数/Prompt/模型变更，必须通过本节门禁后才可上线。

**时序切割（Walk-Forward，防数据泄漏）**：

```text
训练集（可用于参数调优/样本选取）：
  signal_date < data_cutoff（同 FR-06 §5 先验数据时序切割口径：当月1日）

测试集（用于门禁验收，绝对禁止用于参数调优）：
  signal_date ≥ data_cutoff
  要求：至少 30 个已结算信号（与 FR-07 样本门槛一致）

泄漏检查（必须通过）：
  验证 prior_stats.data_cutoff < 任意测试信号的 signal_date
  若发现测试集数据渗入训练集 → 立即终止实验，不得声称通过
```

**上线判定标准（满足其一即视为不劣化，可上线）**：
1. 测试集 `win_rate ≥ 55%`（NFR-04 目标对齐）
2. 或：测试集 `win_rate ≥ 近30日生产 win_rate × 0.95`（不劣化容忍5%）

**回滚触发条件（满足其一即触发回滚）**：
- 上线后 7 日内 `win_rate < 近30日生产基线 × 0.90`（劣化超10%）
- 或：上线后 7 日内出现 3 次以上 FR-11 用户负反馈的研报（同一版本 Prompt）

### §10.5 关联链路与观测点

| 维度 | 内容 |
|------|------|
| **起点** | FR-07 结算完成事件 / FR-11 负反馈达阈 / FR-12 管理员标记 REJECTED |
| **终点** | Prompt 变更或参数更新完成上线（通过实验门禁）；或实验门禁未通过保持现状 |
| **状态落库点** | `experiment_log` 表记录每次实验（data_cutoff、test_win_rate、baseline_win_rate、passed）|
| **告警触发点** | 实验门禁未通过 → NFR-13 告警（`alert_type=EXPERIMENT_GATE_FAILED`）；回滚触发 → NFR-13 告警（`alert_type=PROMPT_ROLLBACK`）|
| **全链路失败** | 改进动作失败不阻塞主研报链路；实验门禁未通过则不上线，保持当前生产 Prompt/参数 |
| **SSOT 联动** | 参数调整须先改 01（FR-05/FR-06 边界），并按 99 行动11 治理流程同步其余文档 |
