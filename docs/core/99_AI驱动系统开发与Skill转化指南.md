# AI 驱动系统开发完整方法论

> **文档编号**：42  
> **核心问题**：从零开始，如何一步步拿到系统全部功能？如何把功能变成 Skill？如何让 AI 实现整个系统？  
> **贯穿示例**：yanbao 研报平台，所有路径、提示词均来自本项目真实过程。

---

## 使用方法

**按编号顺序执行行动清单，每个行动后面标注了它属于哪一章。**  
遇到问题时，跳到对应章节找详细说明和纠偏方法。  
每章末尾有「阶段门禁」——不满足就不能往下走。

## 全量重写模式说明

> **适用场景**：在已有半成品代码库（如本项目 yanbao）上全量重写，而非从空白目录开始。

### 旧代码/旧文档的使用规则

| 资源 | 允许的用法 | 禁止的用法 |
|------|-----------|-----------|
| `docs/old/**` | 作为灵感来源、识别反例（"旧设计哪里有问题"） | **禁止作为规格来源**（不能说"按 old/05 定义实现"） |
| 旧代码 `app/**`（已存在） | 查看现有实现思路，了解坑 | **禁止作为隐性规格**（不能让 AI"参考现有实现"而跳过新 FR 文档） |
| 旧测试 `tests/**` | 参考测试风格/工具链 | **禁止混用旧测试通过率来衡量新 FR 完成度** |

**核心原则**：重写期间，唯一规格来源是 `docs/core/`（01～05），`docs/old/` 与旧代码仅作背景参考。

### 三条硬约束

```
① SSOT 污染预防（禁止模糊引用，允许受控对照）：
  禁止在提示词里出现"参考旧实现"、"保持与老版本一致"等无边界表述；
  但允许且建议使用【旧实现对照（非规格）】块，并在其中明确"仅参考范围"
  （见行动 21 模板及附录 B）。
  两者区别：
    ❌ 禁止："参考现有实现" / "保持与旧代码一致"（无边界，AI 会自行推断规格）
    ✅ 允许：【旧实现对照（非规格）】仅参考：目录结构/函数签名风格/异常抛出方式
             字段名/枚举/错误处理仍以 05/04/FR 为准，old/** 禁止引用

② 旧代码不自动继承：重写不等于"在旧代码上打补丁"，
  若 AI 输出的代码引用了 old/ 目录或旧数据结构，立即停止并指出。

③ 差距分析以新 FR 为基准：行动 28 不评估"旧功能是否还在"，
  只评估"新 FR 是否按新 SSOT 实现"。
```

## 行动 00：重写启动检查清单

在开始任何需求分析或代码生成之前，逐项完成以下决策：

### 0-A：新 SSOT 文档清单

明确本轮重写的权威文档列表，贴到 `AGENTS.md` 顶部（或在对话开头声明）：

```
本轮重写 SSOT 清单（唯一规格来源）：
  01_需求基线.md        → FR 编号与验收标准
  02_系统架构.md        → 分层图与调用规则
  03_详细设计.md        → 执行链路与状态机
  04_数据治理与血缘.md  → 数据模型与枚举值
  05_API与数据契约.md   → 全接口 Schema

旧文档（仅灵感/反例，禁止作为规格来源）：
  docs/old/**          → 已废弃，不引用
  旧代码 app/**         → 可查阅，但实现以新 FR 文档为准
```

### 0-B：数据库处理策略（三选一，必须明确）

```
选项 A（推荐）：清库重建
  - 删除或重命名旧 SQLite 文件（如 yanbao.db → yanbao.db.bak）
  - 由新 models.py 重建所有表
  - 适用于：表结构大幅变化、旧数据不保留

选项 B：保留数据，手动迁移
  - 用 Alembic 生成迁移脚本
  - 必须在行动 10 完成后才能运行迁移
  - 适用于：有真实数据需要保留

选项 C：双库并存（过渡期）
  - 旧库只读，新库由新代码写入
  - 最后切换，风险最低但工作量最大
```

> **本项目默认选 A（清库重建）**：全量重写时数据结构变化较大，旧数据为测试数据，不需要保留。若你的项目有真实用户数据，改选 B 并在行动 10 补充迁移脚本说明。

### 0-C：旧测试处理策略

```
□ 检查 tests/ 目录里有哪些旧测试
□ 决策：
    旧测试与新 FR 兼容 → 保留并在重写时同步更新
    旧测试基于旧接口/旧字段 → 隔离处理（见下方）
□ 禁止：把旧测试通过率作为新 FR 完成度的衡量标准
```

**如何保证旧测试不被 pytest 自动收集**（三选一，必须明确选定一种）：

| 方案 | 操作 | 注意事项 |
|------|------|---------|
| **推荐：重命名文件** | 把 `test_old_xxx.py` 改名为 `legacy_old_xxx.py` | pytest 默认只收集 `test_*.py`，改名后自动排除 |
| 加 `pytest.ini` 排除 | `[pytest] norecursedirs = tests/legacy` | 配合移入 `tests/legacy/` 子目录使用 |
| 加 `@pytest.mark.skip` | 在旧测试文件顶部加 `pytestmark = pytest.mark.skip("legacy")` | 文件仍被收集但全部跳过，适合临时隔离 |

> **本项目推荐方案一（重命名）**：最简单，不依赖配置，任何 pytest 版本都有效。  
> 执行命令示例：
> ```powershell
> # 把旧测试文件重命名，使其不被 pytest 收集
> Get-ChildItem tests/ -Filter "test_*.py" |
>   Where-Object { $_.Name -match "old|legacy|v1" } |
>   Rename-Item -NewName { "legacy_" + ($_.Name -replace '^test_', '') }
> ```
> 重命名后运行 `pytest tests/ --collect-only` 确认旧测试不再出现在收集列表里。

### 0-D：目录重组原则

```
□ 确认 app/ 目录结构是否需要调整（重命名、拆分服务等）
□ 若需调整：先画新目录树，让 AI 确认后再动文件
□ 禁止：边写代码边改目录结构（会导致 import 全部报错）
□ 调整完成后，更新 docs/core/02_系统架构.md 的目录说明
```

### 0-E：本轮工作宣言（贴到每个新对话开头）

```
本次工作说明：
- 模式：全量重写（不是修补旧代码）
- SSOT：docs/core/01/02/03/04/05（见上方清单）
- 旧代码：只用于查阅，不继承旧逻辑
- 数据库：清库重建（选项 A）/ 迁移（选项 B）
- 完成标准：新 FR 全部按新 SSOT 实现并通过 pytest
```

# 第一章：发现全部功能

> **目标**：在写任何代码之前，把系统的所有功能都列出来，每个功能有编号、有输入输出、有边界、有验收方式。  
> **产出**：`docs/core/01_需求基线.md`

AI 生成代码的质量上限，由约束质量决定。需求模糊时，AI 每次填空都不一样，后续每修一处都会引入新的不一致。

## 行动 01：5 问提示词（挖主干功能）

把以下内容复制给 AI，把括号里的内容替换为你的系统描述：

```
我想做一个【在此描述你的系统，越具体越好。例如：
A 股个股研报平台，每天对目标股票生成一份研究报告，
包含买卖建议和价格预测，并通过模拟持仓验证准确率。】

请帮我回答以下 5 个问题，每个问题给 3-5 条清晰答案，不要泛泛而谈：

1. 这个系统要处理哪些「输入数据」？
   每种数据的来源是什么？格式是什么？获取失败时怎么处理？

2. 这个系统会产生哪些「输出」？
   输出给谁看？以什么格式呈现？

3. 哪些处理步骤必须「按顺序」执行？哪些可以并行？

4. 系统中有哪些「状态」需要长期保存？
   （如用户信息、历史记录、配置项等）

5. 哪些功能是「绝对不能出错」的？出错时系统应该怎么反应？
```

**合格输出的判断标准**：每个问题至少 3 条答案，且每条答案是具体的（有名词、有场景），不是泛化的（如「处理好错误」「保存数据」不算具体）。

## 行动 02：6 项遗漏检查（挖隐藏功能）

在同一对话继续说：

```
基于上面的功能清单，请检查以下 6 个方面，
逐一说明「我的系统是否需要」以及「具体要做什么」：

1. 用户身份与权限：谁可以用这个系统？不同用户看到的内容有什么区别？

2. 错误与降级：每个关键步骤失败时，系统应该怎么做？
   （显式报错 / 降级展示 / 静默跳过 / 重试）

3. 定时与调度：哪些任务需要定时触发？触发频率是什么？

4. 历史与查询：用户需要查看哪些历史数据？支持哪些筛选/排序/分页？

5. 监控与运维：系统健康状态如何检查？出问题时如何感知？

6. 外部依赖：哪些功能依赖第三方服务？这些服务不可用时系统怎么运行？
```

**yanbao 通过这步发现的隐藏功能（参考）**：
- Cookie 会话管理（FR-02）：微博/抖音采集依赖 Cookie，需健康探测与自动刷新
- 定时调度（FR-08）：每日 09:00 市场状态计算，15:20 触发研报生成
- 三级会员权益（FR-06）：Free/Pro/Enterprise 看到不同内容
- 回撤熔断（FR-04 子功能）：亏损超 -12% 减仓、超 -20% 停止开仓
- LLM 多层降级（FR-03 依赖）：主 API → 备 API → CLI → 本地模型

## 行动 03：整理成 FR 编号表

```
请把上面所有功能整理成需求基线表，每条 FR 包含：

- 编号：FR-xx（两位数，从 00 开始）
- 功能域名称（3-5 个汉字）
- 输入：谁触发 + 传入哪些参数（写字段名和类型）
- 输出：返回什么（写字段名、类型、枚举值，不能只说"返回数据"）
- 边界：什么情况下降级、报错、静默（每种情况分别说明）
- 验收标准：具体的测试方法（pytest 断言什么 / 人工检查什么）

另外附上：
- NFR 列表（性能阈值、安全约束、真实性约束）
- 范围外说明（明确「不做什么」）

保存到 docs/core/01_需求基线.md
```

### FR 编号稳定性规则（全量重写时必读）

全量重写时 FR 会大量新增/删除，编号是测试命名、证据清单、差距清单的稳定锚点，**不稳定的编号会让所有引用失效**。

| 场景 | 规则 |
|------|------|
| 新增 FR | 追加到末尾（FR-xx 递增），禁止插入中间 |
| 删除 FR | **禁止删除编号**，改为 `[REMOVED] FR-xx：[原名]（已移除，不再实现）` |
| 拆分 FR | 原编号保留为父条目，子条目用 FR-xx-a / FR-xx-b 扩展 |
| 重命名 FR | 只改功能域名称，编号不变 |
| 合并多个 FR | 低编号保留，高编号标记为 `[MERGED INTO FR-xx]` |

**为什么禁止重排编号**：测试函数名（如 `test_fr01_hotspot`）、差距清单（`FR-01 ❌`）、文档引用（`见 FR-03`）都依赖编号，一旦重排，所有引用同时失效，追踪成本极高。

## 行动 04：逐条 5 问自检

每条 FR 必须能回答以下 5 问，缺任何一问就补充完整，再往下走：

```
1. 谁触发？
   → 用户手动 / 定时任务 / 另一个 API 调用 / 系统启动

2. 输入是什么？
   → 字段名（类型，示例值）
   → 不能只说"传入股票信息"，要写"stock_code: str，如 600519.SH"

3. 成功时输出什么？
   → 每个字段的名称、类型、枚举值
   → 不能只说"返回研报数据"，要写具体字段

4. 失败时怎么处理？
   → 抛异常 / 返回 degraded 状态 / 降级用备用数据 / 静默跳过
   → 每种失败原因分别说明处理方式

5. 如何验收？
   → pytest 断言哪个字段等于什么值
   → 或：人工打开哪个页面，检查哪里显示了什么
```

**不完整的 FR 与完整的 FR 对比**：

```
❌ 不完整（AI 看到后会自由发挥）：
FR-03：模型生成研报，输入股票代码，输出研报 JSON。

✅ 完整（AI 看到后有约束，每次实现一致）：
FR-03：模型生成研报
  触发：POST /api/v1/reports/generate 被调用
  输入：stock_code: str（如"600519.SH"）、
        trade_date: str（YYYY-MM-DD，默认当日交易日）
  输出：report_id: str、recommendation: "BUY"|"SELL"|"HOLD"、
        confidence: float(0.0-1.0)、
        citations: list[{source_name, source_url, fetch_time}]（非空）、
        idempotency_key: str（格式 daily:{stock_code}:{trade_date}）
  边界：① 同一 idempotency_key 重复触发 → 返回已有 report_id，不重新生成
        ② LLM 不可用 → 显式标记 llm_fallback=True，不静默填充假数据
        ③ BUY 且 confidence<0.65 → 不生成实操指令卡，不触发模拟开仓
  验收：pytest 断言 recommendation 在枚举内、citations 非空且每条含三元素；
        人工打开 /report/{id}，高级区显示推理链
```

## 行动 08：API 契约

这是最重要的规格文档，精度目标是：**看契约就能写出测试，不需要读代码**。

每个接口用下面的模板写：

```
为「[功能名]」编写 API 契约，写入 docs/core/05_API与数据契约.md。

格式要求（每个接口都要包含以下全部内容）：

方法 + 路径：
  [POST/GET/PATCH/DELETE] /api/v1/[资源路径]

Request：
  必填字段：字段名（类型，说明/枚举值）
  选填字段：字段名（类型，默认值，说明）

Response（HTTP 200）：
  {
    "success": true,
    "request_id": "uuid",
    "data": {
      字段名: 类型（说明/枚举值）
    }
  }

错误情况：
  HTTP 422：参数校验失败（列出哪些字段/值会触发）
  HTTP 404：[资源名]不存在
  HTTP 500：内部错误（含 error_code 业务错误码，列出可能的值）

特殊约束：
  幂等：[是/否]，若是：idempotency_key = [计算方式]
  鉴权：[无/内部 Key/JWT]
  分页：[是/否]，若是：支持 page/page_size/sort 参数
```

**合格输出的判断标准**：随便抽一个接口，你能用纯文档写出 pytest 测试（包含断言具体字段值），不需要看代码。

### 契约自动校验（门禁加固项）

契约写得再细，仍可能和 Pydantic 模型/路由实现不一致。建议在第四章的测试里加入**契约字段名校验**：

```python
# 抽样契约校验示例（针对 /api/v1/reports/generate）
def test_report_contract_fields(client):
    """验证响应字段名与 05_API契约 完全一致"""
    resp = client.post("/api/v1/reports/generate",
                       json={"stock_code": "600519.SH"})
    data = resp.json()
    # 必须存在的顶层字段（来自 05 §N）
    assert "success" in data
    assert "request_id" in data
    assert "data" in data
    # 业务字段枚举约束
    assert data["data"]["recommendation"] in {"BUY", "SELL", "HOLD"}
    # 422 触发条件验证
    bad = client.post("/api/v1/reports/generate", json={})
    assert bad.status_code == 422
```

**门禁要求**：完成行动 08 后，至少为 P0 接口抽样 3 个编写此类契约校验测试，否则"契约只是一篇作文"，AI 生成的代码随时可能和契约分叉。

### 统一错误/降级表示法（整个系统必须一致）

多步骤业务里，每个 FR 各写一套 `degraded/exception` 表示方式，前端和测试难以统一处理。在 `03_详细设计.md` 开头定义**全局标准**，所有执行链路引用这个标准：

```
# 全局错误/降级表示标准（存入 03_详细设计.md §0）

## 错误码枚举（所有接口的 HTTP 500 error_code 必须来自此表）
  DATA_SOURCE_UNAVAILABLE  数据源不可用（采集失败/超时）
  LLM_ALL_FAILED           所有 LLM 降级路径均失败
  VALIDATION_FAILED        内部数据校验失败
  IDEMPOTENCY_CONFLICT     幂等键冲突
  DEPENDENCY_NOT_READY     依赖的上游数据未就绪

## 降级字段标准结构（凡是降级响应，必须包含以下字段）
  {
    "success": false,
    "error_code": "<上方枚举之一>",
    "error_message": "<人类可读说明>",
    "degraded": true,           // 布尔，区分"完全失败"和"降级成功"
    "degraded_reason": "<说明>", // degraded=true 时必填
    "data": null                // 或降级数据（须显式标注来源）
  }
```

> 具体枚举值在 `05_API与数据契约.md` 的「错误情况」章节定义，`03` 只引用，不重复定义（SSOT 原则）。

### 变更治理流程（防止 SSOT 漂移）

重写期间最常见的漂移场景：改了代码里的字段名，但忘了同步文档；或改了文档枚举值，但 Pydantic 模型没同步。

**任何字段/枚举/接口变更，必须按以下顺序执行**：

```
步骤 1：先改 SSOT 文档
  → 05_API与数据契约.md（接口字段/枚举）
  → 04_数据治理与血缘.md（数据库字段/枚举）
  → 01_需求基线.md（FR 边界条件）
  只改一个地方，其他引用该字段的文档用「见 05 §N」

步骤 2：再改代码
  → app/models.py（数据库模型）
  → app/api/routes_*.py（路由/响应）
  → app/services/*.py（业务逻辑）

步骤 3：最后改测试
  → tests/test_*.py（断言字段名/枚举值）

步骤 4：运行全量测试确认
  pytest tests/ -v
```

**禁止行为**：先改代码后补文档（AI 很容易这样做，需要在提示词里明确禁止）。  
**识别信号**：若 AI 说"已实现，文档稍后补"——立即叫停，强制先改文档。

## 行动 15：写 Rule 文件

**Rule 文件格式**（存放路径：`.cursor/rules/role-xxx.mdc`）：

```yaml
---
description: [一句话：这个角色在什么情况下激活，包含触发场景词]
globs: [文件路径模式，逗号分隔]
alwaysApply: false
---

# [角色名称]

你正在以 **[角色名]** 身份工作。职责：[职责 1]、[职责 2]、[关联 FR 编号]。

**本角色实现前必须先读**：
- `docs/core/05_API与数据契约.md` §[章节号]（接口字段定义）
- `docs/core/04_数据治理与血缘.md`（数据规范）

**本角色特有的约束**（不同于其他角色的规范）：
- [约束 1]
- [约束 2]
- [约束 3]
```

**四个关键点**：

| 要素 | 好的写法 | 差的写法 |
|------|---------|---------|
| `globs` | `app/**/hotspot*.py, app/**/collect*.py` | `app/**/*` （太宽，所有文件都触发）|
| `description` | 「以数据工程师角色处理采集、标准化、血缘、report_data_usage」 | 「数据处理」（触发词太少）|
| 必读文档 | 写章节号，如「§1 第 3 条枚举值」 | 只写文件名，没有指向具体章节 |
| 特有约束 | 只写这个角色特有的 | 把 AGENTS.md 里的通用约束复制进来（造成重复） |

**本项目数据工程师 Rule（完整示例，直接参考）**：

```yaml
---
description: 以数据工程师角色处理采集、标准化、血缘、双源行情与 report_data_usage
globs: app/**/collect*.py, app/**/source*.py, app/**/hotspot*.py, app/**/tdx*.py, docs/core/04_*.md
alwaysApply: false
---

# 数据工程师角色

你正在以**数据工程师**身份工作。职责：多源采集（通达信/东财/热搜）、标准化与血缘、双源行情、FR-01/FR-02。

- `docs/core/05_API与数据契约.md`（采集接口字段）
- `docs/core/04_数据治理与血缘.md`（quality_flag 枚举、report_data_usage 结构）
- `docs/core/01_需求基线.md` FR-01/FR-02（验收标准）

**约束**：
- 禁止伪造数据；每条证据含 source_name/source_url/fetch_time
- 来源不可用时显式报错或降级，不静默填充假数据

- status 值域：ok|stale_ok|missing|degraded；非 ok 时必填 status_reason
```

## 行动 20：取当前 FR（按顺序表）

从行动 12 产出的「实现顺序表」取当前优先级最高的未完成 FR。

**如果 FR-B 依赖 FR-A 的数据**（FR-A 还未实现）：
- 先实现 FR-A 的「数据模型和接口骨架」（函数体可以是 stub）
- 让 FR-B 能在测试中 mock FR-A 的返回值
- FR-A 完整实现后再集成测试

### Stub 合格标准（与行动 26/行动 27 完成判定无语义冲突）

> **重要**：这里的"stub"不等于 `pass` 或 `raise NotImplementedError`。  
> 行动 27 禁止 `pass/TODO`，而行动 20 允许 stub——两者不矛盾，因为 **stub 有严格定义**：

| 项目 | stub（允许） | placeholder（禁止） |
|------|------------|-------------------|
| 函数体 | 返回符合契约结构的最小真实数据（即使是固定值，也必须符合字段名/类型/枚举） | `pass`、`return None`、`raise NotImplementedError` |
| 降级处理 | 明确抛具体异常（如 `raise DataSourceError("FR-A not ready")`） | 静默失败、空 except |
| 测试可写性 | FR-B 的 mock 能直接 patch 到 stub 函数 | 无法 mock（没有清晰的函数边界） |

**stub 示例**（FR-A 的采集接口骨架，FR-A 尚未完整实现时）：

```python
async def fetch_market_data(stock_code: str, trade_date: str) -> dict:
    """FR-A 骨架 stub：返回符合 04_数据治理与血缘.md §N（表字段定义）
    和 05_API与数据契约.md §N（接口 Schema）结构的最小数据
    
    STUB:FR-A 替换为真实数据源调用（当前为骨架实现，待 FR-A 完成后集成）
    """
    # 返回符合契约的结构（字段名/类型必须正确）
    return {
        "stock_code": stock_code,
        "trade_date": trade_date,
        "close": 0.0,       # 真实值待 FR-A 实现后填入
        "volume": 0,
        "source": "stub",   # 显式标记来源为 stub
        "quality_flag": "missing"  # 符合枚举，不假装 ok
    }
```

> **Stub 标记规范**：示例中使用 `STUB:FR-A` 而非 `TODO`/`FIXME`。区别在于：  
> - `TODO/FIXME` 会被 placeholder 扫描命令扫到，触发完成判定报警  
> - `STUB:` 单独由完成判定里的第二条扫描命令清零：  
> ```powershell
> rg "STUB:" app/ --include="*.py"   # 行动32：此命令结果必须为空才能完成
> ```  
> **STUB 不是"可以留着"的标记**——它是"骨架期专用、后续必须替换"的临时标记，行动 32 要求清零。  
> 用 `rg "STUB:" app/ --include="*.py"` 可随时查看当前残留骨架数量。

**stub 完成判定**：stub 实现完成后，FR-B 的 mock 测试应该能运行通过（即接口骨架可测），但 FR-A 的集成测试（真实数据源）暂不要求通过。

### 外部依赖与 LLM 的测试可测性策略

全量重写时最容易出现 flaky 测试（时好时坏）或根本不写测试的情况，根本原因是没有处理好外部依赖。

**规则 1：外部依赖必须 mock（数据源、第三方 API）**

```python
# ✅ 正确：mock 外部数据源
@pytest.fixture
def mock_weibo_fetch(monkeypatch):
    async def fake_fetch(top_n=20):
        return [{"topic_title": "测试热搜", "rank": 1,
                 "source_name": "weibo", "source_url": "https://example.com",
                 "fetch_time": "2026-03-01T09:00:00"}]
    monkeypatch.setattr("app.services.hotspot.fetch_weibo_hot", fake_fetch)

# ❌ 错误：测试里真实调用外部 API（网络不稳定导致 flaky）
def test_hotspot_real():
    result = asyncio.run(fetch_weibo_hot())  # 真实网络请求，禁止
```

**规则 2：LLM 调用必须用可复现的桩（录制响应或固定桩）**

```python
# ✅ 正确：LLM 桩返回固定的、符合契约的响应
@pytest.fixture
def mock_llm(monkeypatch):
    def fake_generate(prompt, **kwargs):
        return {
            "recommendation": "BUY",
            "confidence": 0.75,
            "reasoning": "测试推理链",
            "citations": [{"source_name": "测试", "source_url": "http://t.com",
                           "fetch_time": "2026-03-01T09:00:00"}]
        }
    monkeypatch.setattr("app.services.llm_router.generate", fake_generate)

# 或用 pytest-recording 录制真实 LLM 响应，回放时不发网络请求
```

**规则 3：测试边界划分**

| 测试类型 | 覆盖什么 | 是否需要 mock |
|---------|---------|-------------|
| 单元测试 | 单个函数的逻辑（枚举校验/字段组装/降级判断） | 必须 mock 外部依赖 |
| 集成测试 | 路由→服务→数据库的完整链路 | mock 外部 API，但不 mock 数据库 |
| 端到端测试 | 全链路（含真实 LLM/真实数据源） | 不 mock，但标记为 `@pytest.mark.e2e`，不在 CI 里自动运行 |

**在行动 21 的产出文件里**，必须明确说明：哪些测试 mock 什么（防止 AI 写了直接调用真实外部依赖的测试）。

  要求：粘贴最后 50 行完整输出（不能省略，不能说"应该都通过"）

步骤 2：逐条核查 FR
  对照 docs/core/01_需求基线.md 的每条 FR：

     - 不是 placeholder（函数体是 pass 或 return None）
     - 不是硬编码（直接返回测试期望的数据）
     - 不是 TODO（含有 TODO/FIXME 注释）
  ③ 写证据：「读了 app/xxx.py 第 N 行，实现是 [描述]」

步骤 3：输出差距清单

## 行动 30：三类顽固问题的处理

**顽固问题 1：AI 修了 A，破坏了 B（最常见）**

原因：AI 修改了被 B 依赖的公共函数。

```
以下测试之前通过，现在失败了：
[粘贴失败测试名和 AssertionError/TypeError]

请按以下步骤修复：
1. 找到导致这个测试失败的具体代码改动（git diff 或对比原函数）
2. 在不改变原有函数签名的前提下修复
3. 如果必须改函数签名，先列出所有调用这个函数的地方，
   确认每处的影响，我确认后再改
```

**顽固问题 2：AI 说「已实现」但代码是 mock**

原因：AI 为让测试通过，在实现里硬编码了测试期望的结果。

```
请逐行检查 [文件路径] 的实现：
1. 是否存在硬编码的返回值（如 return {"success": True, "count": 5}）
2. 是否判断了测试环境（如 if testing: return mock_data）
3. 是否有未实现的 pass、return None、raise NotImplementedError

如果存在，请改为真实实现。测试应该测试真实逻辑，而不是被伪造的返回值通过。
```

**顽固问题 3：后端实现了但页面没显示**

原因：API 返回了字段，但 Jinja2 模板没有读取该字段。

```
FR-xx 的后端接口已实现，API 返回了 [字段名]（可以通过 curl 验证），
但页面 app/web/templates/[模板名].html 没有显示这个字段。

请参考 docs/core/03_详细设计.md 的展示约束与 UI 相关章节，
在模板中加入 [字段名] 的展示：
- 用中文业务语义显示，不直接暴露英文字段名
- 若字段为空，显示「数据暂不可用」而非空白
```
