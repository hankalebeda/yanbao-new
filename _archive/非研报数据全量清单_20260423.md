# 非研报数据全量清单与补齐复查

日期：2026-04-23  
范围：基于 docs/core 下 SSOT 文档与当前代码实现，对系统除研报正文外所需数据进行完整盘点、补齐说明与缺漏复查。  
说明：为避免直接改写 SSOT，本文件作为本次分析与实施落地文档保存在 _archive。

---

## 1. 结论摘要

本次核查后的结论分三层：

1. 需求侧：docs/core 已经明确了系统需要的核心非研报数据，主要分为股票池、行情、热点、市场状态、公司与估值、资金面、新闻政策、模拟仓、结算、用户鉴权、订单支付、调度审计等数据域。
2. 代码侧：原仓库并非没有这些数据能力，而是存在两类断点：
   - 一类是“已有采集器但未接线”，典型是 stock_profile、main_force_flow、dragon_tiger_list、margin_financing。
   - 一类是“读模型已有上游数据，但对外仍返回空对象”，典型是 financial_analysis、industry_competition。
3. 本次补齐后：
   - 研报详情读模型不再把 financial_analysis / industry_competition 清空。
   - 新增公开快照接口 GET /api/v1/stocks/{stock_code}/snapshot，一次性输出完整非研报数据快照。
   - 新增内部补采接口 POST /api/v1/internal/stocks/{stock_code}/non-report-data/collect，可把 stock_profile、主力资金、龙虎榜、融资融券、北向资金、ETF 资金流按合法 data_batch 血缘落库。

复查结论：

- 对外消费层的“非研报关键数据块”已经不再有空壳返回。
- 可持久化补采的数据链已经接通。
- 仍然属于 live-only 的数据块有：company_overview、news_policy、industry_competition。这三类当前可实时获取、可对外返回，但尚无独立持久化表。它们不是“空缺”，而是“实时拉取型数据”。

## 4. 本次发现的核心断点

### 4.1 断点一：读模型把已有数据清空

问题位置：app/services/ssot_read_model.py

表现：

- report detail 对外 payload 中，financial_analysis 直接返回 {}
- report detail 对外 payload 中，industry_competition 直接返回 {}

影响：

- 前端虽然约定了这两个块存在，但读取到的始终是空对象。
- 这类问题不是上游没数据，而是最后一层序列化时被人为抹掉。

本次修复：

- financial_analysis 现在会回填估值、股本、市值、ret5/ret20、tdx_local_features、资金面状态等字段。
- industry_competition 现在会回填行业名、板块编码、同业 peers、peer_count、data_status。

### 4.2 断点二：采集器存在但没有标准接入口

问题位置：

- app/services/capital_usage_collector.py
- app/services/stock_profile_collector.py

表现：

- persist_capital_usage 已能写 main_force_flow / dragon_tiger_list / margin_financing。
- persist_stock_profile 已能写 stock_profile。
- 但仓库里没有统一对外调用入口把它们接入正常链路。

影响：

- 能力存在但数据不会自然补齐。
- 如果直接裸调 collector，还可能传入随机 batch_id，破坏 batch 血缘一致性。

本次修复：

- 新增内部补采接口 POST /api/v1/internal/stocks/{stock_code}/non-report-data/collect。
- 接口内部先创建合法 data_batch，再调用已有 collector，避免 report_data_usage.batch_id 成为孤儿。

### 4.3 断点三：公开层没有一个“完整非研报快照”入口

问题位置：app/api/routes_business.py

表现：

- 原有 /stocks 只有搜索。
- 原有首页与概览接口只返回少量聚合信息。
- 没有一个接口能一次性把公司、估值、行情、资金、热点、新闻、行业对比完整返回。

本次修复：

- 新增 GET /api/v1/stocks/{stock_code}/snapshot。
- 输出统一的非研报数据快照，便于前端、脚本与人工排查直接消费。

## 5. 本次新增/修复的接口

### 5.1 公开接口：个股非研报快照

接口：GET /api/v1/stocks/{stock_code}/snapshot

用途：

- 一次性返回单只股票当前的非研报数据。
- 供前端详情页、调试脚本、补数核查直接使用。

返回块：

- company_overview
- market_snapshot
- market_state
- valuation
- financial_analysis
- industry_competition
- capital_data
- hotspot
- news_policy
- data_sources
- data_status

### 5.2 内部接口：非研报补采

接口：POST /api/v1/internal/stocks/{stock_code}/non-report-data/collect

用途：

- 为单只股票按交易日补齐非研报持久化数据。
- 当前覆盖：stock_profile、main_force_flow、dragon_tiger_list、margin_financing、northbound_summary、etf_flow_summary。

补采后保证：

- report_data_usage 中存在对应 dataset_name 行。
- 每条 usage 的 batch_id 都能在 data_batch 找到对应记录。

## 8. 建议的后续收敛方向

若要继续把“实时可用”收敛到“可批量持久化”，优先顺序建议如下：

1. 为 company_overview 建独立快照表，避免详情页每次实时请求外部接口。
2. 为 news_policy 建快照表，沉淀近 7 日可复查的事件证据。
3. 为 industry_competition 建快照表，支持横截面对比回放与审计。

- 非研报数据的完整说明文档
