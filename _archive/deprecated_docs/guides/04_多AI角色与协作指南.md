# 多 AI 角色与协作指南

> **用途**：定义日常开发时的多 AI 角色用法 + 重大迭代时的三 AI 协同流程；便于在 Cursor/其他环境中按角色切换或组合使用。  
> **读者**：产品、技术、使用 AI 协作的开发。  
> **关联**：系统目标与范围见 `docs/core/07_系统目标与范围整合.md`；API 与契约见 `docs/core/05_API与数据契约.md`；问题追踪见 `docs/planning/05_问题清单_代码与文档及执行结果.md`。

---

## 1. 为什么用多 AI 角色

- **职责清晰**：每个角色只关注一块领域（数据、研报生成、前端、测试等），减少上下文混杂。
- **口径一致**：角色绑定到同一套文档（需求基线、API 契约、验收计划），输出与 SSOT 对齐。
- **可追溯**：谁（角色）在什么范围改了什么，便于与 RACI、门禁对应。
- **并行与接力**：数据工程师先定采集契约 → 研报工程师接研报生成 → 前端接展示与高级区，减少返工。

## 3. 在 Cursor 中如何使用多角色

### 3.1 方式一：用规则（Rules）固定角色上下文

在 `.cursor/rules/` 下为每个角色建一条规则（见本文 §6），在描述里写明「本规则让 AI 扮演 XX 角色」。  
当你打开对应文件（如 `app/services/collector*.py`）或 @ 提到相关文档时，该角色规则会被优先应用，AI 会以该角色视角回答和改代码。

### 3.2 方式二：在对话里显式指定角色

在提问时直接说明角色，例如：

- 「请以**数据工程师**身份：根据 07 契约设计热搜采集的请求/响应字段。」
- 「请以**研报生成工程师**身份：检查当前 prompt 是否覆盖 evidence_items → analysis_steps → inference_summary 的中文展示。」
- 「请以**测试与质量**身份：对照 05 验收计划，列出本迭代必须通过的检查项。」

这样即使没有规则，AI 也会按你指定的角色约束自己（结合本指南的「必读文档」与「典型产出」）。

### 3.3 方式三：角色接力（手递手）

1. **数据工程师** 完成采集与 `report_data_usage` 结构后，把「输出契约」和示例交给 **研报生成工程师**。
2. **研报生成工程师** 完成生成与推理链后，把研报 JSON 样例和高级区字段交给 **前端与体验**。
3. **前端与体验** 完成页面后，把页面清单与交互交给 **测试与质量** 写验收用例。

在 Cursor 中可通过「新开对话 + 粘贴上一角色的结论与约束」或在一个对话里说「接下来请切换为前端角色，基于下面的研报 JSON 做详情页」来模拟接力。

### 3.4 方式四：多 Agent 编排（可选）

若使用多 Agent 框架（如 Cursor Composer 多步骤、或自建 LangGraph/Crew 等）：

- 为每个角色配置 **系统 prompt**：角色名 + 职责 + 必读文档路径 + 输出格式要求。
- 管道示例：**数据工程师** Agent 输出「数据就绪+report_data_usage」→ **研报生成工程师** Agent 消费并输出研报 JSON → **前端** Agent 消费 JSON 生成或检查页面。
- 契约与错误码以 `05_API与数据契约` 为 SSOT，各 Agent 只读不改。

## 4. 所有角色共同遵守的约束

- **文档 SSOT**：目标/范围/底线以 `07_系统目标与范围整合.md` 为准；API/类型/错误码以 `05_API与数据契约.md` 为准；验收以 `01_测试验收计划.md` 为准。不在此三处之外发明新口径。
- **真实性红线**：禁止伪造数据；证据需 source_name/source_url/fetch_time；来源不可用时显式报错或降级并说明。
- **绩效底线（ADR-003）**：**四维度绩效**：真实胜率≥55%/盈亏比≥1.5/年化Alpha≥10%（运行满12月）；方向命中率仅作系统监控；统计须带样本数、覆盖率；口径以整合文档 §3 与 05 为准。
- **高级区与透明化**：研报高级区必须用中文展示「模型用了哪些数据」与「研报生成全过程」（思维链）；缺失/降级须显式说明。

## 5. 重大迭代：三 AI 协同工作流程

用于**首次 AI 编程**或**重大方案迭代**时的全面审查与验收。三个角色：评判官、优化师、验收官。

### 5.1 角色定义

| 角色 | 代号 | 职责 | 产出 |
|------|------|------|------|
| **AI-1 评判官** | 评判官 | 判断除编写代码以外的部分是否全部细化到位且无问题；所需数据是否都获得 | 问题清单（非代码类） |
| **AI-2 优化师** | 优化师 | 重复分析方案、细化缺失、数据全生命周期分析、测试执行、文档修改 | 优化方案、修改后的文档、测试报告 |
| **AI-3 验收官** | 验收官 | 重新学习文档、验收并提出问题 | 验收报告、遗留问题 |

### 5.2 AI-1 评判官（第一轮）

**检查范围**：需求与范围、数据需求、处理流程、产出与存储、测试项覆盖。  
**产出**：问题清单（`docs/planning/问题清单_三AI协同_YYYYMMDD.md`），按 P0→P1→P2 排序。

### 5.3 AI-2 优化师（第二轮及迭代）

**工作内容**：方案分析、细化缺失、数据全生命周期分析、测试设计、测试执行。  
**测试项**：数据源连通性（`tests/test_data_sources.py`）、AI 接口（`ai-api/WEB_AI_UNIFIED_API.md`，`http://127.0.0.1:8010/api/v1/webai`）。  
**注意**：控制并发与单次调用量；Playwright 建议串行、单 provider、短 prompt。  
**产出**：优化后的文档、测试结果、缺失项清单、核心结论汇总。

### 5.4 AI-3 验收官（第三轮）

**工作内容**：重新学习文档、按 `01_测试验收计划` 与 `07_系统目标与范围整合` 验收、提出遗留问题。  
**产出**：验收报告、遗留问题清单。

### 5.5 协作顺序与迭代

```
AI-1 评判官 → AI-2 优化师（首轮）→ 多 AI 讨论 → AI-2 优化师（次轮：逐文档修改）→ AI-3 验收官
若遗留问题非空 → 回到 AI-2；若为空 → 结束，输出核心结论
```

### 5.6 首次 AI 编程补充

- 环境：Python/虚拟环境、依赖、.env 配置
- 前置：TDX、Ollama、ai-api 手动登录状态
- 文档 SSOT 引用一致
- 问题清单维护：每轮问题进入主问题清单或专项清单

## 7. 附录：五阶段快速检查清单

> 首次 AI 编程或重大迭代前，按顺序执行以下阶段。

| 阶段 | 动作 | 产出 |
|------|------|------|
| **0** | 环境与文档准备 | 目标/SSOT/数据源确认；TDX、Ollama、ai-api 就绪 |
| **1** | AI-1 评判官 | 非代码到位性问题清单（P0→P1→P2） |
| **2** | AI-2 优化师 | 优化方案、测试报告、缺失清单 |
| **3** | 多 AI 讨论 | 共识问题清单、优先级与责任 |
| **4** | 逐文档修改 | 修改后的文档、问题清单已处理标记 |
| **5** | AI-3 验收官 | 验收报告、遗留问题；若无遗留→输出核心结论 |

**迭代**：遗留问题非空 → 回阶段 2/4；遗留问题为空 → 结束。

## 9. 小结

- **日常角色** = 数据工程师、研报生成工程师、预测与回灌、前端与体验、测试与质量、运维与可观测、产品与范围。
- **重大迭代** = AI-1 评判官 → AI-2 优化师 → AI-3 验收官（见 §5）。
- **用法**：Cursor 规则 + 对话显式指定 + 角色接力；可选多 Agent 编排。
- **约束**：SSOT、真实性红线、四维度绩效底线、高级区透明化。
