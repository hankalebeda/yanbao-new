### 0.1 代理环境说明

本机若使用 Clash/V2Ray 等系统代理，访问**北向资金、东方财富**等国内站点可能被拦截导致 `ConnectionResetError`。代码已内置 `bypass_proxy()`（`app/core/proxy_utils.py`），北向、东财请求前会自动关闭代理。若仍失败，可临时关闭系统代理再试。

### 0.2 ai-api 独立启动（可选）

- **路径**：项目根下 `ai-api/`（chatgpt_web、deepseek_web、gemini_web、qwen_web、webai）
- **接口**：`ai-api/WEB_AI_UNIFIED_API.md`
- **依赖**：Playwright、手动登录对应网站；未启动时研报降级到本地 Ollama

**启动顺序与步骤**：
1. 安装 ai-api 依赖：`pip install -r ai-api/requirements.txt`（若有）或复用主项目依赖
2. 启动端口：默认 `ai-api` 各服务端口见各 `router` 或 `main`；统一入口 `webai` 通常为 8001～8010 区间，避免与主应用 8010 冲突（可配置 `AI_API_PORT`）
3. Cookie 维护：Web 自动化需浏览器登录，Cookie 存储在 `ai-api/.cookies/` 或项目 `runtime/cookies/`；定期运行 `scripts/check_web_cookies.py` 检测失效
4. 健康检查：`GET /api/v1/webai/providers` 返回可用 provider 列表；主应用通过 `AI_API_BASE_URL` 调用
5. 失败排查：若主应用报「Web AI 不可用」，依次检查 ① ai-api 进程是否运行 ② 端口是否被占用 ③ Cookie 是否过期（执行 `check_web_cookies.py`）④ 内存是否充足（Web AI 并发时可能 OOM，建议 `--with-webai` 测试单独执行）

## 2. 环境基线（Windows 单机）
1. Windows 10/11 + NVIDIA 驱动可用。
2. 独立 Python 虚拟环境（规避 Anaconda 污染）。
3. Ollama 服务可启动：`ollama serve`。
4. 推荐目录：
   - 模型目录：`C:\Users\Public\ollama_models`（非门禁，以服务可用性优先）
   - 日志目录：`runtime/logs`
   - 缓存目录：`runtime/cache`

## 3. 部署步骤（合并原 18 与 26 Phase 2）
1. 准备环境变量模板：复制 `.env.example -> .env`（必填/可选/默认分层）。
   - 生产必须设置：`INTERNAL_API_KEY`（内部接口鉴权）；若启用业务 JWT/RBAC，必须设置 `JWT_SECRET`（见安全合规基线）。
2. 安装依赖并启动应用（本地/生产方式固定）。
3. 拉取模型并烟测：`ollama pull qwen3:8b`（生产标准）。
4. 部署健康检查：服务启动、核心 API、模型健康、日志落盘。
5. 回滚资产准备：模型版本、提示词版本、检索版本、应用版本。

### 3.0 API 与客户端约定
- 所有业务与内部 API 响应为 **UTF-8** 编码（FastAPI 默认 `application/json; charset=utf-8`）。脚本或终端调用时请按 UTF-8 解码，否则中文等字符可能乱码（如 PowerShell 需设置 `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8` 或使用 `Invoke-RestMethod` 等正确解析 JSON 的方式）。

## 4. 日级运行步骤（单股单日一次）
1. 热搜主链采集。
2. 热搜备链兜底。
3. 特征生成与映射。
4. Ollama 推理生成。
5. 按交易日幂等发布与审计（同股同日复用同一报告）。

## 5. 运行参数与阈值
1. 并发默认 1，上限 2。
2. 超时 60 秒；重试 2 次（1s/2s）。
3. `llm_timeout_rate > 5%` 触发降级。
4. `llm_oom_count`（日）> 3 触发降并发。
5. 连续失败阈值触发主备切换。

## 6. 失败处理与恢复
1. Cookie 失效 -> 自动刷新 -> 失败告警。
2. 热搜双链失败 -> 降级去热搜因子。
3. Ollama 无响应 -> 重启服务 -> 回归检查。
4. OOM/超时异常 -> 降并发 + 降级。
5. 恢复条件：连续 3 个观测窗口指标正常。

### 6.2 数据源全不可用预案

当东财/通达信/热搜全部不可用时：

1. **系统行为**：研报生成会降级或失败；熔断器标记各源 `circuit_open`
2. **用户提示**：研报详情页高级区显式说明「数据源暂时不可用，报告可能不完整」
3. **运维动作**：检查网络、API 可用性；等待 cooldown 后 half-open 探测；必要时人工切换备源或联系数据方
4. **恢复条件**：连续 `SOURCE_RECOVER_SUCCESS_THRESHOLD` 次成功探测后熔断关闭
5. **记录**：全源失败事件写入运维日志，用于事后分析

### 6.3 回撤熔断（drawdown_state=HALT）用户侧说明

当 `sim_account.drawdown_state=HALT` 时：

1. **研报行为**：BUY 研报仍发布，但不写入 `sim_position`；实操指令卡不生成或标注「回撤保护中，暂不提供新信号」
2. **看板提示**：模拟收益看板显式展示「系统处于回撤保护状态，新开仓已暂停」
3. **解除**：回撤恢复到 -12% 以上，或满 `drawdown_halt_review_days`（默认 3 交易日）后自动解除
4. **文案 SSOT**：见 `docs/core/18_免责声明与风险提示文案.md` 或模拟实盘追踪设计 §10.1

### 6.4 告警处置速查（与 SLO 对齐）

> 指标阈值以 `docs/core/06_SLO_SLA与可观测性.md` 为准；此处给出最小处置动作，避免临场决策分歧。

| 告警 | 典型表现 | 最小处置动作 |
|---|---|---|
| `LLMTimeoutHigh` | `llm_timeout_rate` 超阈 | 并发降为 1 → 检查/重启 Ollama → 必要时切换测试模型回归 |
| `LLMOOMHigh` | OOM 频繁/推理卡死 | 并发降为 1 → 释放显存/重启 Ollama → 观察 1 小时 |
| `ReportDegradedHigh` | `report.degraded_rate` 长期偏高 | 查 `missing_fields/recover_actions` → 修复数据源/门禁规则 → 禁止静默长期 degraded |
| `AuthLeak` | 越权漏拦/审计缺失 | 立即封控相关端点 → 回溯审计 → 修复后走灰度发布 |

> 告警渠道（邮件/企微/钉钉）与值班安排由运维策略决定；建议至少配置 S0 立即通知、S1 15 分钟内响应。

### 6.5 告警 Webhook 接入 SOP（企微/钉钉）

**配置项**（`app/core/config.py` 与 `.env`）：
- `ALERT_WEBHOOK_URL`：机器人 Webhook URL
- `ALERT_WEBHOOK_ENABLED=true`：启用推送
- `ADMIN_ALERT_WEBHOOK_URL`：管理员专用（可与上述不同）

**鉴权**：企微/钉钉 Webhook URL 已包含密钥（key/access_token），不得泄露；建议使用环境变量，不写入代码库；生产环境仅授予运维/值班角色配置权限。

**失败重试**：推送失败时建议指数退避重试（如 1s/2s/4s 共 3 次）；失败日志写入 `runtime/logs/alert_send_failed_<date>.log`；连续 5 次失败可降级为邮件或人工通知。

**告警分级**（与 SLO 对齐，建议映射）：
- **S0 立即**：`AuthLeak`、`LLMOOMHigh`（推理卡死）、数据源全不可用
- **S1 15 分钟内**：`LLMTimeoutHigh`、`ReportDegradedHigh`、`drawdown_state=HALT`
- **S2 当日**：`CookieInvalid`、单次结算失败、单股生成超时
- 分级字段建议在推送消息中显式标注（如 `[S0]`），便于值班按优先级处理。

**企微机器人**：
1. 群聊 → 群设置 → 群机器人 → 添加机器人 → 复制 Webhook URL
2. 写入 `.env`：`ALERT_WEBHOOK_URL=https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx`
3. 消息格式（JSON）：`{"msgtype":"text","text":{"content":"告警内容"}}` 或 `{"msgtype":"markdown","markdown":{"content":"**标题**\n> 详情"}}`

**钉钉机器人**：
1. 群设置 → 智能群助手 → 添加机器人 → 自定义 → 复制 Webhook
2. 写入 `.env`：`ALERT_WEBHOOK_URL=https://oapi.dingtalk.com/robot/send?access_token=xxx`
3. 需配置加签或 IP 白名单（机器人安全设置）
4. 消息格式：`{"msgtype":"text","text":{"content":"告警内容"}}`

**测试**：`curl -X POST "$ALERT_WEBHOOK_URL" -H "Content-Type: application/json" -d '{"msgtype":"text","text":{"content":"告警测试"}}'`，收到消息即配置成功。

## 7. 回退动作
1. 回退模型版本。
2. 回退提示词版本。
3. 回退检索版本。
4. 回退应用版本。

### 7.1 数据与配置备份（最低要求）

1. **数据库（SQLite）**：发布前备份 `data/app.db` 到 `data/app.db.bak.<timestamp>`。
2. **配置**：保留当次发布 `.env` 的脱敏快照（仅保留键名与是否配置，不包含密钥明文）。
3. **运行日志**：至少保留 7 天可检索日志；审计事件按安全合规要求保留更久（由策略决定）。

### 7.2 数据库备份与恢复 SOP

**备份频率**：每日自动（推荐 cron/计划任务）；发布前手动增量备份。

**备份命令示例**：
```powershell
# 每日备份（保留最近 7 天）
Copy-Item data/app.db "data/backups/app.db.bak.$(Get-Date -Format 'yyyyMMdd_HHmmss')"
# 或：robocopy data data/backups app.db /COPY:DAT
```

**恢复步骤**：
1. 停止 FastAPI 服务（避免写入冲突）
2. 将备份文件复制回 `data/app.db`（覆盖前再次备份当前损坏库）
3. 启动服务，执行健康检查 §3.1
4. 验证 `GET /api/v1/reports`、`GET /api/v1/sim/positions` 可读
5. 记录恢复时间、原因、操作人于运维日志

**数据保留策略**（建议）：研报、预测、sim_position/sim_account 按业务需求保留；冷备份至少 30 天；归档策略可后续扩展。

**备份恢复演练**（必须执行）：
- **频率**：每季度至少 1 次；发布前可选项
- **RTO 目标**：数据库恢复 ≤ 30 分钟（从备份文件可用到服务健康检查通过）
- **演练步骤**：① 选一备份文件；② 停止服务；③ 覆盖 `data/app.db`；④ 启动服务；⑤ 执行 §3.1 健康检查；⑥ 验证 `GET /api/v1/reports`、`GET /api/v1/sim/positions` 可读
- **记录**：演练时间、备份文件、恢复耗时、结果写入 `runtime/logs/backup_drill_<date>.txt`

### 7.3 回滚步骤 SOP

**回滚触发**：连续失败、重大缺陷、指标持续恶化（如 `degraded_rate` 超阈）。

**回滚顺序**（按依赖）：
1. **应用代码**：`git checkout <commit>` 回退到发布前版本；或从发布包还原
2. **配置**：从脱敏快照恢复 `.env`（密钥需重新填入）
3. **数据库**：仅当数据结构变更导致不兼容时，从 `data/app.db.bak.<timestamp>` 恢复；**注意**：回滚会丢失回滚点之后的数据
4. **模型**：若 Ollama 模型版本变更，`ollama pull qwen3:8b` 恢复指定版本（Ollama 不自动管理多版本，需人工记录）

**回滚后验证**：
- `pytest tests/test_api.py tests/test_trade_calendar.py -v` 通过
- `GET /health`、`GET /api/v1/internal/metrics/summary` 正常
- 生成 1 份研报验证主链路

**记录**：回滚时间、原因、回退 commit、验证结果写入 `runtime/logs/rollback_<date>.txt`。

### 7.4 数据库迁移 SOP（表结构变更时）

1. 设计变更：在 `app/models.py` 修改表定义，在 07 契约中更新字段
2. 迁移脚本：使用 Alembic 或手写 `ALTER TABLE`（SQLite 支持有限，复杂变更需建新表+数据迁移）
3. 执行前：备份 `data/app.db` 至 `data/backups/app.db.bak.<timestamp>`
4. 执行迁移：`alembic upgrade head` 或手工执行 SQL
5. **迁移失败回滚**：① 停止服务；② 将备份复制回 `data/app.db`（覆盖）；③ 检查迁移脚本错误并修复；④ 修复后重新执行 3～4；⑤ 记录于 `runtime/logs/migration_rollback_<date>.txt`
5. 验证：检查新表/新字段存在；跑回归测试
6. 回滚：从备份恢复；若有数据迁移则需逆向脚本（建议迁移前设计可逆方案）

## 9. Cookie/会话维护规程（人工操作 SOP）

> Cookie 是 Web AI 接口（ChatGPT/Gemini/DeepSeek/Qwen Web）能否正常工作的关键依赖，**运维负责人**须定期维护。

### 9.1 检测 Cookie 是否有效

**FR-02「15 分钟」起止定义**（验收口径，见 `01_测试验收计划.md` §2.9a）：
- **起**：满足失效判定条件的那次探测时间点 T0（若需连续 N 次失败才判定失效，则以第 N 次失败的时间为 T0；单次判定则以首次失败为准）。
- **止**：下一次健康探测通过，或研报/热搜调用成功验证 Cookie 生效的时间点 T1。
- **验收**：T1 − T0 ≤ 15 分钟；若为人工重登，则从告警产生到人工完成重登并验证通过 ≤ 15 分钟。

1. 观察日志：若出现 `[WEB] cookie_expired` 或 `[WEB] auth_failed` 告警，立即进入人工重登流程
2. 主动检测：每周一运行健康检查脚本：`python scripts/check_web_cookies.py`
3. Cookie 有效期参考：ChatGPT ≈ 7天，Gemini ≈ 7天，DeepSeek Web ≈ 30天，Qwen Web ≈ 30天

### 9.2 人工重登步骤（以 ChatGPT 为例）

```
1. 使用 Playwright 手动打开浏览器：
   playwright open https://chatgpt.com
2. 登录账号（账号信息存于运维负责人处，不得写入代码库）
3. 登录成功后，在浏览器 DevTools → Application → Cookies 中复制全部 Cookie
4. 将 Cookie JSON 保存到 .env 中对应的变量（如 CHATGPT_COOKIES）
5. 重启 FastAPI 服务以刷新 Cookie 缓存
6. 运行健康检查验证：python scripts/check_web_cookies.py
```

### 9.3 Cookie 刷新优先级（多平台同时失效时）

| 优先级 | 平台 | 说明 |
|--------|------|------|
| P0 | DeepSeek Web | LLM路由主链降级后果最严重 |
| P1 | Qwen Web | 三方审计辅助链 |
| P2 | ChatGPT Web | 零成本备份，失效影响有限 |
| P2 | Gemini Web | 同上 |

> API 版本（DeepSeek API、Gemini API）无 Cookie 依赖，不在此流程内

### 9.4 预防性刷新策略
- 建议每6天主动刷新一次 ChatGPT/Gemini Cookie（提前于7天失效期）
- 若当日研报生成成功率 < 90%，立即检查 Cookie 状态
- Cookie 存储位置：`.env` 文件的 `*_COOKIES` 变量（仅本地，不得提交 git）

## 10. 股票池（全自动选股）

### 10.1 选股方式说明
股票池从 A 股全市场中**自动筛选**，无需人工维护。

- **Tier-1（核心池）**：≤50只，每日全量生成研报。以沪深300成分股为基础，**自动筛选**流动性好、事件活跃的股票
- **Tier-2（轮转池）**：≤250只，每日轮换1/5（每5个交易日覆盖一次）。以中证500成分股为基础，**自动选股**
- **Tier-3（规则触发池）**：按规则条件自动初筛，命中升入 Tier-2 队列

### 10.2 自动选股规则（参考）

**进入条件**（满足任意2条）：
- 日均成交额 > 5亿元（最近20日平均）
- 关联重大事件频率 > 1次/月（公告、媒体热点）
- 机构持仓比例 > 10%
- 行业代表性（各行业至少保留3只）

**退出条件**（满足任意1条）：
- 被实施 ST / *ST
- 连续10个交易日日均成交额 < 2亿元（流动性不足）
- 基本面重大负面事件（退市风险警示）

### 10.3 配置与展示
- **数据源**：自动选股脚本输出或通过 `STOCK_POOL` 环境变量预配置（`config.stock_pool`）
- **展示**：后台 `/admin` 股票池模块为**只读展示**，供运维查看当前池状态
- **变更记录**：自动选股脚本可将变更写入 `runtime/logs/stock_pool_changes_<date>.txt`
