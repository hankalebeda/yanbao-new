# 测试与验收计划

> **与整合文档关系**：目标、范围、底线、完整站点、数据与推理透明化见 `docs/core/07_系统目标与范围整合.md`；本文件为验收与门禁细则，商业化KPI以整合文档 §3 为准（**胜率≥55%、盈亏比≥1.5、年化Alpha≥10%**；旧版预测准确率口径已废弃，见 ADR-003）。

> **与契约关系**：接口/字段/错误码以 `docs/core/05_API与数据契约.md` 为准；本文件关注“怎么测、怎么判定是否通过”。

> **与实现状态**：验收前可参考 `docs/planning/07_生产就绪差距与执行计划.md` §2「当前系统实现状态」核对接口与字段是否与实现一致。

## 0. 验收前置条件（必须满足）

1. 已配置内部鉴权（生产必须）：`INTERNAL_API_KEY`，并能以 `X-Internal-Key` 访问 `/api/v1/internal/*`。
2. 服务端能回显 `request_id`（建议客户端传 `X-Request-ID`），用于把“接口现象 → 日志/审计 → 数据源/模型”串起来。
3. 已准备最小股票池（`STOCK_POOL` 环境变量或自动选股脚本输出）与可运行的 Ollama 模型（`qwen3:8b`，当前生产标准；见 ADR-002 模型选型）。
4. 已配置 AI 接入：ai-api 四个 Web API 或 Codex/Gemini CLI（本部署**不使用** API Key，见 `docs/core/08_AI接入策略.md`），用于验收三方投票审计机制。
5. 本地 TDX 数据目录（`config.tdx_data_dir`）已就位，或已确认降级到规则兜底（固定8%止损）时验收通过。

## 0.1 验收接口清单（对应实现）

- 研报：`POST /api/v1/reports/generate`（请求体见 07 §2.0）、`GET /api/v1/reports/{report_id}`、`GET /api/v1/reports`
- 看板与预测：`GET /api/v1/predictions/stats`；`POST /api/v1/predictions/extract`、`POST /api/v1/predictions/settle`（契约见 07 §2.6、§2.7）
- 内部能力：`/api/v1/internal/hotspot/*`、`/api/v1/internal/llm/*`、`/api/v1/internal/metrics/summary`

## 1. 文档与理论先行门禁（阶段 A）
1. 证据链字段完整：`data_sources/analysis_steps/inference_summary/validation_plan`
2. 事实回验机制完整：预测可在 1/7/14/30/60 日窗口验证
3. 结论必须可追溯到来源与分析步骤
4. 研报必须解释 `MA5/MA20` 含义且展示对应数值

## 2. 功能验收
1. 微博/抖音每小时 Top50 入库成功
2. 每只股票每个交易日仅生成 1 份正式报告；当日重复触发返回同一 `report_id`
3. 热搜映射股票并进入 `social_trend`
4. 买入/卖出/观望 含证据、触发条件、失效条件
5. 抽检 10 条 `citations`，`source_url` 必须可访问（HTTP 200/301/302）
6. 禁止出现“示例热点/伪协议 URL”进入正式研报
7. 抽检 10 条 `social_trend.top_topics`，`relevance_score >= 0.6`
8. 抽检热搜证据时间，主结论证据必须在 72h 内
9. 研报输出 `price_forecast`，且窗口固定 1/7/14/30/60
10. 页面必须包含：公司概况、行业与竞争格局、财务分析、估值、投资要点与风险
11. 页面必须新增“股票基础信息”模块，并展示公司概况/财务摘要/估值摘要
12. 页面必须新增“历史回测面板”，同时展示全历史与近3个月窗口回测
13. 页面必须新增“模型分析链路”，展示 `data_sources/analysis_steps/inference_summary/validation_plan`
14. `reasoning_trace.raw_inputs` 必须包含原始输入对象，供云端二次预测
15. `price_forecast.explain` 必须存在，且包含至少 5 个核心因子贡献
16. 页面需展示“预测为什么成立”的中文论证，不得只给单一数值
17. `price_forecast.backtest` 必须存在，包含 1/7/14/30/60 的准确率与样本数
18. `price_forecast.backtest.horizons_recent_3m` 必须存在，且覆盖 1/7/14/30/60
19. `price_forecast.backtest.summary_recent_3m` 必须包含 `start_date/end_date/overall_accuracy/samples`
20. `price_forecast.explain.historical_position` 必须包含 60日/252日/3年分位与3年区间高低点
21. `price_forecast.readiness.ready_for_use=true` 才可标记“可用版本”
22. 页面“指标解释”和“预测论证”主文案不得出现英文因子名（允许括号缩写，如 MA5）；专业名词旁须有小字解释（如 MA20、ATR、量能比等），保持新手可读
23. **模拟交易胜率 ≥ 55%**（样本 ≥ 30 笔后统计），须同时展示样本数、覆盖率与口径；A/B/C 三类分别统计；方向命中率仅作监控
24. 每个预测窗口必须输出 `confidence_raw`、`confidence_empirical_accuracy`、`confidence_gap`
25. `confidence` 必须为校准后值，不允许与 `confidence_raw` 永远相等
26. 必须输出 `direction_forecast`，并覆盖 `1~7` 天逐日方向与动作建议
27. `direction_forecast.backtest_recent_3m` 必须包含 `actionable_accuracy/actionable_samples/actionable_coverage`
28. 准确率须同时展示 `actionable_samples` 与 `actionable_coverage`（强信号子集口径），不得只展示单一数值
29. 必须输出 `novice_guide`，且包含一句话结论、三条原因、风险等级、后续观察点
30. 页面首屏必须先展示新手结论区，技术细节可折叠，不得反客为主
31. 若7天信号样本<5或覆盖率<5%，页面必须出现“稳定性提醒”，且最终动作不能直接给出强卖出
32. **高级区**须用中文展示「模型用了哪些数据」与「研报生成全过程」（思维链）；数据缺失/降级须显式说明（见整合文档 §5）
33. **风险提示与免责**：研报页或全站页脚须有显式风险提示与免责声明；文案 SSOT：`docs/core/18_免责声明与风险提示文案.md`
34. **完整站点**：首页或导航可进入列表与看板；列表页数据来源为股票池（全自动选股；`STOCK_POOL` 或自动选股脚本输出），支持按代码/名称/日期/结论筛选；后端必须提供 `GET /api/v1/reports` 支撑筛选与分页（含 `stock_name` 参数，07 契约注明当前仅 SQLite 支持按名称筛选）；列表每行「详情入口」须指向该行报告（如 `/reports/{report_id}`），不得仅按股票代码打开最新日报告；统计看板展示历史准确率汇总与趋势（以 `GET /api/v1/predictions/stats` 或等价数据一致）
35. **报告只读**：已发布报告仅允许读接口返回，禁止事后篡改内容；发布/回滚等关键操作须留审计轨迹

### 2.1 关键用例落地方式（建议自动化）

为避免“人工抽检口径漂移”，以下建议以脚本/测试固化：

1. **幂等用例（条款 2）**：
   - 对同一 `stock_code` 在同一交易日重复调用 `POST /api/v1/reports/generate`（不传 `idempotency_key` 或按约定传入）。
   - 判定：两次返回的 `report_id` 必须一致；否则视为失败（`IdempotencyConflict` 或实现缺陷）。
2. **列表筛选用例（条款 34）**：
   - 调用 `GET /api/v1/reports?stock_code=...&stock_name=...&date_from=...&date_to=...&conclusion=...&strategy_type=...&position_status=...&page=1&page_size=20`（strategy_type/position_status 见 05 §2.2）
   - 判定：返回 `items/page/page_size/total`；`items[*].stock_code`、`items[*].stock_name` 等字段满足筛选条件；传入 `strategy_type=A` 时仅返回 strategy_type 为 A 的研报；分页不重复不丢失（按名称筛选当前仅 SQLite 生效）。
3. **证据可访问性（条款 5）**：
   - 抽样 10 条 `citations[*].source_url`，检查 HTTP 状态码为 200/301/302。
   - 判定：若来源不可用必须显式降级/报错，禁止静默填充伪 URL（条款 6）。
4. **高级区透明化（条款 32）**：
   - 判定：页面或返回体中必须有“所用数据清单（来源/范围/规模/状态）”与“生成全过程（中文、步骤化）”两部分；缺失/降级必须有显式说明字段或文案。
5. **安全与审计（条款 35 + 门禁 5）**：
   - 触发一次越权访问（缺 token / 错 token / 权限不足），判定必须返回 `AuthError`（code=4001），并在审计日志中落 `auth.denied` 事件（内部接口鉴权见 `05_API与数据契约` §0.3）。
6. **report_data_usage 北向/ETF 回归（P1-5）**：
   - 北向已集成时：抽检研报 `dimensions.capital_flow.northbound` 含 `status` 字段。
   - **test_report_data_usage_northbound_status** 断言（可自动化）：① 研报响应 `dimensions.capital_flow.northbound` 存在；② `northbound.status` 存在且 ∈ {ok, stale_ok, proxy_ok, realtime_only, missing, degraded, invalid, fallback}（见 `04_数据治理与血缘.md` §2.1）；③ 当 `status="missing"` 时须有 `reason` 字段。契约见 `05_API与数据契约.md` §2.3 capital_flow.northbound。
7. **血缘登记验收（FR-01）**：抽检 N 条采集数据，每条须有 `source`；若使用了 fallback，须有 `fallback_path`；登记完整性由 `report_data_usage` 或等价血缘表验证。
8. **证据绑定验收（FR-03 证据绑定率≥95%）**：
   - **绑定定义**：`evidence_items` 中每条均能在 `citations` 中找到同源 `source_url`（URL 一致或规范化后一致）。
   - **验收方式**：抽检 N 份研报（建议 N≥20），计算每份「绑定条数/总 evidence 条数」，整体加权平均 ≥ 95%。
   - 测试用例建议：`test_evidence_binding_rate`，抽样研报 `content_json.evidence_items` 与 `citations` 逐条匹配。

## 7. 门禁指标（全量）

### 7.1 系统可靠性门禁
1. 日更任务成功率 >= 97%
2. 单股单日重复请求复用率 = 100%
3. 研报覆盖率 >= 95%（Tier-1 必须；Tier-2 当日批次 ≥ 90%）
4. 模拟结算覆盖率 >= 92%（每日 OPEN 持仓中被正确结算的比例）
5. 越权拦截率 = 100%（付费 API 对免费 token 必须返回 403）
6. `llm_timeout_rate` <= 5%
7. `llm_oom_count`（日）<= 3

### 7.2 模拟交易商业化门禁（样本≥30笔后生效）
8. **模拟胜率 ≥ 55%**（所有策略类型综合；A/B/C 各类型分别统计展示）
9. **盈亏比 ≥ 1.5**（平均盈利额 / 平均亏损额）
10. **年化 Alpha ≥ 10%**（相对沪深300）
11. **最大回撤 ≤ 20%**（触发 HALT 前的阈值，自动控制）
12. 对照组表现：系统信号胜率须同时优于随机基线、MA金叉策略和沪深300

### 7.3 信号分类验收
13. 每份 BUY 研报须包含 `strategy_type`（A/B/C）和 `strategy_type_reason`（不得为 null）
14. `strategy_type` 分布不得过度集中：任一类型占比不得超过 70%（**监控周期口径**：最近 5 个交易日内的 BUY 研报；**「最近 5 个交易日」计算**：以当前交易日为 T，取 T、T-1、…、T-4 共 5 日（由 trade_calendar 确定，含非交易日回退）；统计：`Report` 表 `recommendation=BUY` 且 `trade_date` ∈ 该 5 日，按 `content_json.strategy_type` 分组，`count(type)/total` ≤ 0.7；验收：抽检或脚本计算；样本不足时本门禁不生效）
15. A 类信号事件证据 `event_weight >= 0.6`（至少一条高相关性事件触发）

### 7.4 三方投票审计验收（新）
16. BUY 信号触发三方投票审计时，Gemini API 审计结果须在 15 秒内返回（超时降级为主分析方向）
17. 本地 Qwen3 审计响应时间 ≤ 120 秒（`/no_think` 模式）
18. 审计分歧（主+Gemini vs 本地Qwen3 投票2:1）必须在研报高级区中显式标注"存在审计异议"
19. 三方投票完全一致买入时，`audit_flag="unanimous_buy"`；完全一致卖出时 `audit_flag="unanimous_sell"`（字符串类型，非布尔值）；此时研报置信度可提升 ≤ 0.05（见 `min(base_confidence + 0.05, 0.95)` 上限保护）

### 7.5 UI 完整性验收
20. 首页「今日强信号列表」在有/无 BUY 信号时均正常展示（空状态设计）
21. 研报详情页三层渐进式结构全部可展开/折叠
22. 实操指令卡对免费用户正确显示锁定态，对付费用户正确解锁
23. 模拟收益看板净值曲线图渲染正常（样本<30时显示冷启动进度状态）

### 7.5a 冷启动期展示验收（补充）

| 验收项 | 判定标准 | 细则 |
|--------|----------|------|
| 样本不足提示 | 当某类型（A/B/C）`total < 30` 时，该类型胜率/盈亏比显示「样本不足」或 null | 页面展示「样本积累中：已有 N 笔」；N 为当前已结算笔数；`GET /api/v1/sim/account/summary` 返回 `cold_start: true`、`cold_start_message`、`est_days_to_30` |
| 约需交易日估算 | 可展示「约需 M 个交易日达 30 笔」 | M = ceil((30 - N) / daily_avg)；daily_avg 可用配置项（如 `cold_start_daily_signal_avg=1.5`）或近 7 日实际日均 BUY 强信号笔数；N≥30 时不展示；自动化断言：`summary.cold_start==true` 时 `cold_start_message` 含 N 与 M |
| Walk-Forward 回测 | 若有 `sim_position_backtest` 数据，可展示「历史回测（YYYY-YYYY）仅供参考」 | **分离判定**：回测数据来自 `sim_position_backtest` 表，实盘来自 `sim_position`；看板必须分 Tab 或分区块展示，不得合并统计胜率/盈亏比；回测区块须标注「非实盘，仅供参考」 |
| 不得承诺收益 | 冷启动期不得出现「预计收益」「保证胜率」等承诺性表述 | 见 01 需求基线 FR-04 冷启动用户预期管理 |

24. 移动端（375px）首页、列表页、研报详情页布局正常，无横向溢出

**门禁 24 验收方式**：Chrome DevTools 设备模式设 375×667（或 Playwright `viewport={"width":375,"height":667}`）；逐页检查首页、列表、研报详情，判定无横向滚动条、无内容超出视口；可选脚本 `tests/test_responsive_375.py` 使用 Playwright 断言 `page.evaluate("document.documentElement.scrollWidth <= 375")` 或检查 `overflow-x`。

### 7.6 性能门禁
25. 研报详情页首屏 TTFB（Time To First Byte）≤ 800ms（P95，本地开发环境）
26. `GET /api/v1/reports` 列表接口 P95 响应时间 ≤ 500ms（无缓存情况下）
27. `GET /api/v1/sim/account/summary` P95 ≤ 300ms

**门禁 25～27 自动化/判定方式**：
- **本地脚本**：`scripts/benchmark_api.py`（可新增）调用 `GET /api/v1/reports?page=1&page_size=20`、`GET /api/v1/sim/account/summary` 各 20 次，计算 P95 延迟，断言 ≤ 阈值；研报详情页 TTFB 可用 `curl -w "%{time_starttransfer}"` 或 requests 测首字节时间。
- **判定标准**：本地开发环境（无网络抖动、无并发负载）下 P95 满足即通过；生产环境可放宽 1.2× 或按 SLO 另行约定。
- **回归入口**：`pytest tests/test_performance.py -v`（可选，非门禁阻断）；人工压测时记录结果于 07 §2.4 或 `runtime/logs/benchmark_<date>.json`。
- **门禁 25 研报详情页 TTFB**：`scripts/benchmark_api.py` 不覆盖页面；可用 `curl -w "%{time_starttransfer}\n" -o /dev/null -s "http://localhost:8010/demo/report/600519.SH"` 测首字节时间，或 Playwright `page.goto()` 后取 `performance.timing.responseStart`；人工验收时记录 P95。

## 9. 验收流程（建议执行顺序）

```
Phase A（文档门禁）→ Phase B（单元功能验收）→ Phase C（集成链路验收）→ Phase D（商业化验收）→ Phase E（性能/压力）
```

- Phase A：所有 §1 文档门禁通过（人工抽检 + 自动化）
- Phase B：§2/§3/§4/§5/§6 功能验收通过（逐条测试）
- Phase C：研报生成→模拟开仓→每日结算→看板更新 完整链路跑通
- Phase D：付费/免费权益分割正确（§7.5 UI 验收）；越权拦截率=100%
- Phase E：§7.6 性能门禁通过

> 冷启动期（样本<30）时，§7.2 商业化门禁改用 Walk-Forward 回测结果替代；
> 系统上线60天或满30笔结算交易后，必须重新验证商业化门禁。
