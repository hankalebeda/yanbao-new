# 完整前端页面生成提示词

> **用途**：一次生成多套前端方案 → 你作为评委逐套打分 → 选出最高分版本 → 只向用户展示并交付最优版本。  
> **目标输出**：用户最终只收到**一套**完整、美观、与后端 API 匹配的前端页面。

---

## 一、完整提示词（可直接复制）

```
【任务】基于项目现有前端，生成完整的 A 股个股研报平台前端页面。你必须按以下四步执行，最终只向用户交付最优版本。

【执行流程（必须严格按顺序）】

第一步：生成多套方案
- 共生成 3 套完整前端（每套均包含下方「必须包含的页面」全部 11 个页面）
- 3 套须在设计风格上明显差异化，例如：
  - 方案 A：偏商务/专业风格（深色导航、高对比、数据感强）
  - 方案 B：偏轻量/现代风格（浅色、圆角、留白多）
  - 方案 C：偏金融/稳健风格（蓝色主调、卡片式、稳重布局）
- 每套的结构、字段、API 对接须与文档一致，仅在视觉风格、布局细节上区分

第二步：自评打分
- 对 3 套方案分别按以下 5 个维度打分（每项 1～5 分）：
  1. 文档符合度：模块、字段、顺序与 07/11 一致
  2. API 契约匹配度：请求/响应与 05 一致，错误态处理正确
  3. 美观度：布局、配色、字体、层次感
  4. 响应式：375px～桌面端无溢出，信息可读
  5. 可维护性：结构清晰、类名规范、易于接入真实 API
- 输出 3×5 的评分表，并计算每套总分（满分 25 分）

第三步：选出最优
- 选出总分最高的那套（若同分，取美观度更高的）
- 明确说明「选中方案 X，总分 Y 分，原因：……」

第四步：仅交付最优版本
- 只输出选中那一套的完整代码（HTML/CSS/JS 文件）
- 附带简短说明：使用了哪些 API、占位数据结构、缺项/问题清单

【重要】用户最终只应看到：评分表 + 最优版本代码 + 说明。不展示落选版本的完整代码。

---

- docs/core/07_系统目标与范围整合.md（页面顺序、四维度绩效、站点结构）

【现有前端基础】
- 业务模板：app/web/templates/（index.html、reports_list.html、report_view.html、dashboard.html、sim_dashboard.html 等）
- Demo 参考：demo/（含 demo.css、11 个静态页）

【约束】
1. 功能基于文档，不得凭空增加未在 07/11/17/12 中约定的功能
2. 与后端 API 严格匹配：请求参数、响应字段、错误处理与 05 契约一致
3. 无数据时使用占位数据，但字段结构、枚举值、类型须符合规范
4. 占位数据示例：至少 3 只股票（600519、300750、002594）；BUY/SELL/HOLD；A/B/C 信号；持仓状态 持有中/已止盈/已止损/已过期
5. 免费用户视角：实操指令卡、14/30/60 日趋势、本股历史明细、高级区、模拟收益分策略 须模糊/锁定
6. 页面 375px～桌面端无横向溢出，关键信息小屏可读

【必须实现的 API 对接】（无后端时可 Mock 响应，结构须与契约一致）
- GET /api/v1/reports（筛选：stock_code, stock_name, trade_date, date_from, date_to, conclusion, q, page, page_size, sort）
- GET /api/v1/reports/{report_id}

- GET /api/v1/membership/subscription/status；POST /api/v1/report-feedback
- POST /api/v1/reports/generate
- GET /api/v1/predictions/stats
- GET /api/v1/sim/positions（stock_code, status, date_from, date_to）
- GET /api/v1/sim/positions/by-report/{report_id}
- GET /api/v1/sim/account/snapshots（capital_tier）
- GET /api/v1/sim/account/summary（capital_tier）
- GET /api/v1/market/state

【必须包含的页面】（与 11 文档 §4、§7 一致）
1. 首页 index.html
2. 研报列表 reports_list.html（7 种筛选：代码/名称、日期、结论、信号类型、持仓状态、市场状态、排序）
3. 研报详情 report_view.html（三层渐进式；分享入口；用户反馈「有帮助/无帮助」+ 无帮助子选项）
4. 统计看板 dashboard.html
5. 模拟收益看板 sim_dashboard.html（档位 1w/10w/50w；回撤横幅；冷启动提示）
6. 登录 login.html、注册 register.html、忘记密码 forgot_password.html
7. 用户个人中心 profile.html
8. 管理员后台 admin.html
9. 充值/订阅 subscribe.html

【美观与响应式】
- 调用 frontend-design skill：避免泛用 AI 审美，采用有辨识度的配色、字体、布局
- 调用 responsive-design skill：mobile-first、fluid typography、断点 375/640/768/1024px
- Design Tokens 见 11 文档 §2，须定义 --space-5、--space-10（11 §2 已含）
- 中文 line-height ≥ 1.75；letter-spacing 负值仅用于纯数字

【缺项与问题报告】
若发现以下情况，必须明确列出并说明处理方式：
- 文档与现有实现冲突（如字段名、枚举值不一致）
- API 契约中未定义但页面需要的字段
- 占位数据无法满足的边界（如冷启动、空状态）
- 无法在纯前端实现的逻辑（如鉴权、支付回调）

【验收 AC】（生成完成后自检）
- 免费用户视角：实操指令卡价格显示 ¥**.**，高级区锁定
- API 路径：所有调用含 /api/v1 前缀，与 05 契约一致
- 占位数据：字段名与 05 report_list_item、SimPosition 一致

【最终输出（仅最优版本）】
1. 3×5 评分表 + 每套总分 + 选中说明（如「选中方案 B，总分 23 分」）
2. 最优版本的完整 HTML/CSS/JS 文件（可多页或 SPA）
3. 占位数据可放在单独 .js 或内联
4. 简短说明：使用了哪些 API、占位数据结构、缺项/问题清单
```

## 二、用法说明

1. 复制上方「一、完整提示词」整段（从 【任务】 到 【最终输出】）
2. 粘贴到 Cursor 对话，并用 `@` 引用：`@docs/core/11_UI设计规范.md`、`@docs/core/07_系统目标与范围整合.md` 等
3. 可选：补充 `请结合 frontend-design 与 responsive-design skill 执行`
4. 发送后，AI 将自动完成：生成 3 套 → 自评打分 → 选出最优 → 只输出最优版本代码
5. 你最终得到：评分表 + 一套完整前端页面 + 说明，无需手动对比

## 三、评委评分表（AI 自评用，已嵌入提示词）

执行提示词时，AI 会按此表对 3 套方案分别打分：

| 维度 | 1分 | 3分 | 5分 |
|------|-----|-----|-----|
| 文档符合度 | 多处遗漏或顺序错误 | 少量遗漏 | 完全符合 07/11 |
| API 契约匹配 | 字段/枚举明显不符 | 基本匹配，少量缺省 | 完全匹配 05 |
| 美观度 | 单调、堆砌 | 基本统一 | 有辨识度、层次清晰 |
| 响应式 | 小屏溢出或严重错位 | 基本可用 | 各断点体验良好 |
| 可维护性 | 结构混乱、硬编码多 | 部分模块化 | 结构清晰、易扩展 |

## 六、Skills 调用建议
